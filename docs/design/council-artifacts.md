# Консилиум: Артефакты и связи в Myrmex Control

**Дата:** 2026-05-11
**Вопрос:** Единый JSON (монолит) vs. граф связей между файлами?

---

## Позиция: Граф связей с индексной точкой входа

### Тезис

**Не засовывать всё в один JSON.** Вместо этого — построить граф связей между файлами с лёгким индексным корнем (`myrmex.json` как **роутер**, а не хранилище).

### Аргумент

1. **Разные артефакты имеют разный жизненный цикл.** Changelog растёт непрерывно, агенты обновляются при каждом каскадном брейншторме, настройки меняются редко, дизайн-документы — по мере созревания решений. Монолитный JSON превращается в 2000+ строк, где merge-конфликты неизбежны, а поиск по файлу — это `jq` вместо нормального grep/glob.

2. **Навигация уже существует — это файловая система.** 22+ файлов в `docs/design/` — это не проблема, а признак зрелости. Каждый файл — атомарная единица с собственным заголовком, датой, автором. Поиск через `grep -r`, навигация через структуру каталогов. Перенос всего в JSON убивает эти преимущества.

3. **`myrmex.json` уже перегружен.** Сейчас он содержит задачи, проекты, агентов, changelog, настройки — 5 разных доменов в одном файле. Это нарушает SRP даже для JSON. Каждый раз при правке одного домена приходится парсить/валидировать весь файл.

4. **Граф связей решает проблему навигации без монолита.** Достаточно лёгкого индекса — маппинг `тип_артефакта → путь_к_файлу` — и перекрёстных ссылок внутри файлов. Это даёт и поиск, и навигацию, и атомарность.

### Рекомендуемая структура

```
myrmex.json              → роутер/индекс (только ссылки + критичные настройки)
agents/                  → по одному файлу на агента (или группировка по роли)
skills/                  → карточки скиллов
docs/design/             → дизайн-решения (уже есть)
docs/council/            → протоколы консилиумов
memory/                  → память лаборатории (уже есть)
changelog/               → по файлу на релиз/квартал
```

`myrmex.json` как роутер:
```json
{
  "meta": { "version": "...", "last_updated": "..." },
  "routes": {
    "agents": "agents/",
    "skills": "skills/",
    "design": "docs/design/",
    "changelog": "changelog/"
  },
  "settings": { ... }
}
```

### Риск если игнорировать

**Монолитный JSON станет узким горлышком.** При текущем темпе роста (22 файла за один брейншторм) через 2-3 итерации `myrmex.json` раздуется до нечитаемого состояния. Правки будут требовать полной валидации, merge-конфликты при параллельной работе агентов станут нормой, а поиск по артефактам превратится в навигацию по вложенным JSON-массивам вместо естественного grep по файлам. Файловая система — уже готовая база данных с индексацией, не нужно изобретать её заново внутри JSON.

## Позиция: Гибрид — JSON-индекс + файловый граф с семантическими ссылками

### Тезис

**Ни чистый JSON, ни чистый файловый граф — а гибрид.** `myrmex.json` остаётся операционным ядром (задачи, проекты, статусы), а дизайн-документы, протоколы консилиумов и карточки агентов живут как файлы с явными семантическими ссылками между собой. Связи — не имплицитные (через структуру каталогов), а эксплицитные через frontmatter-ссылки в каждом файле.

### Аргумент

1. **Структура каталогов ≠ граф связей.** 22 файла в `docs/design/archive/` — это плоский список. Без явных ссылок между ними навигация требует ручного чтения каждого файла. Добавление frontmatter-полей `links: [kanban-visual.md, kanban-automation.md]` в каждый артефакт превращает плоский список в навигируемый граф без переноса контента в JSON.

2. **Операционные данные и знания — разные домены.** Задачи, проекты, статусы — это операционные данные, которые меняются часто и должны быть доступны через API. Дизайн-решения, протоколы консилиумов, карточки агентов — это знания, которые меняются редко и должны быть читаемы как документация. Смешивать их в одном JSON — значит размывать границу между данными и знаниями.

3. **Масштабируемость без переписывания.** При гибридном подходе добавление нового типа артефакта (например, `skills/` или `council/`) не требует изменения схемы `myrmex.json` — достаточно создать каталог и добавить маршрут в `routes`. Это соответствует принципу Open/Closed: расширяем без модификации ядра.

### Риск если игнорировать

**Плоский файловый граф без явных связей превратится в свалку документов.** Через месяц 22 файла превратятся в 50+, и поиск нужного решения будет требовать перебора. Без эксплицитных ссылок между артефактами каждый новый консилиум будет начинать с нуля, не зная, что уже решено. Семантические ссылки в frontmatter — минимальная цена за навигируемость, которая окупается при первом же обращении к архиву.
