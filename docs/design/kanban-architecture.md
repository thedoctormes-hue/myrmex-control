# Информационная архитектура канбан-системы Myrmex Control

> **Версия:** 2.0 | **Дата:** 2026-05-11 | **Автор:** OWL (Information Architect)

---

## 1. Обзор системы досок

Три доски, три контекста:

- **ЗАВЛАБ** — стратегическая. Проекты, эксперименты, инфраструктура. Колонки: BACKLOG, TODO, IN PROGRESS, REVIEW, DONE.
- **МУРАВЕЙ** — тактическая. Работа агентов. Колонки: QUEUED, WORKING, BLOCKED, TESTING, COMPLETED.
- **КОТ** — личная. Личные задачи, идеи. Колонки: INBOX, THIS WEEK, DOING, WAITING, DONE.

Ключевые отличия:

| Аспект | ЗАВЛАБ | МУРАВЕЙ | КОТ |
|--------|--------|---------|-----|
| Владелец | Евгений + агенты | AI-агенты | Евгений |
| Гранулярность | Проекты, эпики | Задачи агентов | Личные задачи |
| Автоматистика | Весы -> МУРАВЕЙ | Учёт веса | Нет |
| Группировка | По эпикам | По исполнителям | Плоский список |
| Ритм | Еженедельно | Ежедневно | По необходимости |

Междосочные связи: ЗАВЛАБ -> МУРАВЕЙ (назначение), МУРАВЕЙ -> ЗАВЛАБ (эскалация), КОТ -> ЗАВЛАБ (промоция), КОТ -> МУРАВЕЙ (делегирование).

## 2. Иерархия целей (Goals -> Epics -> Tasks)

### 2.1. Проблема

Текущая структура плоская: Project -> Task. Нет промежуточных уровней, нет целей, нет связи со стратегией.

### 2.2. Иерархия

```
Уровень 0: Цель компании          — 12 мес.
 └── Уровень 1: Цель лаборатории   — 3-6 мес.
      └── Уровень 2: Проект        — 1-3 мес.
           └── Уровень 3: Эпик     — 1-4 нед.
                └── Уровень 4: Задача — 1-3 дня
                     └── Уровень 5: Подзадача — 2-4 часа
```

### 2.3. Детализация

**Уровень 0 (Цель компании):** Пример: «Запустить платформу AI-агентов с 60k пользователей». Не отображается на доске.

**Уровень 1 (Цель лаборатории):** Пример: «Myrmex Control управляет 25 агентами с 99.9% uptime». Отображается в шапке проекта.

**Уровень 2 (Проект):** Пример: «Рефакторинг канбан-системы». Обязательные: goal_id, владелец, статус, приоритет.

**Уровень 3 (Эпик):** Пример: «Система зависимостей между задачами». Swimlane на доске. Обязательные: project_id, статус, прогресс.

**Уровень 4 (Задача):** Пример: «Добавить поле dependencies в API». Обязательные: epic_id, статус, приоритет, исполнитель, вес.

**Уровень 5 (Подзадача):** Чеклист внутри карточки. Наследует статус задачи.

### 2.4. Правила

1. Не глубже 5 уровней. 2. Эпик без задач > недели — архивировать. 3. Задача без эпика — нормально. 4. Подзадача без задачи — невозможна. 5. Одна задача — один эпик. 6. Проект без цели — подозрителен.

### 2.5. Примеры целей на 2026

**Цель 1: Масштабирование инфраструктуры** — VPN Infrastructure, Server Monitoring, Syncthing Dashboard.
**Цель 2: Автоматизация лаборатории** — Myrmex Control, Evolution Engine, Content Pipeline.
**Цель 3: Монетизация** — OpenClawBox, B2B Sales Pipeline, Portfolio & Marketing.

## 3. User Story Mapping

Колонки = этапы процесса, строки = активности.

**Карта ЗАВЛАБ:** Строки: Цели, Проекты, Эпики, Задачи, Баги, Техдолг. Колонки: BACKLOG -> TODO -> IN PROGRESS -> REVIEW -> DONE.

**Карта МУРАВЕЙ:** Строки: по агентам. Колонки: QUEUED -> WORKING -> BLOCKED -> TESTING -> COMPLETED.

**Карта КОТ:** Строки: Идеи, Задачи, Обучение. Колонки: INBOX -> THIS WEEK -> DOING -> WAITING -> DONE.

Принципы: горизонт = время, вертикаль = приоритет, строки = контекст, размер = объём, цвет = тип.

## 4. Связи и блокировки

### 4.1. Типы связей

**blocks** — зависимость (A не начнётся без B). **parent-child** — подзадача (чеклист). **related** — смысловая связь. **duplicate** — дубликат. **belongs-to** — принадлежность проекту. **escalates** — эскалация между досками.

### 4.2. Визуализация

Заблокированная: красный замок, «Заблокирована: [задача]», opacity 0.6. Блокирующая: оранжевый щит. На доске: пунктирные стрелки, счётчик в заголовке колонки.

### 4.3. Правила

1. Одна задача может блокировать несколько (1:N). 2. Одна задача может быть заблокирована несколькими (N:1). 3. Циклические зависимости запрещены. 4. Заблокированная задача не может перейти в IN PROGRESS. 5. При завершении блокирующей — уведомление. 6. Максимальная глубина цепочки — 5.

### 4.4. Колонка BLOCKED (МУРАВЕЙ)

Полноценная колонка. Задача должна иметь причину. Максимум 48 часов без обновления — автоэскалация на ЗАВЛАБ.

## 5. Контекст задач

### 5.1. Формула

ЧТО (title), ЗАЧЕМ (why), ДЛЯ КОГО (who), КРИТЕРИИ (done), КОГДА (when), КОНТЕКСТ (context).

### 5.2. Карточка

Минимальная: бейдж эпика, заголовок, вес + приоритет, исполнитель, блокировка, теги, прогресс.
Развёрнутая: + ЗАЧЕМ, ДЛЯ КОГО, критерии (чеклист), контекст (ссылки), зависимости, тип, дедлайн.

### 5.3. Шаблоны

**Фича:** ЗАЧЕМ, ДЛЯ КОГО, КРИТЕРИИ, КОНТЕКСТ. **Баг:** ЧТО НЕ РАБОТАЕТ, ОЖИДАЕМОЕ ПОВЕДЕНИЕ, ШАГИ, ОКРУЖЕНИЕ. **Исследование:** ВОПРОС, ГИПОТЕЗА, МЕТОД, РЕЗУЛЬТАТ. **Инцидент:** СЕРЬЁЗНОСТЬ (P1-P4), ЧТО ПРОИЗОШЛО, ВЛИЯНИЕ, ШАГИ.

### 5.4. Правила

1. Заголовок = ЧТО, описание = ЗАЧЕМ и КАК. 2. Критерии обязательны для задач > 4 часов. 3. Ссылки обязательны для межсистемных задач. 4. Дедлайн обязателен для TODO/IN PROGRESS. 5. Без контекста — в BACKLOG. 6. Вес обязателен для ЗАВЛАБ.

## 6. Группировка задач

Режимы: по эпикам (ЗАВЛАБ по умолчанию), по проектам, по исполнителям (МУРАВЕЙ по умолчанию), по приоритету, по тегам, по типу (feature, bug, tech_debt, research, chore), без группировки (КОТ по умолчанию).

Правила: одна группировка за раз, пустые группы скрываются, группа сворачивается, счётчик в заголовке.

---

## 7. Фильтры и виды (Views)

### 7.1. Фильтры (AND-логика)

По исполнителю, проекту, приоритету, тегам, дате, статусу, блокировкам, весу (диапазон SP), типу задачи, доске.

### 7.2. Предустановленные виды

- **Мой день** (МУРАВЕЙ): исполнитель = я, статус = TODO/IN PROGRESS.
- **Обзор лаборатории** (ЗАВЛАБ): статус = IN PROGRESS/REVIEW, по проектам.
- **Заблокированные:** все заблокированные, по проектам.
- **Просроченные:** дедлайн < сегодня, != DONE, по исполнителям.
- **Backlog Grooming:** статус = BACKLOG, по дате создания.
- **Неделя** (КОТ): дедлайн = эта неделя.
- **Эскалации** (ЗАВЛАБ): задачи с МУРАВЕЙ, по агенту.

### 7.3. Сохранённые виды

Произвольная комбинация фильтров с именем. Можно расшарить и закрепить.

---

## 8. Дашборд доски

### 8.1. Сводные метрики

Всего задач, по статусам, по приоритетам, заблокированных, просроченных, без исполнителя.

### 8.2. Прогресс

Общий (% завершённых), по проектам, по эпикам, по исполнителям.

### 8.3. Аналитика потока

**CFD:** накопление задач по статусам за 30 дней. **Cycle Time:** среднее IN PROGRESS -> DONE. **Lead Time:** среднее BACKLOG -> DONE. **Throughput:** завершённых за период. **WIP:** текущие IN PROGRESS + REVIEW (правило: WIP <= исполнители x 2).

### 8.4. Узкие места

Колонка с наибольшим количеством/временем задач. Подсветка заголовка (жёлтый/красный), список «застрявших», тепловая карта.

### 8.5. Метрики по исполнителям (МУРАВЕЙ)

Количество задач, cycle time, throughput (задач/неделю), заблокированных, соотношение типов.

### 8.6. Виджеты

Task Distribution (donut), Priority Breakdown (bar), CFD (area), Cycle Time Trend (line), Throughput (bar), Bottleneck Alert, Agent Load (horizontal bar), Blocked Tasks (list), Overdue Tasks (list), Recently Completed (feed).

## 9. Ритуалы

### 9.1. Ежедневный стендап

Ежедневно, 15 минут. Каждый агент: что сделал, что делаю, блокировки. Автоотчёт «Что изменилось за сутки», подсветка задач без движения > 2 дней.

### 9.2. Пополнение бэклога

Еженедельно, 30-60 мин, Евгений. Просмотр, удаление устаревших, добавление новых, уточнение, приоритеты, разбиение. TODO заполнен на 2-3 недели.

### 9.3. Планирование спринта

Раз в 2 недели, 30-60 мин. Цели (1-3), выбор задач, назначение, оценка, проверка ёмкости. Ёмкость = исполнители x throughput x 2 недели.

### 9.4. Ретроспектива

Раз в 2 недели, 30 мин. Формат 4L: Liked, Learned, Lacked, Longed for. Метрики, паттерны, задачи на улучшение.

### 9.5. Еженедельный обзор

Еженедельно (пятница), 20-30 мин, Евгений. Все доски, исполнители/дедлайны, застрявшие задачи, приоритеты.

### 9.6. Ежемесячный стратегический обзор

Ежемесячно, 60 мин. Цели, метрики, завершённые проекты, корректировка, архивирование.

### 9.7. Сводная таблица

| Ритуал | Частота | Длительность | Участники |
|--------|---------|-------------|-----------|
| Daily Standup | Ежедневно | 15 мин | Евгений + агенты |
| Backlog Replenishment | Еженедельно | 30-60 мин | Евгений |
| Sprint Planning | Раз в 2 недели | 30-60 мин | Евгений + агенты |
| Retrospective | Раз в 2 недели | 30 мин | Евгений + агенты |
| Weekly Review | Еженедельно | 20-30 мин | Евгений |
| Monthly Strategic | Ежемесячно | 60 мин | Евгений |

## 10. Автоматическое распределение задач через весы сложности

### 10.1. Концепция

Задачи на ЗАВЛАБ имеют вес (story points). Система автоматически распределяет задачи агентам на МУРАВЕЙ, учитывая вес, навыки и загрузку.

### 10.2. Шкала весов

Модифицированный Фибоначчи: 1, 2, 3, 5, 8, 13, 21.

- 1 SP — тривиальная (до 1 часа)
- 2 SP — простая (1-2 часа)
- 3 SP — стандартная (2-4 часа)
- 5 SP — средняя (1-2 дня)
- 8 SP — сложная (2-4 дня)
- 13 SP — очень сложная (1-2 недели, рекомендуется разбить)
- 21 SP — экстремальная (рекомендуется разбить на проекты)

### 10.3. Профиль агента (AgentCapacity)

Каждый агент имеет: max_sp_per_week, current_sp_load, skills, preferred_task_types, availability (0.0-1.0).

Примеры: КотОлизатОр (max_sp=21, availability=0.5), deploy-bot (max_sp=13, availability=0.8), anti-dpi-legend (max_sp=13, availability=0.7), content-agent (max_sp=8, availability=0.9).

### 10.4. Алгоритм

1. **Определение веса** — при создании задачи на ЗАВЛАБ. Если > 13 — рекомендация разбить.
2. **Подходящие агенты** — фильтрация по навыкам, доступности, ёмкости (current_sp_load + task_sp <= max_sp_per_week).
3. **Ранжирование** — по совпадению навыков, наименьшей загрузке, предпочтениям.
4. **Назначение** — копирование на МУРАВЕЙ, статус IN PROGRESS на ЗАВЛАБ, уведомление.
5. **Балансировка** — перегруженным не назначается. Если нет подходящего — «Требует ручного назначения».

### 10.5. Ручное переопределение

Евгений может: переназначить, изменить вес, отменить автораспределение, заблокировать агента.

### 10.6. Визуализация

Бейдж «5 SP» на карточке. Загрузка агентов в SP на дашборде МУРАВЕЙ. Гистограмма распределения по весам на ЗАВЛАБ.

## 11. Междосочная маршрутизация

### 11.1. Триггеры

**ЗАВЛАБ -> МУРАВЕЙ (авто):** задача переведена в TODO, имеет вес, найден агент.
**ЗАВЛАБ -> МУРАВЕЙ (ручное):** Евгений назначает явно.
**МУРАВЕЙ -> ЗАВЛАБ (эскалация):** BLOCKED > 48ч, агент не может выполнить, требуется решение Евгения.
**КОТ -> ЗАВЛАБ (промоция):** перетаскивание с получением проекта/эпика/веса.
**КОТ -> МУРАВЕЙ (делегирование):** назначение личной задачи агенту.

### 11.2. Правила

1. При перемещении сохраняется cross-reference связь. 2. Оригинал получает ссылку на копию и наоборот. 3. Синхронизация статусов: при завершении на МУРАВЕЙ — обновление на ЗАВЛАБ. 4. История перемещений в changelog. 5. Нельзя удалить задачу на одной доске, если есть связанная на другой.

### 11.3. Конфликты

Если задача изменена на обеих досках одновременно — конфликт. Последнее изменение выигрывает, конфликт логируется и виден на дашборде.

## Приложение A: Глоссарий

- **Доска** — визуальное представление рабочего процесса с колонками и карточками.
- **Карточка** — визуальное представление задачи на доске.
- **Колонка** — вертикальная группа карточек, представляющая этап процесса.
- **Эпик** — крупная единица работы, содержащая несколько задач.
- **Swimlane** — горизонтальная группа карточек внутри доски.
- **WIP** — Work In Progress, задачи в работе.
- **Cycle Time** — время от начала работы до завершения.
- **Lead Time** — время от создания до завершения.
- **Throughput** — количество завершённых задач за период.
- **CFD** — Cumulative Flow Diagram.
- **View** — сохранённая комбинация фильтров, группировки и сортировки.
- **Story Points (SP)** — единица измерения сложности задачи.
- **Cross-reference** — связь между задачами на разных досках.

## Приложение B: Принципы проектирования

1. **Задача без контекста — не задача.** Не помещай на доску то, что не можешь объяснить.
2. **Доска — зеркало реальности.** Если доска не отражает реальное состояние — она бесполезна.
3. **Меньше задач в работе — быстрее результат.** Ограничивай WIP.
4. **Блокировки — сигнал к действию.** Заблокированная задача требует немедленного внимания.
5. **Метрики — не для отчёта, а для улучшения.** Собирай метрики, чтобы принимать решения.
6. **Ритуалы — скелет процесса.** Без ритуалов доска превращается в мусорку.
7. **Простота > Полнота.** Лучше простая доска, которую используют, чем сложную, которую игнорируют.
8. **Автоматизация — помощник, не замена.** Весы сложности помогают распределять, но Евгений всегда может переопределить.
