## 1. Обзор системы досок

### 1.1. Три доски — три контекста

Myrmex Control управляет тремя досками, каждая из которых обслуживает свой контекст:

| Доска | Владелец | Цель | Аудитория |
|-------|----------|------|-----------|
| **ЗавЛаб** | Евгений + AI-агенты | Управление лабораторией: проекты, эксперименты, инфраструктура | Команда лаборатории |
| **МУРАВЕЙ** | AI-агенты | Работа агентов: задачи, статусы, производительность | Евгений (надзор), агенты (исполнение) |
| **КОТ** | Евгений (личное) | Личные задачи, идеи, саморазвитие | Только Евгений |

### 1.2. Принцип разделения

**ЗавЛаб** — стратегическая доска. Здесь живут проекты, эпики и высокоуровневые задачи. Это «карта местности» лаборатории.

**МУРАВЕЙ** — тактическая доска. Здесь агенты видят свои задачи, обновляют статусы, отмечают прогресс. Это «рабочий стол» колонии.

**КОТ** — личная доска. Здесь Евгений управляет своими задачами без шума лаборатории. Это «личный блокнот».

### 1.3. Междосочные связи

Задачи могут перетекать между досками:
- **ЗавЛаб → МУРАВЕЙ**: когда задача из лаборатории назначается агенту
- **МУРАВЕЙ → ЗавЛаб**: когда агент создаёт задачу, требующую внимания лаборатории
- **КОТ → ЗавЛаб**: когда личная идея становится проектом лаборатории

## 2. Иерархия целей (Goals → Epics → Tasks)

### 2.1. Проблема

Сейчас данные имеют плоскую структуру: `Project → Task` с опциональным `parent_id`. Нет промежуточного уровня между проектом и задачей, нет целей проекта, нет связи с целями компании.

### 2.2. Предлагаемая иерархия

```
🎯 Цель компании (Company Goal)
 └── 🎯 Цель лаборатории (Lab Goal)
      └── 📦 Проект (Project)
           └── 🏔️ Эпик (Epic)
                └── 📋 Задача (Task)
                     └── 📝 Подзадача (Subtask)
```

### 2.3. Уровни иерархии

**Уровень 0: Цель компании**
- Пример: «Запустить платформу AI-агентов для малого бизнеса»
- Горизонт: 12 месяцев
- Владелец: Евгений
- Не отображается на доске напрямую, но видна в контексте проекта

**Уровень 1: Цель лаборатории**
- Пример: «Myrmex Control должен управлять 50 агентами без сбоев»
- Горизонт: 3-6 месяцев
- Владелец: Евгений + старшие агенты
- Отображается как «Goal» в шапке проекта

**Уровень 2: Проект**
- Пример: «Рефакторинг канбан-системы»
- Горизонт: 1-3 месяца
- Владелец: агент или Евгений
- Отображается как проект на доске ЗавЛаб

**Уровень 3: Эпик**
- Пример: «Реализовать систему зависимостей между задачами»
- Горизонт: 1-4 недели
- Владелец: агент
- Отображается как группировка (swimlane) или сворачиваемый блок на доске

**Уровень 4: Задача**
- Пример: «Добавить поле dependencies в API задач»
- Горизонт: 1-3 дня
- Владелец: агент
- Отображается как карточка на доске

**Уровень 5: Подзадача**
- Пример: «Написать миграцию для поля dependencies»
- Горизонт: 2-4 часа
- Владелец: агент
- Отображается как чеклист внутри карточки задачи

### 2.4. Правила иерархии

1. **Не глубже 5 уровней.** Если нужен 6-й — нужно реструктурировать.
2. **Эпик без задач — мусор.** Если в эпике нет задач больше недели — архивировать.
3. **Задача без эпика — нормально.** Мелкие задачи могут жить напрямую в проекте.
4. **Подзадача без задачи — невозможна.** Подзадача всегда принадлежит задаче.
5. **Одна задача — один эпик.** Задача не может принадлежать нескольким эпикам.

### 2.5. Визуализация иерархии

На доске иерархия отображается через:

- **Цвет карточки** — наследуется от проекта
- **Бейдж эпика** — метка на карточке задачи
- **Прогресс-бар эпика** — сколько задач в эпике завершено
- **Сворачивание** — эпик можно свернуть, скрыв задачи
- **Связи** — стрелки от задачи к задаче (зависимости)

## 3. User Story Mapping

### 3.1. Концепция

User Story Mapping — это способ визуализации пути пользователя через систему. Для канбан-доски это означает: **колонки = этапы процесса, строки = активности пользователя**.

### 3.2. Карта для доски ЗавЛаб

```
                    BACKLOG    →    TODO    →    IN PROGRESS    →    REVIEW    →    DONE
─────────────────────────────────────────────────────────────────────────────────────────
🎯 Цели            │ [G1][G2]  │          │                │           │ [G0]
📦 Проекты         │ [P3][P4]  │ [P2]     │ [P1]           │           │ [P0]
🏔️ Эпики           │ [E5][E6]  │ [E3][E4] │ [E1][E2]       │           │ [E0]
📋 Задачи          │ [T]x12    │ [T]x5    │ [T]x3          │ [T]x2     │ [T]x8
🐛 Баги            │ [B]x4     │ [B]x1    │ [B]x2          │           │ [B]x15
```

### 3.3. Карта для доски МУРАВЕЙ

```
                    QUEUED    →    WORKING    →    BLOCKED    →    TESTING    →    COMPLETED
────────────────────────────────────────────────────────────────────────────────────────────
🤖 Агент Кошка     │ [T]x2     │ [T]x1      │            │            │ [T]x5
🤖 Агент Собака    │ [T]x3     │            │ [T]x1       │ [T]x1      │ [T]x3
🤖 Агент Попугай   │           │ [T]x2      │            │            │ [T]x7
```

### 3.4. Карта для доски КОТ

```
                    INBOX    →    THIS WEEK    →    DOING    →    WAITING    →    DONE
──────────────────────────────────────────────────────────────────────────────────────────
💡 Идеи            │ [I]x5     │              │           │            │ [I]x2
📋 Задачи          │ [T]x8     │ [T]x3        │ [T]x1     │ [T]x2      │ [T]x12
📚 Обучение        │ [L]x3     │ [L]x1        │           │            │ [L]x4
```

### 3.5. Принципы Story Mapping

1. **Горизонт = время** (от начала к завершению)
2. **Вертикаль = приоритет** (сверху — важнее)
3. **Строки = контекст** (кто/что работает)
4. **Размер карточки = объём работы** (маленькая = быстрая, большая = сложная)
5. **Цвет карточки = тип работы** (фича, баг, техдолг, исследование)

## 4. Связи и блокировки

### 4.1. Типы связей

| Тип связи | Описание | Визуализация |
|-----------|----------|--------------|
| **Зависимость** (blocks) | Задача A не может начаться, пока не завершена задача B | Стрелка B → A, красная пунктирная |
| **Подзадача** (parent-child) | Задача A содержит подзадачи B, C, D | Вложенный чеклист внутри карточки |
| **Связанная** (related) | Задачи A и B связаны по смыслу, но не блокируют друг друга | Тонкая серая линия |
| **Дубликат** (duplicate) | Задача A дублирует задачу B | Ссылка с пометкой «дубликат» |
| **Принадлежит** (belongs-to) | Задача A принадлежит проекту P | Цвет карточки = цвет проекта |

### 4.2. Визуализация блокировок

**На карточке задачи:**
- Красный значок «замок» если задача заблокирована
- Текст «Заблокирована: [название задачи]» в описании
- Серая карточка с пониженной прозрачностью (opacity: 0.6)

**На карточке блокирующей задачи:**
- Оранжевый значок «щит» — эта задача блокирует другие
- Список «Блокирует: [список задач]» в детальном виде

**На доске:**
- Пунктирные стрелки между карточками (если обе видны)
- Индикатор в заголовке колонки: «3 задачи заблокированы»

### 4.3. Правила блокировок

1. **Одна задача может блокировать несколько задач** (1:N)
2. **Одна задача может быть заблокирована несколькими задачами** (N:1) — все должны быть завершены
3. **Циклические зависимости запрещены.** Система должна обнаруживать циклы и предупреждать
4. **Заблокированная задача не может быть перемещена в IN PROGRESS**
5. **При завершении блокирующей задачи** — уведомление владельцу заблокированной задачи

### 4.4. Статус «BLOCKED»

Отдельный статус «BLOCKED» — это антипаттерн. Вместо этого:
- Задача остаётся в текущей колонке (например, TODO)
- На карточке появляется красный индикатор «Заблокирована»
- Фильтр «Показать заблокированные» позволяет быстро найти все заблокированные задачи

## 5. Контекст задач

### 5.1. Проблема потери контекста

Задача «Сделать отчёт» — это не задача. Это пустышка. Без контекста она вызывает вопросы:
- Какой отчёт?
- Для кого?
- Зачем?
- Какой формат?
- Какие данные?

Через неделю даже автор не вспомнит, что имел в виду.

### 5.2. Формула контекста задачи

Каждая задача должна отвечать на вопросы:

```
ЧТО сделать?      → title (заголовок)
ЗАЧЕМ сделать?    → why (цель/обоснование)
ДЛЯ КОГО?         → who (аудитория/получатель)
КАК понять что     → done (критерии готовности)
  готово?
КОГДА нужно?      → when (дедлайн)
ЧТО нужно ЗНАТЬ?  → context (ссылки, ресурсы, заметки)
```

### 5.3. Структура карточки задачи

**Минимальная карточка (видна на доске):**
```
┌─────────────────────────────────┐
│ 🏔️ Эпик: Рефакторинг API       │  ← бейдж эпика
│ 📋 Добавить поле dependencies  │  ← заголовок
│ 🔴 Высокий | 🤖 Агент Кошка    │  ← приоритет + исполнитель
│ 🔒 Заблокирована: T-123        │  ← блокировка (если есть)
│ 🏷️ backend, api, v0.2          │  ← теги
│ ████████░░ 80%                  │  ← прогресс (из подзадач)
└─────────────────────────────────┘
```

**Развёрнутая карточка (модалка/панель):**
```
┌─────────────────────────────────────────┐
│ 📋 Добавить поле dependencies           │
│                                         │
│ 🎯 ЗАЧЕМ: Чтобы задачи могли            │
│    ссылаться друг на друга и            │
│    показывать блокировки                │
│                                         │
│ 👤 ДЛЯ КОГО: Агенты, которые            │
│    работают над связанными задачами     │
│                                         │
│ ✅ КРИТЕРИИ ГОТОВНОСТИ:                 │
│    □ Миграция БД                        │
│    □ API endpoint PATCH /tasks/:id      │
│    □ Валидация циклических зависимостей │
│    □ Тесты                              │
│                                         │
│ 📎 КОНТЕКСТ:                            │
│    • Схема: /docs/api/tasks.md          │
│    • Пример: GitHub Projects API        │
│    • Обсуждение: #42                    │
│                                         │
│ 🔗 ЗАВИСИМОСТИ:                         │
│    ← T-123 «Обновить схему БД»          │
│    → T-456 «UI для связей»              │
│                                         │
│ 🏷️ backend, api, v0.2                  │
│ 📅 Дедлайн: 2026-05-15                 │
│ 🤖 Исполнитель: Агент Кошка             │
│ 📊 Прогресс: 2/4 подзадачи              │
└─────────────────────────────────────────┘
```

### 5.4. Шаблоны задач

Для типовых задач — шаблоны с предзаполненным контекстом:

**Шаблон «Фича»:**
```
ЗАЧЕМ: [какая проблема решается]
ДЛЯ КОГО: [кто получит пользу]
КРИТЕРИИ:
  □ [конкретный критерий 1]
  □ [конкретный критерий 2]
КОНТЕКСТ:
  • [ссылка на дизайн/спецификацию]
```

**Шаблон «Баг»:**
```
ЧТО НЕ РАБОТАЕТ: [описание бага]
ОЖИДАЕМОЕ ПОВЕДЕНИЕ: [что должно быть]
ШАГИ ВОСПРОИЗВЕДЕНИЯ: [пошагово]
ОКРУЖЕНИЕ: [версия, ОС, браузер]
```

**Шаблон «Исследование»:**
```
ВОПРОС: [что исследуем]
ГИПОТЕЗА: [что предполагаем]
МЕТОД: [как проверяем]
РЕЗУЛЬТАТ: [что нашли — заполняется после]
```

### 5.5. Правила контекста

1. **Заголовок — это ЧТО, описание — это ЗАЧЕМ и КАК**
2. **Критерии готовности обязательны** для задач больше 4 часов
3. **Ссылки на ресурсы обязательны** для задач, связанных с другими системами
4. **Дедлайн обязателен** для задач в колонках TODO и IN PROGRESS
5. **Без контекста — в BACKLOG.** Если не можешь описать задачу — она не готова к работе

## 6. Группировка задач на доске

### 6.1. Режимы группировки

Доска поддерживает несколько режимов группировки, переключаемых пользователем:

**По эпикам (swimlanes):**
```
🏔️ Эпик: Рефакторинг API
  BACKLOG │ TODO │ IN PROGRESS │ REVIEW │ DONE
  [T-101] │[T-102]│  [T-103]   │        │[T-100]

🏔️ Эпик: Новый UI
  BACKLOG │ TODO │ IN PROGRESS │ REVIEW │ DONE
  [T-201] │       │  [T-203]   │[T-202] │
```

**По проектам:**
```
📦 Проект: Myrmex Control
  BACKLOG │ TODO │ IN PROGRESS │ REVIEW │ DONE
  [T-101] │[T-102]│  [T-103]   │        │[T-100]

📦 Проект: VPN Infrastructure
  BACKLOG │ TODO │ IN PROGRESS │ REVIEW │ DONE
  [T-201] │       │  [T-203]   │[T-202] │
```

**По исполнителям (для доски МУРАВЕЙ):**
```
🤖 Агент Кошка
  QUEUED │ WORKING │ BLOCKED │ TESTING │ COMPLETED
  [T-101] │ [T-102] │         │         │ [T-100]

🤖 Агент Собака
  QUEUED │ WORKING │ BLOCKED │ TESTING │ COMPLETED
  [T-201] │         │ [T-204] │ [T-202] │
```

**По приоритету:**
```
🔴 Критический
  BACKLOG │ TODO │ IN PROGRESS │ REVIEW │ DONE
  [T-101] │       │  [T-103]   │        │

🟡 Высокий
  BACKLOG │ TODO │ IN PROGRESS │ REVIEW │ DONE
          │[T-102]│            │        │[T-100]

🟢 Средний
  BACKLOG │ TODO │ IN PROGRESS │ REVIEW │ DONE
  [T-201] │       │            │        │
```

**По тегам:**
```
🏷️ backend
  BACKLOG │ TODO │ IN PROGRESS │ REVIEW │ DONE
  [T-101] │[T-102]│  [T-103]   │        │

🏷️ frontend
  BACKLOG │ TODO │ IN PROGRESS │ REVIEW │ DONE
  [T-201] │       │  [T-203]   │[T-202] │
```

### 6.2. Режим по умолчанию

| Доска | Режим по умолчанию |
|-------|--------------------|
| ЗавЛаб | По эпикам (swimlanes) |
| МУРАВЕЙ | По исполнителям |
| КОТ | Без группировки (плоский список) |

### 6.3. Правила группировки

1. **Одна группировка за раз** — нельзя группировать по эпикам И проектам одновременно
2. **Пустые группы скрываются** — если в группе нет задач, она не отображается
3. **Группа сворачивается** — можно свернуть группу, скрыв задачи
4. **Счётчик в заголовке группы** — «Эпик: Рефакторинг API (5)»

## 7. Фильтры и виды (Views)

### 7.1. Фильтры

Фильтры позволяют сузить набор видимых задач. Фильтры комбинируются (AND-логика).

**Фильтр по исполнителю:**
- Множественный выбор: один или несколько агентов
- Опция «Без исполнителя» — показать неназначенные задачи
- Опция «Мои задачи» — быстрый фильтр для текущего пользователя

**Фильтр по проекту:**
- Множественный выбор: один или несколько проектов
- Иерархический выбор: проект → эпик → задача

**Фильтр по приоритету:**
- Множественный выбор: critical, high, medium, low
- Быстрые кнопки: «Только критические», «Высокий и выше»

**Фильтр по тегам:**
- Множественный выбор из списка существующих тегов
- Поиск по тегам
- Опция «Без тегов»

**Фильтр по дате:**
- Дедлайн: сегодня, на этой неделе, в этом месяце, просроченные
- Создано: сегодня, на этой неделе, в этом месяце
- Завершено: сегодня, на этой неделе

**Фильтр по статусу:**
- Множественный выбор из колонок
- Быстрый фильтр: «Активные» (всё кроме DONE), «Завершённые» (только DONE)

**Фильтр по блокировкам:**
- «Заблокированные» — задачи, которые ждут другие
- «Блокирующие» — задачи, от которых зависят другие
- «Без зависимостей» — изолированные задачи

**Текстовый поиск:**
- Поиск по заголовку и описанию
- Поддержка точного совпадения в кавычках
- Поиск по ID задачи (T-123)

### 7.2. Предустановленные виды (Views)

Вид — это сохранённая комбинация фильтров, группировки и сортировки.

**Вид «Мой день» (для доски МУРАВЕЙ):**
- Фильтр: исполнитель = текущий агент, статус = TODO или IN PROGRESS
- Группировка: по приоритету
- Сортировка: по дедлайну

**Вид «Обзор лаборатории» (для доски ЗавЛаб):**
- Фильтр: статус = IN PROGRESS или REVIEW
- Группировка: по проектам
- Сортировка: по приоритету

**Вид «Заблокированные»:**
- Фильтр: заблокированные = да
- Группировка: по проектам
- Сортировка: по количеству заблокированных задач

**Вид «Просроченные»:**
- Фильтр: дедлайн < сегодня, статус != DONE
- Группировка: по исполнителям
- Сортировка: по дедлайну (самые старые сверху)

**Вид «Backlog Grooming»:**
- Фильтр: статус = BACKLOG
- Группировка: по приоритету
- Сортировка: по дате создания (самые старые сверху)

**Вид «Неделя» (для доски КОТ):**
- Фильтр: дедлайн = эта неделя
- Группировка: без группировки
- Сортировка: по приоритету, затем по дедлайну

### 7.3. Сохранённые виды

- Пользователь может сохранить произвольную комбинацию фильтров как именованный вид
- Виды можно расшарить (для команды лаборатории)
- Виды можно закрепить в быстром доступе

## 8. Дашборд доски

### 8.1. Метрики доски

Дашборд доски — это отдельная вкладка или панель, показывающая состояние доски в цифрах и графиках.

### 8.2. Сводные метрики

**Общая статистика:**
- Всего задач на доске
- Задач по статусам (распределение по колонкам)
- Задач по приоритетам (распределение по приоритетам)
- Заблокированных задач
- Просроченных задач
- Задач без исполнителя

**Прогресс:**
- Общий прогресс по доске (% завершённых задач)
- Прогресс по проектам (% завершённых задач в каждом проекте)
- Прогресс по эпикам (% завершённых задач в каждом эпике)
- Прогресс по исполнителям (сколько задач завершено каждым агентом)

### 8.3. Аналитика потока (Flow Metrics)

**Cumulative Flow Diagram (CFD):**
- График накопления задач по статусам во времени
- Показывает: растёт ли бэклог, ускоряется ли работа, есть ли заторы
- Горизонт: время (последние 30 дней)
- Вертикаль: количество задач
- Цвета: по статусам (BACKLOG, TODO, IN PROGRESS, REVIEW, DONE)

**Cycle Time:**
- Среднее время от начала работы (IN PROGRESS) до завершения (Done)
- Показывает: сколько в среднем занимает задача
- Цель: уменьшать cycle time

**Lead Time:**
- Среднее время от создания задачи (BACKLOG) до завершения (Done)
- Показывает: сколько в среднем задача живёт в системе
- Цель: уменьшать lead time

**Throughput:**
- Количество завершённых задач за период (день/неделя/месяц)
- Показывает: производительность команды
- Целевой ориентир: стабильный или растущий throughput

**Work In Progress (WIP):**
- Текущее количество задач в работе (IN PROGRESS + REVIEW)
- Показывает: загруженность команды
- Правило: WIP не должен превышать (количество исполнителей × 2)

### 8.4. Узкие места (Bottlenecks)

**Определение узких мест:**
- Колонка с наибольшим количеством задач — потенциальное узкое место
- Колонка с наибольшим средним временем нахождения — подтверждённое узкое место
- Задачи, которые дольше всего находятся в колонке — «застрявшие»

**Визуализация:**
- Заголовок колонки подсвечивается жёлтым (предупреждение) или красным (критично)
- Список «застрявших» задач с указанием времени в колонке
- Тепловая карта: время нахождения каждой задачи в каждой колонке

### 8.5. Метрики по исполнителям (для доски МУРАВЕЙ)

- Количество задач у каждого агента
- Средний cycle time по агенту
- Throughput по агенту (задач/неделю)
- Количество заблокированных задач по агенту
- Соотношение типов задач (фичи/баги/техдолг)

### 8.6. Виджеты дашборда

| Виджик | Тип | Описание |
|--------|-----|----------|
| Task Distribution | Donut Chart | Распределение задач по статусам |
| Priority Breakdown | Bar Chart | Распределение по приоритетам |
| CFD | Area Chart | Накопительный поток |
| Cycle Time Trend | Line Chart | Тренд cycle time |
| Throughput | Bar Chart | Завершённые задачи по неделям |
| Bottleneck Alert | Alert | Узкие места на доске |
| Agent Load | Horizontal Bar | Загрузка каждого агента |
| Blocked Tasks | List | Список заблокированных задач |
| Overdue Tasks | List | Список просроченных задач |
| Recently Completed | Feed | Лента недавно завершённых |

## 9. Ритуалы

### 9.1. Ежедневный стендап (Daily Standup)

**Когда:** Каждый день, фиксированное время
**Длительность:** 15 минут
**Участники:** Евгений + активные агенты

**Формат:**
1. Каждый агент отвечает на 3 вопроса:
   - Что я сделал вчера?
   - Что я делаю сегодня?
   - Есть ли блокировки?
2. Евгений просматривает доску МУРАВЕЙ
3. Заблокированные задачи обсуждаются
4. При необходимости — переопределение приоритетов

**Интеграция с доской:**
- Автоматический отчёт «Что изменилось за сутки» (на основе changelog)
- Подсветка задач, которые не двигались больше 2 дней
- Быстрый доступ к заблокированным задачам

### 9.2. Пополнение бэклога (Backlog Replenishment)

**Когда:** Раз в неделю
**Длительность:** 30-60 минут
**Участники:** Евгений

**Формат:**
1. Просмотр бэклога: все ли задачи актуальны?
2. Удаление устаревших задач
3. Добавление новых задач из списка идей
4. Уточнение описаний и критериев готовности
5. Расстановка приоритетов
6. Разбиение крупных задач на эпики

**Правило:** Колонка TODO должна быть заполнена на 2-3 недели вперёд. Если меньше — пора пополнять.

### 9.3. Планирование спринта (Sprint Planning)

**Когда:** Раз в 2 недели (или по необходимости)
**Длительность:** 30-60 минут
**Участники:** Евгений + агенты

**Формат:**
1. Определение целей спринта (1-3 цели)
2. Выбор задач из бэклога для спринта
3. Назначение исполнителей
4. Оценка объёма (можно в сторипоинтах или часах)
5. Проверка: не превышает ли объём ёмкость команды

**Правило:** Ёмкость команды = количество исполнителей × средний throughput × 2 недели

### 9.4. Ретроспектива (Retrospective)

**Когда:** Раз в 2 недели (после спринта)
**Длительность:** 30 минут
**Участники:** Евгений + агенты

**Формат (4L):**
1. **Liked** — что понравилось в этом спринте?
2. **Learned** — что узнали?
3. **Lacked** — чего не хватало?
4. **Longed for** — чего хотелось бы?

**Интеграция с доской:**
- Просмотр метрик спринта (throughput, cycle time, blocked tasks)
- Выявление паттернов (например, задачи в REVIEW застревают)
- Создание задач на улучшение процесса

### 9.5. Еженедельный обзор (Weekly Review)

**Когда:** Раз в неделю (например, пятница вечер)
**Длительность:** 20-30 минут
**Участники:** Евгений

**Формат:**
1. Просмотр всех досок: ЗавЛаб, МУРАВЕЙ, КОТ
2. Проверка: все ли задачи имеют исполнителя и дедлайн?
3. Проверка: нет ли застрявших задач?
4. Обновление приоритетов
5. Планирование следующей недели

### 9.6. Ежемесячный стратегический обзор

**Когда:** Раз в месяц
**Длительность:** 60 минут
**Участники:** Евгений

**Формат:**
1. Обзор целей лаборатории: движемся ли мы к ним?
2. Анализ метрик за месяц (CFD, throughput, cycle time)
3. Обзор завершённых проектов и эпиков
4. Корректировка целей и приоритетов
5. Архивирование завершённых проектов

### 9.7. Чеклист ритуалов

| Ритуал | Частота | Длительность | Участники |
|--------|---------|-------------|-----------|
| Daily Standup | Ежедневно | 15 мин | Евгений + агенты |
| Backlog Replenishment | Еженедельно | 30-60 мин | Евгений |
| Sprint Planning | Раз в 2 недели | 30-60 мин | Евгений + агенты |
| Retrospective | Раз в 2 недели | 30 мин | Евгений + агенты |
| Weekly Review | Еженедельно | 20-30 мин | Евгений |
| Monthly Strategic | Ежемесячно | 60 мин | Евгений |

## 10. Модель данных (рекомендации к расширению)

### 10.1. Текущая модель

Сейчас схема данных (`myrmex.schema.json`) содержит:
- `Project` — проект с полями id, title, description, status, category, tags, priority, color, icon
- `Task` — задача с полями id, projectId, title, description, status, priority, assigneeId, parentId, dependencies, tags, dueDate
- `Agent` — агент с полями id, name, role, status, skills, currentTasks

### 10.2. Рекомендуемые расширения

**Новый тип: Epic (Эпик)**
```
Epic {
  id: string
  projectId: string        // проект-владелец
  title: string
  description: string
  status: active | completed | archived
  color: string            // наследуется от проекта
  goals: string[]          // цели эпика
  progress: number         // % завершения (вычисляемое)
  createdAt: datetime
  updatedAt: datetime
  completedAt: datetime
}
```

**Новый тип: Goal (Цель)**
```
Goal {
  id: string
  title: string
  description: string
  level: company | lab | project
  status: active | achieved | archived
  targetDate: datetime
  metrics: string[]        // метрики для измерения достижения
  createdAt: datetime
  updatedAt: datetime
  achievedAt: datetime
}
```

**Расширение Task:**
```
Task {
  // ... существующие поля
  epicId: string           // ссылка на эпик (опционально)
  why: string              // обоснование (ЗАЧЕМ)
  who: string              // для кого
  acceptanceCriteria: string[]  // критерии готовности
  contextLinks: string[]   // ссылки на ресурсы
  storyPoints: number      // оценка объёма
  type: feature | bug | tech_debt | research | chore
  board: zavlab | muravey | kot  // на какой доске задача
}
```

**Расширение Project:**
```
Project {
  // ... существующие поля
  goalIds: string[]        // ссылки на цели
  parentProjectId: string  // для иерархии проектов
  progress: number         // % завершения (вычисляемое)
  health: green | yellow | red  // здоровье проекта
}
```

**Новый тип: BoardConfig (Конфигурация доски)**
```
BoardConfig {
  id: string
  name: string             // zavlab | muravey | kot
  columns: Column[]        // колонки доски
  defaultView: string      // вид по умолчанию
  wipLimits: object        // лимиты WIP по колонкам
}

Column {
  id: string
  name: string
  order: number
  wipLimit: number         // максимум задач в колонке
  color: string
}
```

**Новый тип: View (Сохранённый вид)**
```
View {
  id: string
  name: string
  boardId: string          // для какой доски
  filters: Filter[]        // набор фильтров
  groupBy: string          // режим группировки
  sortBy: string           // режим сортировки
  isShared: boolean        // расшарен ли
  createdBy: string
  createdAt: datetime
}
```

### 10.3. Связи между сущностями

```
Goal 1───* Project 1───* Epic 1───* Task 1───* Subtask
  │          │           │          │
  │          │           │          *─── Agent (assignee)
  │          │           │
  │          │           *─── Task (без эпика)
  │          │
  │          *─── Epic (без проекта — не рекомендуется)
  │
  *─── Goal (parent-child для целей)
```

## Приложение A: Глоссарий

| Термин | Определение |
|--------|-------------|
| **Доска** | Визуальное представление рабочего процесса с колонками и карточками |
| **Карточка** | Визуальное представление задачи на доске |
| **Колонка** | Вертикальная группа карточек, представляющая этап процесса |
| **Эпик** | Крупная единица работы, содержащая несколько задач |
| **Swimlane** | Горизонтальная группа карточек внутри доски (для группировки) |
| **WIP** | Work In Progress — задачи в работе |
| **Cycle Time** | Время от начала работы до завершения задачи |
| **Lead Time** | Время от создания задачи до её завершения |
| **Throughput** | Количество завершённых задач за период |
| **CFD** | Cumulative Flow Diagram — накопительная диаграмма потока |
| **View** | Сохранённая комбинация фильтров, группировки и сортировки |

## Приложение B: Принципы проектирования

1. **Задача без контекста — не задача.** Не помещай на доску то, что не можешь объяснить.
2. **Доска — зеркало реальности.** Если доска не отражает реальное состояние — она бесполезна.
3. **Меньше задач в работе — быстрее результат.** Ограничивай WIP.
4. **Блокировки — сигнал к действию.** Заблокированная задача требует немедленного внимания.
5. **Метрики — не для отчёта, а для улучшения.** Собирай метрики, чтобы принимать решения.
6. **Ритуалы — скелет процесса.** Без ритуалов доска превращается в мусорку.
7. **Простота > Полнота.** Лучше простая доска, которую используют, чем сложную, которую игнорируют.
