## 1. Обзор системы

Канбан-система Myrmex Control — это не просто доска задач. Это **система управления потоком работы колонии AI-агентов**, где:

- Задачи — карточки, которые двигаются по колонкам от «задумано» к «готово»
- Главное правило — не брать новое, пока не доделал старое (принцип Toyota)
- Три доски с разной семантикой связаны через общий бэклог
- Автоматическая система весов распределяет задачи между каналами
- Человек получает только IRL-задачи, которые агенты выполнить не могут

**Ключевые принципы:**

- Видеть весь поток работы как на ладони
- Ограничивать WIP (work in progress)
- Задача не просто «сделать X», а «сделать X для Y, чтобы Z»
- Автоматизация помогает, а не мешает
- Расширяемость — третий канал уже заложен в архитектуру

## 2. Три доски — три контекста

### ЗАВЛАБ — Стратегическая доска

IRL-задачи лаборатории. Полный контроль человека. Проекты, эксперименты, инфраструктура, найм.

**Колонки:** BACKLOG → TODO → IN PROGRESS → REVIEW → DONE

**Владелец:** Безумный Доктор + агенты

**Гранулярность:** Проекты, эпики

**Ритм:** Еженедельно

### МУРАВЕЙ — Тактическая доска

Задачи AI-агентов. TG-бот (Qwen Code CLI). Получает задачи из бэклога автоматически через весы.

**Колонки:** QUEUED → WORKING → BLOCKED → TESTING → COMPLETED

**Владелец:** AI-агенты

**Гранулярность:** Задачи агентов

**Ритм:** Ежедневно

### КОТ — Личная доска

Личные задачи. TG-бот (Qwen Code CLI). Получает задачи из бэклога автоматически через весы.

**Колонки:** INBOX → THIS WEEK → DOING → WAITING → DONE

**Владелец:** Безумный Доктор (личное)

**Гранулярность:** Личные задачи, идеи

**Ритм:** По необходимости

### Ключевые отличия

- ЗАВЛАБ — стратегическая, проекты и эпики, ручное управление
- МУРАВЕЙ — тактическая, задачи агентов, автоматическое распределение
- КОТ — личная, идеи и задачи, автоматическое распределение
- Междосочные связи: ЗАВЛАБ → МУРАВЕЙ (назначение), МУРАВЕЙ → ЗАВЛАБ (эскалация), КОТ → ЗАВЛАБ (промоция), КОТ → МУРАВЕЙ (делегирование)

## 3. Архитектура распределения задач

### Принцип: Бэклог — единый источник правды

```
┌─────────────────────────────────────────┐
│           ОБЩИЙ БЭКЛОГ                  │
│  (единый пул задач с весами,           │
│   приоритетами, тегами)                 │
└──────────────────┬──────────────────────┘
                   │
          ┌────────▼────────┐
          │  ДИСПЕТЧЕР      │
          │  ВЕСОВ          │
          │  (Weight Router)│
          └────────┬────────┘
                   │
     ┌─────────────┼─────────────┐
     │             │             │
┌────▼────┐  ┌────▼────┐  ┌────▼────┐
│ ЗАВЛАБ  │  │ МУРАВЕЙ │  │  КОТ    │
│ (IRL)   │  │ (TG #1) │  │ (TG #2) │
└─────────┘  └─────────┘  └─────────┘
```

Задача физически живёт в бэклоге. На досках — только ссылки + статус выполнения.

### Каналы распределения

Сейчас два канала (МУРАВЕЙ, КОТ) + ЗАВЛАБ для IRL. Архитектура заложена для третьего канала.

### Балансировка

Система следит чтобы нагрузка была равной между каналами. Учитывает текущую загрузку, сложность задач, приоритеты.

## 4. Колонки и статусы

### ЗАВЛАБ

- **BACKLOG** — идеи и задачи, которые пока не планируются
- **TODO** — запланировано, готово к работе
- **IN PROGRESS** — в работе (WIP-лимит: 3)
- **REVIEW** — на проверке (WIP-лимит: 2)
- **DONE** — завершено

### МУРАВЕЙ

- **QUEUED** — очередь задач агента
- **WORKING** — агент работает (WIP-лимит: 1 на агента)
- **BLOCKED** — заблокирована (зависимость, внешний фактор)
- **TESTING** — на проверке / тестировании
- **COMPLETED** — завершено

### КОТ

- **INBOX** — входящие идеи и задачи
- **THIS WEEK** — запланировано на неделю
- **DOING** — делаю сейчас (WIP-лимит: 1)
- **WAITING** — жду внешнего события
- **DONE** — завершено

### WIP-лимиты

WIP-лимиты — жёсткие. Нельзя добавить задачу в колонку, если лимит исчерпан. Визуализация: колонка подсвечивается красным при превышении.

## 5. Система весов

### Итоговый вес задачи

```
W_total = W_priority + W_age + W_deps + W_channel + W_recurring
```

**Компоненты:**

- **W_priority** (0–40): Базовый приоритет. P0=критический(40), P1=высокий(30), P2=средний(20), P3=низкий(10)
- **W_age** (0–20): Штраф за старение. Растёт со временем нахождения в бэклоге
- **W_deps** (0–15): Бонус за разблокировку зависимых задач
- **W_channel** (-10..+10): Корректировка под канал (МУРАВЕЙ/КОТ)
- **W_recurring** (0–15): Штраф за просрочку повторяющейся задачи

### Распределение по каналам

Диспетчер весов распределяет задачи между МУРАВЬЁМ и КОТОМ, учитывая:
- Текущую загрузку каждого канала (суммарный вес активных задач)
- Сложность новой задачи
- Целевое соотношение нагрузки (50/50 по умолчанию, настраивается)
- Расширяемость для третьего канала

### Визуализация веса

На карточке задачи — индикатор веса. Цвет: зелёный (лёгкая), жёлтая (средняя), красная (тяжёлая). Иконка: чем тяжелее, тем больше «полосок».

## 6. Карточка задачи

### Структура карточки

```
┌─────────────────────────────────────┐
│ Заголовок задачи                    │  ← 1-2 строки, font-medium
│ 🔴 Высокий · 🏷 backend · 🏷 api    │  ← приоритет + теги (до 3)
│ 👤 МУРАВЕЙ · 📅 2 дня назад        │  ← исполнитель + дата
│ ████████░░ 60%                      │  ← прогресс-бар (если есть чеклист)
└─────────────────────────────────────┘
```

### Элементы карточки

- **Заголовок** — 1-2 строки, чёткий и конкретный
- **Приоритет** — цветной индикатор (🔴 критический, 🟠 высокий, 🟡 средний, 🟢 низкий)
- **Теги** — до 3 видимых, остальные как «+N»
- **Исполнитель** — аватар/иконка агента или канала
- **Дата** — когда создана / обновлена
- **Прогресс** — если есть чеклист, показывает процент выполнения
- **Вес** — визуальный индикатор сложности
- **Связи** — иконка если есть зависимости или блокировки

### Детальная панель задачи (Slide-over)

При тапе на карточку — выезжает панель справа (desktop) или снизу (mobile):

- Полное описание (Markdown)
- Чеклист подзадач
- Комментарии / история изменений
- Связанные задачи
- Кнопки действий (переместить, редактировать, удалить, назначить)

### Старение карточек

Карточки, которые долго не двигаются, визуально «стареют»:
- 1-3 дня: нормальный вид
- 4-7 дней: лёгкое желтое свечение рамки
- 8-14 дней: пульсирующая рамка, предупреждение
- 15+ дней: красная рамка, автоматический алерт

## 7. Визуальный дизайн

### Концепция: «Терминал муравейника»

Тёмный минимализм с элементами терминальной эстетики. Не киберпанк (слишком агрессивный), не чистый минимализм (слишком стерильный).

**Ключевые характеристики:**

- Плотный фон `#0a0e1a` — почти чёрный с холодным синим оттенком
- Карточки и колонки приподняты над фоном за счёт контрастного фона `#1e293b`
- Акцентные цвета emerald и amber используются точечно, как сигналы
- Скругления умеренные (8px). Не «пузырьки», не острые углы
- Моноширинный шрифт для чисел, таймстампов, статусов
- Доска «дышит» — пульсирующие индикаторы, тонкие прогресс-бары

### Цветовая схема досок

Каждая доска имеет свой акцентный цвет:

- **ЗАВЛАБ** — emerald `#10b981` (основной акцент)
- **МУРАВЕЙ** — amber `#f59e0b` (рабочий акцент)
- **КОТ** — фиолетовый `#8b5cf6` (личный акцент)

### Семантические цвета

- `--success` `#22c55e` — успех, готово, онлайн
- `--warning` `#f59e0b` — внимание, в работе
- `--error` `#ef4444` — ошибка, критический, блокер
- `--info` `#3b82f6` — информация, назначено

### Типографика

- **Заголовки:** Inter (sans-serif), font-semibold
- **Body:** Inter (sans-serif), font-normal
- **Данные/код:** JetBrains Mono (моноширинный) — только для чисел, таймстампов, статусов
- **Размеры:** 12px (метаданные), 14px (body), 16px (заголовки карточек), 20px (заголовки колонок)

### Пустые состояния

- Пустая колонка: иллюстрация + текст «Перетащите задачу сюда» + подсказка о WIP-лимите
- Пустая доска: онбординг-экран с кнопкой «Создать первую задачу»
- Нет результатов поиска: «Ничего не найдено» с предложением изменить фильтры

### Анимации

- Перемещение карточки: slide 200ms ease-out
- Появление карточки: fade-in 150ms
- Удаление: fade-out 100ms + scale(0.95)
- Стареющая карточка: pulse 2s infinite (при превышении лимита)
- Превышение WIP: shake 300ms + красная подсветка
- Все анимации ≤ 500ms (кроме бесконечных loop-анимаций)

## 8. Мобильный опыт и TWA

### Мобильный Layout

**Проблема:** Текущий Board.tsx рендерит 5 колонок горизонтально — на телефоне это не работает.

**Решение:** Одна колонка + горизонтальный таб-переключатель.

- На экране всегда видна ровно одна колонка
- Переключение между колонками — через горизонтальный ряд чипов вверху
- Свайп влево/вправо для переключения колонок
- Вертикальный скролл карточек внутри колонки
- Sticky-хедер и sticky-табы

**Breakpoint:** `>= 768px` — все 5 колонок (desktop). `< 768px` — одна колонка + табы (mobile).

### Touch-взаимодействия

- **Тап** на карточку → открыть детальную панель (Bottom Sheet на mobile)
- **Long press** → контекстное меню (переместить, редактировать, удалить)
- **Swipe влево** на карточке → быстрое действие (переместить в следующую колонку)
- **Swipe вправо** → вернуть в предыдущую колонку
- **Pull-to-refresh** → синхронизация данных
- **FAB** (плавающая кнопка) → быстрое создание задачи

### Telegram Web App интеграция

**Принцип:** TWA — расширение, а не зависимость. Всё работает и в обычном браузере.

- **MainButton** — «+ Задача» на странице Board
- **BackButton** — навигация назад с детальной панели
- **Haptic feedback** — при перемещении карточки, при превышении WIP
- **Deep links** — `t.me/bot?start=task_123` → открыть карточку задачи
- **Создание задач из TG** — отправить сообщение боту → создаётся карточка в INBOX
- **Inline keyboards** — быстрые действия прямо в чате бота
- **Уведомления** — через TG-ботов при смене статуса, блокировке, старении

### Offline режим

- Лог действий с синхронизацией при восстановлении связи
- Очередь изменений (optimistic updates)
- Индикатор офлайн-режима

## 9. Аналитика и метрики

### Принципы

- Данные первичны — каждая метрика выводится из фактов (changelog + текущее состояние)
- Реальное время — пересчитывается при каждом запросе
- Доказуемость — за каждым числом стоит список задач
- Три уровня: доска (мини-виджеты), проект (полные графики), колония (сводная)

### Метрики

**Кумулятивная диаграмма потока (CFD):** Ось X — время, ось Y — количество задач. Цвета по статусам. Показывает узкие места и скорость завершения.

**Время цикла (Cycle Time):** Сколько времени задача проходит от начала до конца. Гистограмма: среднее / медиана / 90-й перцентиль. Тренд по неделям.

**Пропускная способность (Throughput):** Сколько задач завершается за период (день/неделя). Сравнение по доскам.

**WIP-анализ:** Текущий WIP по колонкам. Сравнение с WIP-лимитами. Тренд WIP по времени.

**Старение задач:** Heatmap задач по времени в колонке. Список «стареющих» задач (>N дней без движения). Автоматические алерты.

**Эффективность каналов:** Сравнение КОТ vs МУРАВЕЙ: количество завершённых задач, среднее время цикла, текущая загрузка.

**Эффективность агентов:** Количество завершённых задач по агентам. Среднее время цикла по агентам. Текущая загрузка.

### Дашборд доски

KPI-карточки прямо над доской:
- WIP (текущий / лимит)
- Throughput за неделю
- Avg Cycle Time
- Количество стареющих задач
- Индикатор здоровья потока (зелёный/жёлтый/красный)

## 10. Безопасность и RBAC

### Слои защиты

1. **Сеть:** HTTPS, CORS, Security Headers (Helmet)
2. **Аутентификация:** JWT access (15 мин) + refresh (7 дней) + TOTP 2FA
3. **Авторизация (RBAC):** admin > operator > viewer
4. **Валидация действий:** State machine переходов, проверка прав
5. **Аудит:** Changelog — кто, что, когда, откуда
6. **Защита от злоупотреблений:** Rate limiting, brute-force protection

### Матрица прав по доскам

- **admin** — полный доступ ко всем доскам, назначение ролей, просмотр аудита
- **operator** — создание/редактирование/перемещение задач на ЗАВЛАБ, просмотр МУРАВЕЙ и КОТ
- **viewer** — только просмотр всех досок

### Валидация переходов

State machine для статусов задач. Запрещены недопустимые переходы (например, BACKLOG → DONE минуя IN PROGRESS).

### Аудит

Все действия логируются: кто, что, когда, с какого IP, какое изменение. Неудачные попытки доступа тоже логируются.

## 11. Автоматизация

### Автоматические действия

- Задача долго не двигается → уведомление + визуальное старение
- Задача заблокирована → уведомление ответственному
- Все задачи в колонке выполнены → уведомление о завершении этапа
- Агент завершил задачу → автоматический переход в TESTING/REVIEW
- Превышен WIP-лимит → блокировка добавления + визуальный алерт

### Definition of Ready (DoR)

Задача не может перейти в TODO/IN PROGRESS без:
- Чёткого заголовка
- Описания (что и зачем)
- Указанного приоритета
- Ответа на вопрос «зачем это нужно?»

### Definition of Done (DoD)

Задача не может перейти в DONE без:
- Выполнения всех пунктов чеклиста
- Подтверждения ответственного
- Отсутствия открытых блокировок

### Повторяющиеся задачи

- Шаблоны задач (task templates)
- Автосоздание по расписанию (ежедневно/еженедельно)
- Штраф за просрочку повторяющейся задачи (W_recurring)

### Интеграция между досками

- ЗАВЛАБ → МУРАВЕЙ: автоматическое создание карточки при назначении агенту
- МУРАВЕЙ → ЗАВЛАБ: эскалация при блокировке > N часов
- КОТ → ЗАВЛАБ: промоция личной задачи в проект
- КОТ → МУРАВЕЙ: делегирование агенту

## 12. Информационная архитектура

### Иерархия целей

```
Уровень 0: Цель компании          — 12 мес.
 └── Уровень 1: Цель лаборатории   — 3-6 мес.
      └── Уровень 2: Проект        — 1-3 мес.
           └── Уровень 3: Эпик     — 1-4 нед.
                └── Уровень 4: Задача — 1-3 дня
                     └── Уровень 5: Подзадача — 2-4 часа
```

Не глубже 5 уровней. Эпик без задач > недели — архивировать. Задача без эпика — нормально.

### User Story Mapping

Колонки = этапы процесса, строки = активности, размер = объём, цвет = тип.

### Связи и блокировки

- **blocks** — зависимость (A не начнётся без B)
- **parent-child** — подзадача (чеклист)
- **related** — смысловая связь
- **escalates** — эскалация между досками

Заблокированная карточка: красный замок, opacity 0.6, не может перейти в IN PROGRESS.

### Фильтры и виды

- По исполнителю (агент, канал)
- По проекту
- По приоритету
- По тегу
- По дате создания / обновления
- По статусу
- По весу (сложности)

## 13. Ритуалы

### Daily Standup (ежедневно, 5 минут)

- Что сделано вчера?
- Что планируется сегодня?
- Есть ли блокировки?

Автоматический отчёт от системы: задачи перемещённые за сутки, заблокированные, стареющие.

### Replenishment (еженедельно, 30 минут)

- Заполнение бэклога
- Приоритизация задач
- Распределение по каналам
- Проверка WIP-лимитов

### Retrospective (ежемесячно, 1 час)

- Что работает хорошо?
- Что можно улучшить?
- Какие метрики изменились?
- Корректировка WIP-лимитов и весов

### Автоматические отчёты

- Ежедневный: перемещённые задачи, блокировки, старение
- Еженедельный: throughput, cycle time, WIP-анализ
- Ежемесячный: сводная аналитика, тренды, рекомендации

## Приложение A: Словарь терминов

- **WIP** — Work In Progress, задачи в работе
- **CFD** — Cumulative Flow Diagram, кумулятивная диаграмма потока
- **Cycle Time** — время от начала до завершения задачи
- **Throughput** — количество завершённых задач за период
- **Swimlane** — горизонтальная дорожка на доске для группировки
- **DoR** — Definition of Ready, критерии готовности задачи
- **DoD** — Definition of Done, критерии завершения задачи
- **TWA** — Telegram Web App
- **RBAC** — Role-Based Access Control
