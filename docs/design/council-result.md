# 🏛 КОНСИЛИУМ: Архитектура артефактов Myrmex Control

> 🟢 STANDARD — архитектурный вопрос
> 2026-05-11

---

## Преамбула

**Контекст:** Myrmex Control — full-stack дашборд (React + Express + JSON DB). `myrmex.json` хранит задачи, проекты, агентов, changelog, настройки. После каскадного брейншторма появились 22+ файлов в `docs/design/`. Артефакты размазаны по `agents/`, `skills/`, `memory/`, `docs/`.

**Симптом:** Нет единой картины. Данные размазаны. Непонятно как связаны задачи, документы, агенты.

**Ограничения:** Не ломать код. Не создавать монолит. Сохранить поиск и навигацию.

**Критерий успеха:** Чёткая архитектура где каждый артефакт — отдельный файл, а связи прослеживаются.

## Позиции экспертов

### artifact-specialist (Primary 3x)

**ТЕЗИС:** Гибрид — лёгкий индекс-файл (artifact-registry.json) как навигационная карта поверх распределённых файлов.

**АРГУМЕНТ:** Единый монолит-JSON — откат к проблеме которую myrmex.json уже демонстрирует: при росте нечитаем, конфликтует при параллельных правках, хрупок. Распределённые файлы уже существуют — ломать бессмысленно. Но без индекса навигация потеряна: 22+ файлов без карты = хаос.

**РИСК:** Без навигационного слоя каждый новый участник тратит непропорционально много времени на поиск. Дублирование и мёртвые ссылки нарастают экспоненциально.

### context-session-specialist (Primary 3x)

**ТЕЗИС:** Гибрид правильный, но artifact-registry.json должен быть не просто навигационной картой, а **версионированным графом связей** с обратной совместимостью и авто-валидацией.

**АРГУМЕНТ:** Плоский индекс решает навигацию, но не решает проблему **связей между артефактами** — какой скилл относится к какому агенту, какая карточка ссылается на какой дизайн-док. Нужен формат графа (nodes + edges), а не плоский список.

**РИСК:** Плоский индекс устаревает быстрее чем его обновляют. Без версионирования — рассинхрон.

### evolve-activator (Primary 3x)

**ТЕЗИС:** artifact-registry.json должен быть не статическим реестром, а **живым графом с инкрементальным обновлением**.

**АРГУМЕНТ:** Регенерируемый индекс (как crawl в поисковых системах), который пересобирается при каждом коммите или по крону. Это даёт актуальную картину без ручного обновления.

**РИСК:** Статический реестр дрейфует. Через месяц половина ссылок неактуальна.

### skill-architect (Secondary 1x)

**ТЕЗИС:** Ни монолитный JSON, ни чистый граф — гибрид. myrmex.json = операционное ядро (задачи/проекты/статусы через API), файлы = знания (дизайн, протоколы, карточки).

**АРГУМЕНТ:** Разные домены имеют разный жизненный цикл. Задачи меняются каждый день, скиллы — раз в неделю, дизайн-документы — раз в месяц. Засовывание всего в один JSON ломает атомарность.

**РИСК:** Монолитный JSON раздувается. Merge-конфликты. Одна ошибка валидации — весь файл падает.

### OWL (Secondary 1x)

**ТЕЗИС:** Граф связей с визуализацией в Myrmex Control (страница Graph).

**АРГУМЕНТ:** Страница Graph сейчас пустая — идеально для визуализации графа артефактов. Задачи ссылаются на документы, документы на задачи, агенты на скиллы. Это уже граф — нужно только его отобразить.

**РИСК:** Без визуализации граф существует только в голове. С визуализацией — это рабочий инструмент.

### incident-commander (Диссентер)

**ЗАМЕЧАНИЯ К КОНСЕНСУСУ:**

1. **Все эксперты предлагают создать ещё один файл** (artifact-registry.json). Это ещё один артефакт который нужно поддерживать. Не превращается ли это в бесконечность?

2. **Авто-генерация индекса** — звучит хорошо, но кто будет писать crawler? Это дополнительная работа которую никто не заложил в roadmap.

3. **Граф связей** — красиво на бумаге, но кто будет поддерживать edges между артефактами? При 24 сотрудниках и сотнях файлов это хаос.

4. **Нет ответа на главный вопрос:** Что делать СЕЙЧАС? Не через месяц, не после написания crawler — а сегодня?

## СИНТЕЗИРОВАННОЕ РЕШЕНИЕ

### Архитектура: Трёхслойная

```
Слой 1: ОПЕРАЦИОННОЕ ЯДРО
  myrmex.json — задачи, проекты, агенты, статусы
  (меняется каждый день, API-driven)

Слой 2: ЗНАНИЯ
  agents/ — карточки сотрудников
  skills/ — скиллы
  memory/ — память лаборатории
  docs/design/ — дизайн-документы
  (меняется редко, file-driven)

Слой 3: НАВИГАЦИЯ
  artifact-registry.json — лёгкий индекс со ссылками
  (регенерируется автоматически)
```

### Что делаем СЕЙЧАС (без кода):

1. **Создать artifact-registry.json** — один файл со ссылками на все артефакты. Структура: id, type, path, tags, links[]

2. **Проставить обратные ссылки** — в каждом документе docs/design/ добавить поле `related_tasks` с ID задач из myrmex.json

3. **Визуализировать в Graph** — использовать пустую страницу Myrmex Control для отображения графа связей

### Что делаем ПОТОМ:

4. **Авто-генерация индекса** — crawler который сканирует файлы и обновляет artifact-registry.json

5. **Валидация связей** — проверка что все ссылки актуальны, нет мёртвых ссылок

6. **Версионирование графа** — хранение истории изменений связей

## Отвергнутые варианты

- **Единый монолитный JSON** — хрупкий, конфликтующий, неподдерживаемый
- **Чистый граф без индекса** — красиво но непрактично без инструментов
- **Полная реструктуризация** — слишком дорого, ломает существующее

---

## Ограничения и риски

- artifact-registry.json требует ручного обновления пока нет crawler'а
- Обратные ссылки нужно проставить вручную (22 файла)
- Граф связей будет полезен только при >10 связях
