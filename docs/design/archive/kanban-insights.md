## Топ-30 инсайтов

### 1. Бэклог — единый источник правды
Задача физически живёт в бэклоге. На досках — только ссылки + статус. Это исключает дублирование и рассинхронизацию между ЗАВЛАБ, МУРАВЕЙ и КОТ.

### 2. Диспетчер весов — сердце системы
W_total = W_priority + W_age + W_deps + W_channel + W_recurring. Пять компонентов дают точную картину сложности. Система сама решает куда направить задачу.

### 3. Старение карточек — визуальный сигнал проблемы
1-3 дня норма, 4-7 жёлтое свечение, 8-14 пульсация, 15+ красный алерт. Задача которая долго не двигается — проблема, а не «отложено».

### 4. WIP-лимиты — жёсткие, не мягкие
Нельзя добавить задачу в колонку если лимит исчерпан. Колонка подсвечивается красным. Это главный принцип Toyota — не брать новое пока не доделал старое.

### 5. Три доски — три разных ритма
ЗАВЛАБ еженедельно (стратегия), МУРАВЕЙ ежедневно (тактика), КОТ по необходимости (личное). Смешение ритмов = хаос.

### 6. Мобильный layout — одна колонка + табы
5 колонок горизонтально не работают на телефоне. Решение: одна колонка + горизонтальный таб-переключатель + свайп между колонками.

### 7. TWA — расширение, а не зависимость
Всё работает и в обычном браузере. isTWA() → включаем MainButton, BackButton, haptic. Никаких блокирующих зависимостей от Telegram API.

### 8. Touch > Drag-and-drop
HTML5 drag-and-drop не работает на мобильных. Решение: тап → Bottom Sheet, long press → контекстное меню, swipe → быстрое перемещение.

### 9. Семантические цвета обязательны
--success, --warning, --error, --info — без них доска «плоская». Текущий accent и secondary имеют одинаковое значение — баг.

### 10. Приоритет кодируется ТОЛЬКО левой границей и точкой
Не заливать всю карточку цветом приоритета — это создаёт визуальный шум. Левая граница 3px + точка перед заголовком.

### 11. Задачи весом 13+ должны декомпозироваться
Если задача весит 13 из 15 — это сигнал: «разбей на части». Автоматическая рекомендация при создании.

### 12. Пустое состояние — приглашение к действию
Каждый empty state содержит: иллюстрацию, заголовок, подсказку, CTA. Не пустота, а онбординг.

### 13. Анимации ≤ 500ms
Все анимации быстрые и осмысленные. Пользователь должен ЧУВСТВОВАТЬ отклик, но не ждать. Исключение — бесконечные loop-анимации (пульсация стареющих).

### 14. ТОЛЬКО Lucide React для иконок
Никаких emoji в навигации. Emoji = визуальный разрыв. Lucide React = единый стиль.

### 15. JetBrains Mono ТОЛЬКО для данных
Моноширинный шрифт для чисел, таймстампов, статусов. Не для body text — убивает читаемость.

### 16. 6 слоёв безопасности
Сеть → Аутентификация → Авторизация → Валидация → Аудит → Защита от злоупотреблений. Каждый слой независим.

### 17. State machine для переходов статусов
Запрещены недопустимые переходы (BACKLOG → DONE минуя IN PROGRESS). Серверная валидация, не только клиентская.

### 18. Аудит логирует НЕУДАЧНЫЕ попытки доступа
Не только успешные действия, но и неудачные. Без этого нет видимости атак.

### 19. Тихие часы 23:00–08:00
Только CRITICAL и P1 алерты. WARNING копится в утренний дайджест. Не будить человека без необходимости.

### 20. Жизненный цикл инсайта
pending → analyzing → testing → validated → applied → confirmed. Или rejected → archived. Каждый инсайт отслеживается.

### 21. A/B тестирование параметров весов
ab_tests.json хранит активные и завершённые тесты. Каждый тест: id, insight_id, parameter, control_value, experiment_value, verdict.

### 22. Канал 3 заложен в архитектуру
Диспетчер весов работает с N каналами, не только двумя. Добавление третьего канала — конфигурация, а не переписывание.

### 23. Deep links на карточки
t.me/bot?start=task_123 → открыть карточку задачи. Критично для интеграции с Telegram.

### 24. Создание задач из Telegram
Отправить сообщение боту → создаётся карточка в INBOX. Без захода в веб-приложение.

### 25. Pull-to-refresh на доске
Нативный жест для синхронизации данных. Ожидаемое поведение на мобильных.

### 26. Виртуализация списка карточек
Критично для колонки DONE где может накопиться 50+ задач. Без виртуализации — лаг.

### 27. Offline: лог действий + очередь изменений
Optimistic updates + синхронизация при восстановлении связи. Индикатор офлайн-режима.

### 28. Кумулятивная диаграмма потока (CFD) — главный график
Ось X — время, ось Y — количество задач, цвета по статусам. Показывает узкие места и скорость завершения.

### 29. Метрика Flow Efficiency
work_time / total_time. Показывает сколько времени задача реально в работе vs ожидании. Цель > 40%.

### 30. Ритуалы важнее инструмента
Daily standup (5 мин), replenishment (30 мин), retrospective (1 час). Без ритуалов доска превращается в мусорку.

## Скрытые фишки (неочевидные идеи)

- **Swimlanes по эпикам** — горизонтальные дорожки на доске ЗАВЛАБ для визуализации проектов
- **Ready-queues** — очередь «готово к следующему этапу», следующий исполнитель сам «вытягивает» задачу
- **Вложенные доски** — карточка «Проект X» → открывает под-доску с задачами проекта
- **Автоматическая эскалация** — задача в BLOCKED > N часов → автоматический переход в ЗАВЛАБ
- **Шаблоны задач** — повторяющиеся паттерны сохраняются как шаблоны
- **Голосовой ввод задач** — через TWA микрофон → текст → карточка в INBOX
- **Шеринг карточки в TG-чат** — отправить ссылку на задачу прямо в чат
- **Heatmap старения** — визуализация какие задачи застревают и где
- **Индикатор здоровья потока** — зелёный/жёлтый/красный на основе метрик
- **Автоматический дайджест** — утренний отчёт: что сделано, что заблокировано, что стареет

## Принципы (выжимка)

- Не брать новое, пока не доделал старое
- Видеть весь поток работы как на ладони
- Задача не просто «сделать X», а «сделать X для Y, чтобы Z»
- Автоматизация помогает, а не мешает
- TWA — расширение, а не зависимость
- Код не трогаем, только концепция
