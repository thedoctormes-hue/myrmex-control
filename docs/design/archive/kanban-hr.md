## 1. Роли и ответственности на досках

### 1.1. Общая карта ролей

| Роль | Доска | Кол-во | Ответственность |
|------|-------|--------|-----------------|
| **Заведующий лабораторией** | ЗАВЛАБ | 1 (Евгений) | Стратегия, приоритеты, финальные решения |
| **Старший агент** | МУРАВЕЙ | 2-3 | Менторство, ревью, эскалации |
| **Агент-исполнитель** | МУРАВЕЙ | 15-18 | Выполнение задач, отчётность |
| **Агент-исследователь** | МУРАВЕЙ | 2-3 | R&D, прототипы, анализ |
| **Личный ассистент** | КОТ | 1-2 | Личные задачи Евгения |
| **Технический писатель** | МУРАВЕЙ | 1-2 | Документация, changelog, ADR |

### 1.2. Доска ЗАВЛАБ — кто что делает

**Заведующий лабораторией (Евгений):**
- Создаёт проекты и эпики
- Устанавливает приоритеты (P0-P4)
- Утверждает Definition of Ready для крупных задач
- Проводит еженедельный обзор и стратегические ретроспективы
- Принимает решения об эскалациях и перераспределении ресурсов
- Финальное ревью и приёмка ключевых задач

**Старшие агенты (2-3 человека):**
- Ревью кода и артефактов младших агентов
- Разрешение блокировок внутри своей зоны ответственности
- Менторство новых агентов
- Участвуют в планировании спринтов
- Эскалация на Евгения только по стратегическим вопросам

**Агенты-исполнители:**
- Выполняют задачи из бэклога МУРАВЕЙ
- Обновляют статусы задач в реальном времени
- Создают задачи-блокировки при обнаружении препятствий
- Участвуют в ежедневных стендапах

### 1.3. Доска МУРАВЕЙ — кто что делает

**Диспетчер колонии (автоматический + ручной надзор):**
- Автоматическая маршрутизация задач через систему весов
- Ребалансировка нагрузки между агентами
- Эскалация просроченных задач
- Мониторинг WIP-лимитов

**Агенты-исполнители:**
- Берут задачи из колонки READY по приоритету (весу)
- Переводят задачи через жизненный цикл: READY -> IN_PROGRESS -> IN_REVIEW -> DONE
- Отмечают прогресс, прикладывают артефакты
- Блокируют задачи при обнаружении зависимостей

**Агенты-исследователи:**
- Работают с задачами типа `research`
- Создают отчёты по результатам исследований
- Прототипируют решения перед передачей исполнителям

### 1.4. Доска КОТ — кто что делает

**Евгений (лично):**
- Управляет личными задачами и идеями
- Планирует обучение и саморазвитие
- Контролирует баланс работы и отдыха

**Личный ассистент (1-2 агента):**
- Напоминает о дедлайнах
- Готовит дайджесты по запросу
- Ведёт личные заметки и списки

### 1.5. Матрица ответственности (RACI)

| Активность | Евгений | Старший агент | Агент-исполнитель | Агент-исследователь |
|-----------|---------|--------------|-------------------|---------------------|
| Создание проекта | **R/A** | C | I | I |
| Приоритизация | **R/A** | C | I | I |
| Выполнение задачи | I | C | **R/A** | R (research) |
| Ревью кода | A | **R** | C | I |
| Эскалация | **R/A** | R | R | R |
| Архивирование | A | R | I | I |
| Онбординг | **R/A** | R | I | I |
| Ретроспектива | **R/A** | R | R | R |

*R = Responsible, A = Accountable, C = Consulted, I = Informed*

## 2. Система онбординга новых агентов

### 2.1. Фазы онбординга

```
ДЕНЬ 0          ДЕНЬ 1-2         ДЕНЬ 3-5         ДЕНЬ 6-10        ДЕНЬ 11-30
┌──────────┐    ┌──────────┐     ┌──────────┐     ┌──────────┐     ┌──────────┐
│ СОЗДАНИЕ │───►│ ОБУЧЕНИЕ │────►│ ПЕРВАЯ   │────►│ САМОСТО- │────►│ ПОЛНАЯ   │
│ ПРОФИЛЯ  │    │ И ДОСТУП │     │ ЗАДАЧА   │     │ ЯТЕЛЬНАЯ │     │ ЗРЕЛОСТЬ │
└──────────┘    └──────────┘     └──────────┘     └──────────┘     └──────────┘
```

### 2.2. День 0 — Создание профиля

**Чеклист:**
- [ ] Создана карточка агента в системе (Agent profile)
- [ ] Назначена роль (исполнитель / исследователь / писатель)
- [ ] Определены навыки (skills) — минимум 3 тега
- [ ] Назначен ментор из числа старших агентов
- [ ] Определена доска и канал (МУРАВЕЙ или КОТ)
- [ ] Установлены лимиты: WIP <= 2 (для новичка)
- [ ] Создан онбординг-эпик на доске ЗАВЛАБ

**Шаблон профиля нового агента:**
```
Имя: [уникальное имя]
Роль: [исполнитель | исследователь | писатель | ассистент]
Навыки: [тег1, тег2, тег3]
Канал: [muravey | kat]
Ментор: [имя старшего агента]
WIP-лимит: 2 (новичок) -> 5 (после 30 дней)
Статус: onboarding
Дата онбординга: [дата]
```

### 2.3. День 1-2 — Обучение и доступ

**Программа обучения:**

1. **Знакомство с системой (2 часа)**
   - Как устроены три доски
   - Жизненный цикл задачи (BACKLOG -> READY -> IN_PROGRESS -> IN_REVIEW -> DONE)
   - Система весов и приоритетов
   - Как работают зависимости и блокировки

2. **Знакомство с процессами (1 час)**
   - Ежедневный стендап: формат, правила
   - Как эскалировать проблемы
   - Как проходит ревью
   - Definition of Ready / Definition of Done

3. **Знакомство с инструментами (1 час)**
   - Как обновлять статусы задач
   - Как создавать задачи
   - Как работать с changelog
   - Как использовать шаблоны задач

4. **Знакомство с командой (30 минут)**
   - Кто есть кто (карта команды)
   - К кому с какими вопросами
   - Правила коммуникации

**Чеклист:**
- [ ] Агент прошёл обучение (отметка в онбординг-эпике)
- [ ] Агент имеет доступ к нужным каналам
- [ ] Агент знает своего ментора
- [ ] Агент понимает DoR и DoD

### 2.4. День 3-5 — Первая задача

**Принцип первой задачи:**
- Низкий приоритет (P3) или учебная задача
- Чёткие критерии готовности
- Оценка: 2-4 часа максимум
- Ментор на связи для вопросов

**Процесс:**
1. Ментор назначает первую задачу из бэклога
2. Агент переводит задачу в IN_PROGRESS
3. Агент выполняет задачу, просит ревью у ментора
4. Ментор проводит ревью, даёт обратную связь
5. Агент исправляет замечания (если есть)
6. Задача переходит в DONE
7. Ментор оценивает: готов ли агент к самостоятельной работе

**Критерии перехода к самостоятельной работе:**
- [ ] Первая задача завершена
- [ ] Ревью пройдено с минимальными замечаниями
- [ ] Агент обновлял статусы вовремя
- [ ] Агент задал минимум 3 содержательных вопроса (не молча!)

### 2.5. День 6-10 — Самостоятельная работа под наблюдением

- WIP-лимит повышен до 3
- Агент берёт задачи из бэклога самостоятельно
- Ментор проводит ревью каждой задачи
- Ежедневный 10-минутный чек-ин с ментором
- Агент участвует в ежедневном стендапе

### 2.6. День 11-30 — Полная зрелость

- WIP-лимит повышен до 5 (стандартный)
- Ментор переходит в режим «по запросу»
- Ревью проводится по стандартному процессу (не обязательно ментором)
- Агент может участвовать в ретроспективах
- Оценка адаптации на 30-й день

### 2.7. Оценка адаптации (30-й день)

| Метрика | Порог «отлично» | Порог «удовлетворительно» | Порог «нужна помощь» |
|---------|-----------------|--------------------------|----------------------|
| Завершённых задач | >= 10 | >= 6 | < 6 |
| Средний cycle time | < 1 дня | < 2 дней | >= 2 дней |
| % задач без замечаний на ревью | > 80% | > 60% | <= 60% |
| Просроченных задач | 0 | <= 2 | > 2 |
| Участие в стендапах | 100% | > 80% | <= 80% |

**Решение по результатам:**
- **Отлично:** полная самостоятельность, возможно повышение до старшего агента
- **Удовлетворительно:** продолжение работы, фокус на слабых зонах
- **Нужна помощь:** продление менторинга на 2 недели, снижение WIP-лимита

## 3. Распределение нагрузки

### 3.1. Принципы распределения

1. **Вес задачи определяет приоритет.** Система весов (W_total) автоматически ранжирует задачи.
2. **Канал определяет маршрутизацию.** Теги `channel:muravey`, `channel:kat`, `channel:any`.
3. **Закрепление за исполнителем.** Если `assignee_id` установлен — задача закреплена.
4. **Балансировка автоматическая.** Диспетчер ребалансирует каждые 5 минут.
5. **Ручное переопределение возможно.** Евгений и старшие агенты могут вручную назначать задачи.

### 3.2. WIP-лимиты по ролям

| Роль | WIP-лимит | Обоснование |
|------|-----------|-------------|
| Новичок (онбординг) | 2 | Фокус на качестве и обучении |
| Агент-исполнитель | 5 | Стандартная загрузка |
| Старший агент | 3 | Часть времени на ревью и менторство |
| Агент-исследователь | 2 | Исследования требуют глубокого фокуса |
| Технический писатель | 3 | Документация + ревью документации |

### 3.3. Правила назначения задач

**Автоматическое назначение (диспетчер):**
- Задача с `channel:any` -> наименее загруженный агент с подходящими навыками
- Задача с `channel:muravey` -> агент канала МУРАВЕЙ с подходящими навыками
- Задача с `channel:kat` -> агент канала КОТ с подходящими навыками
- Задача с тегом `research` -> только агенты-исследователи
- Задача с тегом `docs` -> в первую очередь технические писатели

**Ручное назначение (Евгений / старшие агенты):**
- Критические задачи (P0, P1) — Евгений назначает лично
- Задачи с зависимостями — старший агент координирует
- Междосковые задачи — Евгений определяет канал

### 3.4. Балансировка нагрузки

**Алгоритм:**
```
1. Рассчитать нагрузку каждого агента:
   Load(agent) = Σ W_total всех активных задач (READY + IN_PROGRESS + IN_REVIEW)

2. Рассчитать среднюю нагрузку:
   Load(avg) = Σ Load(all_agents) / count(active_agents)

3. Найти перегруженных (Load > Load(avg) * 1.5)
4. Найти недогруженных (Load < Load(avg) * 0.5)
5. Перенести задачи от перегруженных к недогруженным
   Приоритет переноса: задачи с наименьшим |W_channel| (наименее привязанные к каналу)
6. Максимум 3 переноса за одну ребалансировку
```

**Пороги алертов:**
- Агент с WIP > лимит -> жёлтый алерт
- Агент с WIP > лимит + 2 -> красный алерт + автоматическая блокировка новых назначений
- Разница нагрузки между каналами > threshold (20) -> уведомление диспетчеру

### 3.5. Учёт специализации

Навыки агента (skills) должны соответствовать тегам задачи:

| Навык агента | Подходящие теги задач |
|-------------|----------------------|
| `backend` | backend, api, database, server |
| `frontend` | frontend, ui, css, react |
| `devops` | deploy, infrastructure, ci-cd, monitoring |
| `security` | security, audit, auth, encryption |
| `docs` | documentation, readme, changelog, adr |
| `research` | research, prototype, analysis, spike |
| `testing` | testing, qa, e2e, unit-test |

**Правило:** Задача назначается агенту с совпадением >= 1 тега. Если нет совпадений — задача остаётся в бэклоге с пометкой «нет специалиста».

### 3.6. Перегрузка и недогрузка

**Сценарий: перегрузка канала МУРАВЕЙ**
1. Диспетчер обнаруживает: Load(МУРАВЕЙ) > Load(КОТ) + threshold
2. Находит задачи с `channel:any` на МУРАВЕЕ
3. Переносит наименее привязанные задачи на канал КОТ
4. Уведомляет Евгения о ребалансировке
5. Если перенос невозможен — эскалация на Евгения

**Сценарий: недогрузка агента**
1. Агент завершил все задачи, колонка пуста
2. Диспетчер ищет задачи в READY с подходящими тегами
3. Если задач нет — агент переходит в режим «ожидание»
4. Старший агент может назначить задачу из бэклога вручную
5. Если недогрузка > 2 дней — пересмотр распределения ролей

## 4. Ретроспективы и развитие

### 4.1. Типы ретроспектив

#### Ежедневный стендап (Daily Standup)

**Когда:** Ежедневно, 09:00
**Длительность:** 15 минут
**Формат:** Асинхронный (агенты обновляют статусы до 09:00)

**Структура:**
```
1. Что завершено вчера? (ссылка на задачи в DONE)
2. Что в работе сегодня? (ссылка на задачи в IN_PROGRESS)
3. Блокировки? (если есть — описание + ссылка)
4. Нужна помощь? (конкретный запрос)
```

**Правила:**
- Максимум 3 предложения на пункт
- Блокировки — с конкретным запросом помощи
- Если задача не двигалась 2+ дня — обязательное объяснение

#### Еженедельная ретроспектива (Weekly Retro)

**Когда:** Пятница, 16:00
**Длительность:** 30 минут
**Участники:** Евгений + старшие агенты + по желанию исполнители

**Формат (4L):**
1. **Liked** — что понравилось на этой неделе?
2. **Learned** — что узнали нового?
3. **Lacked** — чего не хватало?
4. **Longed for** — чего хотелось бы?

**Выход:** 2-3 конкретных действия на следующую неделю, оформленные как задачи в бэклоге.

#### Спринтовая ретроспектива (Sprint Retro)

**Когда:** Каждые 2 недели, в конце спринта
**Длительность:** 45 минут
**Участники:** Все активные агенты + Евгений

**Формат:**
1. **Метрики спринта** (10 мин) — throughput, cycle time, blocked ratio
2. **Что прошло хорошо** (10 мин) — конкретные примеры
3. **Что можно улучшить** (10 мин) — конкретные проблемы
4. **Действия** (15 мин) — 1-3 конкретных улучшения с ответственными

**Правило:** Каждое улучшение = задача в бэклоге с тегом `process-improvement`.

#### Стратегическая ретроспектива (Monthly)

**Когда:** Последний понедельник месяца
**Длительность:** 60 минут
**Участники:** Евгений + старшие агенты

**Формат:**
1. **Обзор целей** — движемся ли к целям лаборатории?
2. **Анализ метрик** — тренды за месяц (CFD, throughput, cycle time)
3. **Завершённые проекты** — что закрыли, чему научились
4. **Кадровые вопросы** — кто готов к повышению, кому нужна помощь
5. **Корректировка целей** — при необходимости

### 4.2. Развитие агентов

#### Карьерная лестница

```
Уровень 1: Новичок (Onboarding)
  └── Уровень 2: Исполнитель (Independent)
       └── Уровень 3: Старший агент (Senior)
            └── Уровень 4: Лид направления (Lead)
                 └── Уровень 5: Заместитель завлаба (Deputy)
```

**Критерии перехода:**

| Из -> В | Критерии | Срок минимум |
|--------|----------|-------------|
| Новичок -> Исполнитель | Завершён онбординг, оценка «удовлетворительно» или выше | 30 дней |
| Исполнитель -> Старший | >= 50 завершённых задач, средний cycle time < 1.5 дня, провёл >= 5 ревью | 3 месяца |
| Старший -> Лид | Менторил >= 2 новичков, ведёт >= 1 направление, стратегическое мышление | 6 месяцев |
| Лид -> Заместитель | Управляет >= 2 направлениями, участвует в стратегических решениях | 12 месяцев |

#### План развития (Individual Development Plan)

Для каждого агента старше уровня «Новичок» создаётся IDP:

```
Индивидуальный план развития: [Имя агента]
Дата: [дата]
Текущая роль: [роль]
Целевая роль: [роль]

Сильные стороны:
  1. [сила 1]
  2. [сила 2]

Зоны роста:
  1. [зона 1] -> план: [конкретные действия]
  2. [зона 2] -> план: [конкретные действия]

Цели на квартал:
  1. [цель 1] — метрика: [как измеряем]
  2. [цель 2] — метрика: [как измеряем]

Ментор: [имя]
Следующий обзор: [дата + 3 месяца]
```

### 4.3. Обучение и обмен знаниями

**Еженедельные технические сессии (Tech Talks):**
- Формат: 15-минутный доклад агента на техническую тему
- Частота: раз в неделю, среда 15:00
- Темы: новые инструменты, паттерны, разбор сложных задач
- Записываются в базу знаний (docs/knowledge-base/)

**База знаний:**
- Каждый агент обязан создать минимум 1 статью в базу знаний в месяц
- Статьи рецензируются старшими агентами
- Теги: `onboarding`, `how-to`, `architecture`, `troubleshooting`, `best-practices`

**Парное выполнение (Pair Working):**
- Новичок + старший агент на первых задачах
- Агент-исследователь + агент-исполнитель на сложных задачах
- Формат: совместная работа над задачей, обмен контекстом

## 5. Метрики продуктивности

### 5.1. Индивидуальные метрики

| Метрика | Описание | Формула | Целевое значение |
|---------|----------|---------|-----------------|
| **Throughput** | Завершённых задач в неделю | count(DONE) / неделя | >= 5 (исполнитель), >= 3 (старший) |
| **Cycle Time** | Среднее время от IN_PROGRESS до DONE | avg(completed_at - started_at) | < 1.5 дня |
| **Lead Time** | Среднее время от создания до завершения | avg(completed_at - created_at) | < 3 дней |
| **Review Pass Rate** | % задач, прошедших ревью с первого раза | count(first-pass) / count(reviewed) | > 70% |
| **Blocked Ratio** | % времени в заблокированном состоянии | sum(blocked_time) / sum(total_time) | < 15% |
| **SLA Compliance** | % задач, выполненных в рамках SLA | count(in-SLA) / count(total) | > 90% |
| **Contribution Score** | Взвешенный вклад в проекты | Σ (story_points × priority_multiplier) | Рост 10% м/к м |

### 5.2. Командные метрики

| Метрика | Описание | Формула | Целевое значение |
|---------|----------|---------|-----------------|
| **Team Throughput** | Суммарный throughput команды | Σ throughput всех агентов | Рост 10% м/к м |
| **Average Cycle Time** | Средний cycle time по команде | avg(cycle_time) всех задач | < 2 дней |
| **Blocked Tasks** | Количество заблокированных задач | count(status=BLOCKED) | < 10% от общего числа |
| **WIP Health** | Соотношение текущих WIP к лимитам | sum(current_WIP) / sum(WIP_limits) | 60-80% (не перегрузка, не простой) |
| **Balance Index** | Равномерность нагрузки между каналами | 1 - (std_dev(Load_channels) / avg(Load_channels)) | > 0.7 |
| **Onboarding Success** | % новичков, прошедших онбординг за 30 дней | count(passed) / count(total) | > 80% |
| **Process Adoption** | % задач, прошедших через полный цикл DoR->DoD | count(full_cycle) / count(total) | > 90% |

| Виджет | Тип | Описание |

| Agent Scorecard | Table | Таблица агентов с ключевыми метриками |

### 5.4. Ранжирование агентов

**Итоговый балл агента (Agent Score):**
```
Score = 0.30 × Throughput_norm +
        0.25 × CycleTime_inv_norm +
        0.20 × ReviewPassRate +
        0.15 × SLA_compliance +
        0.10 × Contribution_score_norm
```

Все компоненты нормализуются к диапазону [0, 1].

**Категории:**
- **A (90-100):** Выдающийся результат — кандидат на повышение
- **B (70-89):** Хороший результат — стабильный вклад
- **C (50-69):** Удовлетворительно — есть зоны роста
- **D (0-49):** Ниже ожиданий — нужен план улучшения

## 7. Кадровый резерв и ротация

### 7.1. Кадровый резерв

**Правило 1:1:** На каждую ключевую роль должен быть минимум 1 резервист.

| Ключевая роль | Основной | Резервист | Готовность резерва |
|---------------|----------|-----------|-------------------|
| Старший агент #backend | Агент А | Агент Б | 70% (готов через 1 мес) |
| Старший агент #devops | Агент В | Агент Г | 50% (готов через 3 мес) |
| Технический писатель | Агент Д | Агент Е | 80% (готов сейчас) |
| Агент-исследователь | Агент Ж | — | Нет резерва! |

**Действие:** При отсутствии резерва — немедленно начать обучение кандидата.

### 7.2. Ротация задач

**Цель ротации:** Снижение bus factor, развитие навыков, предотвращение выгорания.

**Правила:**
- Каждый квартал минимум 20% задач агента должны быть вне его основной специализации
- Ротация между backend и frontend — по желанию
- Ротация в devops — обязательно для всех исполнителей (1 задача/квартал)
- Ротация в docs — обязательно для всех (1 статья/месяц)

**Формат ротации:**
1. Агент выбирает задачу из другой области
2. Назначается наставник из целевой области
3. Задача оценивается по стандартным метрикам
4. Результат фиксируется в IDP

### 7.3. Управление выгоранием

**Индикаторы выгорания:**
- Cycle time растёт 2+ недели подряд
- Throughput падает 30%+ от личного среднего
- Агент не участвует в стендапах 3+ дня
- Агент берёт только задачи P3-P4 (избегание сложных)

**Действия при обнаружении:**
1. Старший агент проводит 1-on-1 беседу
2. Снижение WIP-лимита на 50% на 1 неделю
3. Переключение на задачи по интересам (research, обучение)
4. Если не улучшается за 2 недели — эскалация на Евгения

## 8. Эскалации и конфликты

### 8.1. Уровни эскалации

```
Уровень 1: Агент сам решает
  └── Уровень 2: Старший агент помогает
       └── Уровень 3: Евгений принимает решение
            └── Уровень 4: Стратегическое решение (планирование)
```

**Уровень 1 — Агент:**
- Технические вопросы в рамках своей компетенции
- Переход задач между статусами
- Создание задач-блокировок
- Запрос помощи у коллег

**Уровень 2 — Старший агент:**
- Блокировки, которые агент не может решить сам
- Конфликты приоритетов между задачами агента
- Вопросы качества (ревью не пройдено 2+ раза)
- Перераспределение задач внутри своей зоны

**Уровень 3 — Евгений:**
- Конфликты приоритетов между проектами
- Перераспределение ресурсов между каналами
- Кадровые решения (повышение, понижение, исключение)
- Стратегические вопросы (новые направления, закрытие проектов)

**Уровень 4 — Стратегический:**
- Изменение целей лаборатории
- Крупные организационные изменения
- Бюджетные вопросы
- Внешние обязательства

### 8.2. Процедура эскалации

1. **Агент обнаруживает проблему** -> создаёт задачу с тегом `escalation`
2. **Указывает уровень эскалации** -> назначает ответственного
3. **Ответственный рассматривает** -> в течение SLA (P0: 15 мин, P1: 1 час, P2: 4 часа)
4. **Решение фиксируется** -> в комментарии к задаче
5. **Если не решено** -> автоматическая эскалация на уровень выше

### 8.3. Конфликты приоритетов

**Сценарий: два P0 одновременно**
1. Евгений определяет порядок выполнения
2. Второй P0 временно понижается до P1
3. Оба агента уведомлены о приоритете

**Сценарий: агент перегружен, приходит новая P0**
1. Диспетчер определяет наименее приоритетную текущую задачу агента
2. Задача приостанавливается (переходит в BLOCKED с пометкой «приоритетная замена»)
3. Назначается P0
4. Приостановленная задача возвращается в READY

**Сценарий: конфликт между каналами**
1. Задача остаётся на текущем канале
2. Создаётся зеркальная задача на другом канале с пометкой `duplicate-of`
3. Евгений принимает решение при следующем обзоре

### 8.4. Обратная связь

**1-on-1 встречи:**
- Старший агент + исполнитель: раз в 2 недели, 20 минут
- Евгений + старший агент: раз в неделю, 30 минут
- Евгений + исполнитель: раз в месяц, 15 минут

**Формат 1-on-1:**
1. Как дела? (5 мин)
2. Что беспокоит? (5 мин)
3. Обратная связь от руководителя (5 мин)
4. Цели на следующий период (5 мин)

**Правило:** Обратная связь — конкретная, своевременная, направленная на поведение (не на личность).

## Приложение: Глоссарий

| Термин | Определение |
|--------|-------------|
| **WIP** | Work In Progress — задачи в работе |
| **DoR** | Definition of Ready — критерии готовности задачи к работе |
| **DoD** | Definition of Done — критерии завершённости задачи |
| **Cycle Time** | Время от начала работы до завершения |
| **Lead Time** | Время от создания задачи до завершения |
| **Throughput** | Количество завершённых задач за период |
| **SLA** | Service Level Agreement — соглашение о сроках |
| **IDP** | Individual Development Plan — индивидуальный план развития |
| **RACI** | Responsible, Accountable, Consulted, Informed |
| **Bus Factor** | Минимальное число сотрудников, потеря которых приведёт к остановке проекта |
