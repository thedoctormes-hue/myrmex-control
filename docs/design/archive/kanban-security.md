> **Версия:** 1.0 | **Дата:** 2026-05-11 | **Автор:** OWL (Security Engineer)
>
> Этот документ описывает концепцию безопасности для системы канбан-досок Myrmex Control.

## 1. Обзор модели безопасности

### 1.1. Принцип наименьших привилегий

Каждый пользователь получает минимальный набор прав, необходимый для его роли. Повышение привилегий — явное действие администратора, логируемое в аудите.

### 1.2. Слои защиты

```
┌─────────────────────────────────────────────────┐
│ Слой 1: Сеть                                     │
│   HTTPS, CORS, Security Headers (Helmet)         │
├─────────────────────────────────────────────────┤
│ Слой 2: Аутентификация                           │
│   JWT access (15мин) + refresh (7д) + TOTP 2FA  │
├─────────────────────────────────────────────────┤
│ Слой 3: Авторизация (RBAC)                       │
│   admin > operator > viewer                      │
├─────────────────────────────────────────────────┤
│ Слой 4: Валидация действий                       │
│   State machine переходов, проверка прав         │
├─────────────────────────────────────────────────┤
│ Слой 5: Аудит                                    │
│   Changelog: кто, что, когда, откуда             │
├─────────────────────────────────────────────────┤
│ Слой 6: Защита от злоупотреблений                │
│   Rate limiting, brute-force protection          │
└─────────────────────────────────────────────────┘
```

### 1.3. Текущий статус

**Уже реализовано (v1.0):**
- JWT access + refresh tokens с ротацией
- TOTP 2FA (setup/verify/disable)
- RBAC middleware: `requireAuth`, `requireRole`
- Rate limiting: 100 req/min/IP
- Security headers (Helmet: CSP, HSTS)
- Changelog для всех изменений состояния
- HttpOnly cookies с SameSite=strict
- bcrypt хеширование паролей (12 rounds)
- Refresh token theft detection (revoke all при повторном использовании)

**Что требует усиления (этот документ):**
- RBAC не привязан к доскам и проектам (глобальные роли без контекста)
- Нет валидации переходов статусов задач (state machine)
- Нет разграничения доступа на уровне досок
- Rate limiting не дифференцирован по эндпоинтам
- Нет brute-force protection на логин
- Аудит не логирует неудачные попытки доступа

## 2. Права доступа (RBAC)

### 2.1. Роли

| Роль | Описание | Типичные пользователи |
|------|----------|----------------------|
| **admin** | Полный доступ ко всем доскам, настройкам, пользователям | Евгений (владелец) |
| **operator** | Работа с задачами: создание, перемещение, редактирование. Без управления пользователями и настройками системы | AI-агенты, доверенные участники |
| **viewer** | Только просмотр. Без создания, редактирования, перемещения | Гости, стажёры |

### 2.2. Иерархия ролей

```
admin
 └── наследует ВСЕ права operator
      └── наследует ВСЕ права viewer
```

Admin всегда имеет доступ ко всем эндпоинтам. Проверка ролей каскадная: если эндпоинт требует `operator`, то `admin` тоже проходит.

### 2.3. Действующие лица (actors)

Помимо пользовательских ролей, в системе есть системные акторы:

| Актор | Роль | Контекст |
|-------|------|----------|
| **Пользователь (человек)** | admin / operator / viewer | Веб-интерфейс, API |
| **AI-агент** | operator (ограниченно) | API через сервисный токен |
| **Система** | admin (внутренний) | Watchdog, automation engine |
| **Demo-пользователь** | admin | Только в demo mode (read-only операции не должны менять прод-данные) |

## 3. Матрица прав по доскам

### 3.1. Привязка досок к проектам

Каждая доска — это проект с полем `board_type`:

```
Project {
  ...
  board_type: 'zavlab' | 'muravey' | 'kot'
}
```

Права на доску определяются ролью пользователя + типом доски.

### 3.2. Матрица прав: ЗавЛаб

| Действие | admin | operator | viewer |
|----------|-------|----------|--------|
| Просмотр доски | да | да | да |
| Создание задач | да | да | нет |
| Редактирование задач | да | свои | нет |
| Перемещение задач | да | свои | нет |
| Удаление задач | да | нет | нет |
| Создание проектов | да | нет | нет |
| Архивирование проектов | да | нет | нет |
| Управление эпиками | да | да | нет |
| Изменение настроек доски | да | нет | нет |
| Просмотр аудита | да | нет | нет |

### 3.3. Матрица прав: МУРАВЕЙ

| Действие | admin | operator (агент) | viewer |
|----------|-------|-----------------|--------|
| Просмотр доски | да | да (своих) | да |
| Создание задач | да | нет | нет |
| Редактирование задач | да | только статус своих | нет |
| Перемещение задач | да | только своих (todo→in_progress→review) | нет |
| Удаление задач | да | нет | нет |
| Просмотр задач других агентов | да | да | да |
| Назначение задач агентам | да | нет | нет |
| Изменение конфигурации агентов | да | нет | нет |

**Особенность:** Агент (operator) на доске МУРАВЕЙ имеет ограниченные права перемещения — только по прямому потоку (todo → in_progress → review). Не может перемещать в обход состояний.

### 3.4. Матрица прав: КОТ

| Действие | admin | operator | viewer |
|----------|-------|----------|--------|
| Просмотр доски | да | нет | нет |
| Создание задач | да | нет | нет |
| Редактирование задач | да | нет | нет |
| Перемещение задач | да | нет | нет |
| Удаление задач | да | нет | нет |

**Особенность:** Доска КОТ — личная. Только admin (Евгений) имеет полный доступ. Остальные роли не имеют доступа вообще.

### 3.5. Матрица прав: системные операции

| Действие | admin | operator | viewer |
|----------|-------|----------|--------|
| Управление пользователями | да | нет | нет |
| Изменение ролей | да | нет | нет |
| Просмотр аудита | да | нет | нет |
| Экспорт данных | да | нет | нет |
| Изменение настроек безопасности | да | нет | нет |
| Управление TOTP | да (своё) | да (своё) | да (своё) |
| Сброс TOTP другого пользователя | да | нет | нет |

### 3.6. Матрица прав: API-эндпоинты

| Эндпоинт | Метод | Требуемая роль | Доска |
|----------|-------|----------------|-------|

| `/api/tasks/:id/move` | POST | operator+ (с ограничениями) | любая |

| `/api/projects/:id` | DELETE | admin | любая |

| `/api/auth/totp/disable` | POST | admin (любой), auth (свой) | — |

## 4. Аудит

### 4.1. Принцип

**Каждое действие, изменяющее состояние, логируется.** Каждая неудачная попытка доступа логируется. Логи неизменяемы из приложения (только добавление).

### 4.2. Структура записи аудита

Существующий `ChangelogEntry` расширяется для поддержки security-событий:

```
AuditEntry {
  id: string
  timestamp: string
  source: string              // кто инициировал (username, agent-id, system)
  action: string              // create, update, delete, move, login, logout,
                              // access_denied, role_change, totp_enable, ...
  entity_type: string         // task, project, user, auth, board, settings
  entity_id: string
  diff: Record<string, any>   // что изменилось (old → new)
  ip: string                  // IP-адрес
  user_agent: string          // User-Agent браузера/клиента
  success: boolean            // успешно ли выполнено
  reason?: string             // причина отказа (при success=false)
}
```

### 4.3. Что логировать: категории событий

#### Аутентификация и сессии

| Событие | Уровень | Поля |
|---------|---------|------|
| Успешный логин | INFO | username, ip, user_agent, totp_used |
| Неудачный логин | WARN | username, ip, user_agent, reason (wrong_password / user_not_found / invalid_totp) |
| Логут | INFO | username, ip |
| Обновление токена | DEBUG | username, ip |
| Неудачное обновление токена | WARN | ip, reason (expired / revoked / theft_detected) |
| Обнаружение кражи refresh token | CRITICAL | user_id, ip, action_taken (revoke_all) |
| Блокировка аккаунта (brute-force) | CRITICAL | username, ip, attempts, lockout_duration |
| Установка TOTP | INFO | user_id |
| Включение/выключение TOTP | INFO | user_id, action (enable/disable) |
| Смена пароля | INFO | user_id |

#### Управление задачами

| Событие | Уровень | Поля |
|---------|---------|------|
| Создание задачи | INFO | task_id, project_id, title, source |
| Редактирование задачи | INFO | task_id, changed_fields (old → new) |
| Перемещение задачи | INFO | task_id, from_status → to_status, source |
| Неудачное перемещение (нарушение state machine) | WARN | task_id, attempted_transition, user, reason |
| Удаление задачи | WARN | task_id, title, project_id, source |
| Назначение исполнителя | INFO | task_id, old_assignee → new_assignee |
| Добавление зависимости | INFO | task_id, depends_on |
| Снятие блокировки | INFO | task_id, unblocked_by |

#### Управление доступом

| Событие | Уровень | Поля |
|---------|---------|------|
| Создание пользователя | WARN | user_id, username, role, created_by |
| Изменение роли | CRITICAL | user_id, old_role → new_role, changed_by |
| Удаление пользователя | CRITICAL | user_id, username, deleted_by |
| Отказ в доступе (403) | WARN | user, endpoint, required_role, actual_role |
| Отказ в доступе (401) | WARN | endpoint, ip, reason |

#### Системные события

| Событие | Уровень | Поля |
|---------|---------|------|
| Изменение настроек доски | WARN | board_type, changed_settings, changed_by |
| Архивирование проектa | WARN | project_id, archived_by |
| Экспорт данных | WARN | entity_type, exported_by, ip |
| Изменение настроек безопасности | CRITICAL | setting, old_value → new_value, changed_by |
| Запуск/остановка сервиса | INFO | service, action |

### 4.4. Уровни критичности

| Уровень | Описание | Действие |
|---------|----------|----------|
| **DEBUG** | Рутинные операции без влияния на безопасность | Логирование |
| **INFO** | Обычные действия авторизованных пользователей | Логирование |
| **WARN** | Потенциально подозрительные действия, неудачные попытки | Логирование + уведомление админу (если паттерн) |
| **CRITICAL** | Нарушение безопасности, кража токенов, массовые отказы | Логирование + немедленное уведомление админу |

### 4.5. Хранение и ротация

- Аудит-хранится в `myrmex.json` → `changelog[]` (текущий лимит: 1000 записей)
- При достижении лимита — старые записи вытесняются (FIFO)
- **Рекомендация:** для production вынести аудит в отдельный файл `audit.json` с ротацией по размеру (например, 10MB → новый файл)
- Записи аудита нельзя удалить или изменить через API (только добавление)
- Для CRITICAL событий — отдельный файл `security-incidents.json` без лимита вытеснения

### 4.6. Просмотр аудита

- Эндпоинт `GET /api/audit` — только для admin
- Фильтры: `entity_type`, `entity_id`, `source`, `action`, `date_from`, `date_to`, `severity`
- Пагинация: `limit` (max 200), `offset`
- Экспорт: `GET /api/audit/export?format=json|csv` — только для admin

## 5. Валидация при перемещении карточек

### 5.1. State Machine переходов

Перемещение задач между статусами контролируются конечным автоматом. Недопустимые переходы отклоняются с описанием причины.

**Разрешённые переходы:**

```
backlog ──► todo ──► in_progress ──► review ──► done
   ▲          ▲           ▲             ▲
   │          │           │             │
   └──────────┴───────────┴─────────────┴── blocked
```

| Из | В | Кто | Условие |
|----|---|-----|---------|
| `backlog` | `todo` | admin, operator | DoR выполнен (checklist) |
| `todo` | `in_progress` | admin, operator (свой assignee) | Назначен исполнитель |
| `in_progress` | `review` | admin, operator (свой assignee) | DoD выполнен (checklist) |
| `review` | `done` | admin | Одобрено |
| `review` | `in_progress` | admin | Отклонено (с комментарием) |
| *любой* | `blocked` | admin, operator | Указана причина блокировки |
| `blocked` | `todo` | admin, operator | Блокировка снята |
| *любой* | `backlog` | admin | Отправлено в бэклог |
| `done` | `in_progress` | admin | Реоткрытие (с причиной) |

**Запрещённые переходы (возвращают 409 Conflict):**
- `done` → * (кроме `in_progress` для admin — реоткрытие)
- `review` → `todo` (только через `in_progress`)
- `backlog` → `in_progress` (только через `todo`)
- `blocked` → `done` (только через полный цикл)
- `cancelled` → * (терминальное состояние)

### 5.2. Проверки при перемещении

При каждом перемещении (`POST /api/tasks/:id/move`) сервер выполняет:

**1. Проверка аутентификации:**
- Валидный JWT access token? → 401 если нет

**2. Проверка авторизации:**
- Пользователь имеет право перемещать эту задачу?
- Admin — любую. Operator — только свои (assignee_id = user.id)
- → 403 если нет прав

**3. Проверка state machine:**
- Переход из текущего статуса в целевой разрешён?
- → 409 если нет, с телом: `{ "error": "invalid_transition", "allowed": ["todo", "blocked"] }`

**4. Проверка зависимостей:**
- При переходе в `in_progress`: все ли зависимости в статусе `done`?
- → 409 если есть незавершённые зависимости, с телом: `{ "error": "dependencies_not_met", "blocking": ["task-123"] }`

**5. Проверка DoD (Definition of Done):**
- При переходе в `review`: все ли обязательные checklist items отмечены?
- → 409 если нет, с телом: `{ "error": "dod_not_met", "missing": ["Тесты", "Документация"] }`

**6. Проверка DoR (Definition of Ready):**
- При переходе из `backlog` в `todo`: заполнены ли обязательные поля (title, description, priority)?
- → 400 если нет

**7. Проверка WIP-лимитов:**
- При переходе в `in_progress`: не превышен ли WIP-лимит для исполнителя?
- По умолчанию: max 3 задачи в `in_progress` на одного агента
- → 409 если превышен

### 5.3. Формат ошибки валидации

```json
{
  "error": "invalid_transition",
  "message": "Переход из 'review' в 'todo' не разрешён",
  "current_status": "review",
  "requested_status": "todo",
  "allowed_transitions": ["done", "in_progress"],
  "conditions": {
    "in_progress": "Требуется причина отклонения (rejection_reason)"
  }
}
```

### 5.4. Логирование перемещений

Каждое перемещение (успешное и неуспешное) логируется:

```
Успешное:
  action: "move", entity_type: "task", entity_id: "task-123"
  diff: { "from": "todo", "to": "in_progress" }
  source: "agent-owl", ip: "192.168.1.1", success: true

Неуспешное:
  action: "move_denied", entity_type: "task", entity_id: "task-123"
  diff: { "from": "review", "to": "todo", "reason": "invalid_transition" }
  source: "agent-owl", ip: "192.168.1.1", success: false
```

## 6. Защита от злоупотреблений

### 6.1. Rate Limiting (дифференцированный)

Текущий rate limit (100 req/min/IP) — слишком грубый. Нужна дифференциация по эндпоинтам:

| Категория | Эндпоинты | Лимит | Окно |
|-----------|-----------|-------|------|
| **Аутентификация** | `/api/auth/login`, `/api/auth/setup` | 5 | 1 минута |
| **TOTP** | `/api/auth/totp/*` | 3 | 1 минута |
| **Запись** | POST, PUT, DELETE на `/api/tasks`, `/api/projects` | 30 | 1 минута |
| **Перемещение** | POST `/api/tasks/:id/move` | 20 | 1 минута |
| **Чтение** | GET на все `/api/*` | 120 | 1 минута |
| **Аудит** | `/api/audit` | 10 | 1 минута |
| **State** | `/api/state` | 30 | 1 минута |

**Заголовки ответа:**
```
X-RateLimit-Limit: 5
X-RateLimit-Remaining: 3
X-RateLimit-Reset: 1715431200
Retry-After: 45        (при превышении)
```

### 6.2. Brute-force protection на логин

**Текущий статус:** нет защиты от перебора паролей.

**Предлагаемая схема:**

```
1-3 неудачные попытки: без задержки, ответ 401
4-5 неудачных попыток: задержка 2с перед ответом
6+ неудачных попыток: блокировка IP на 15 минут, ответ 429
```

**Реализация:**
- In-memory map: `Map<ip, { count, firstAttempt, lockedUntil }>`
- Очистка записей старше 1 часа (по таймеру)
- Логирование каждой неудачной попытки (WARN)
- Логирование блокировки (CRITICAL)
- При успешном логине — счётчик IP сбрасывается

### 6.3. Проверка прав на каждое действие

**Принцип:** проверка прав происходит на уровне эндпоинта, а не только middleware.

**Текущая проблема:** `requireRole` middleware проверяет только глобальную роль. Не проверяет:
- Является ли пользователь исполнителем задачи?
- Имеет ли пользователь доступ к доске?
- Не заблокирован ли пользователь?

**Предлагаемая схема:**

```
1. requireAuth — JWT валиден? (middleware)
2. requireRole — роль достаточна? (middleware)
3. requireBoardAccess — есть ли доступ к доске задачи? (handler)
4. requireTaskOwnership — является ли пользователь исполнителем? (handler, для operator)
5. validateTransition — допустим ли переход? (handler)
```

### 6.4. Защита от CSRF

**Текущий статус:** SameSite=strict cookies + CORS — базовая CSRF-защита есть.

**Дополнительные меры:**
- Для мутационных эндпоинтов (POST/PUT/DELETE) проверять заголовок `Origin` или `X-Requested-With`
- Не использовать GET для изменения состояния (уже соблюдается)

### 6.5. Защита от инъекций

**JSON-first архитектура** — SQL-инъекции невозможны (нет SQL).

**Потенциальные риски:**
- **NoSQL инъекция через query params:** `GET /api/tasks?status[$gt]=` — если ORM/драйвер интерполирует параметры
  - **Митигация:** строгая валидация query params, приведение к известным enum-значениям
- **Path traversal через file API:** `GET /api/files?name=../../etc/passwd`
  - **Митигация:** нормализация пути, проверка что файл внутри `inbox/` или `outbox/`
- **JSON payload overflow:** очень большие тела запросов
  - **Митигация:** `express.json({ limit: '100kb' })` — уже есть дефолтный лимит

### 6.6. Защита от злоупотреблений через API агентов

AI-агенты работают через API. Риски:

| Риск | Митигация |
|------|-----------|
| Агент создаёт слишком много задач | Rate limit на создание (10/мин на агента) |
| Агент перемещает чужие задачи | Проверка assignee_id в handler |
| Агент удаляет задачи | Запрет DELETE для operator |
| Агент циклически перемещает задачу | Подсчёт перемещений: max 10 перемещений за минуту на задачу |
| Агент обходит state machine | Серверная валидация переходов (не доверять клиенту) |

### 6.7. Защита от злоупотреблений через массовые операции

| Сценарий | Митигация |
|----------|-----------|
| Массовое удаление задач | Нет batch DELETE. Только по одной задаче. |
| Массовое перемещение | Нет batch move. Только по одной задаче. |
| Создание 1000 задач | Rate limit + лимит 100 задач на проект (настраиваемый) |
| Спам в комментариях/описаниях | Лимит длины: title 200 символов, description 5000 символов |

## 7. Безопасность аутентификации

### 7.1. JWT

**Текущий статус:** access 15мин, refresh 7дней, ротация refresh — хорошо.

**Рекомендации:**
- JWT secret должен быть из env var (`JWT_SECRET`), не генерироваться на старте
  - При рестарте все сессии сбрасываются — приемлемо для single-instance
  - При масштабировании — общий secret через env или JWKS
- Access token хранится в памяти клиента (не в localStorage — уязвим к XSS)
- Refresh token — HttpOnly cookie, путь `/api/auth/refresh` only
- При обнаружении кражи refresh token — отзыв ВСЕХ токенов пользователя (уже реализовано)

### 7.2. TOTP 2FA

**Текущий статус:** TOTP setup/verify/disable — реализовано.

**Рекомендации:**
- TOTP secret хранится в `myrmex.json` в открытом виде (поле `totp_secret`)
  - **Риск:** при компрометации файла — злоумышленник может генерировать коды
  - **Митигация:** шифровать TOTP secret симметричным ключом из `TOTP_ENCRYPTION_KEY` env var
- После включения TOTP — невозможен повторный вход без кода
- Отключение TOTP требует подтверждения текущим TOTP-кодом (или паролем + задержка 24ч)
- Backup codes: сгенерировать 10 одноразовых кодов при включении TOTP

### 7.3. Пароли

**Текущий статус:** bcrypt, 12 rounds, мин. 8 символов — хорошо.

**Рекомендации:**
- Добавить проверку на слабые пароли (top-10000 список)
- Минимальная длина 12 символов для admin
- Индикатор прочности пароля на UI
- Кнопка «Показать пароль» на формах ввода
- При смене пароля — отзыв всех refresh tokens (кроме текущей сессии)

### 7.4. Управление сессиями

- Список активных сессий в профиле пользователя
- Возможность отозвать конкретную сессию (по token ID)
- Возможность отозвать все сессии («Выйти везде»)
- Автоматический logout после 30 минут неактивности (client-side)
- Максимум 5 активных сессий на пользователя

## 8. Рекомендации по реализации

### 8.1. Приоритеты

| Приоритет | Задача | Сложность | Эффект |
|-----------|--------|-----------|--------|
| **P0** | Серверная валидация state machine переходов | Средняя | Критично — сейчас любой переход принимается |
| **P0** | Проверка прав на уровне handler (не только middleware) | Средняя | Критично — operator может двигать чужие задачи |
| **P0** | Brute-force protection на логин | Низкая | Критично — нет защиты от перебора |
| **P0** | Логирование неудачных попыток доступа | Низкая | Критично — нет видимости атак |
| **P1** | Дифференцированный rate limiting | Средняя | Высокий — защита от злоупотреблений |
| **P1** | Разграничение доступа по доскам (КОТ — только admin) | Средняя | Высокий — утечка личных данных |
| **P1** | Шифрование TOTP secret | Средняя | Высокий — защита при компрометации файла |
| **P1** | WIP-лимиты при перемещении | Низкая | Средний — предотвращение перегрузки агентов |
| **P2** | Расширенный аудит (ip, user_agent, success) | Средняя | Средний — улучшение расследования инцидентов |
| **P2** | Управление сессиями (список, отзыв) | Высокий | Средний — контроль доступа |
| **P2** | Backup codes для TOTP | Средняя | Средний — восстановление доступа |
| **P2** | Экспорт аудита (JSON/CSV) | Низкая | Низкий — удобство анализа |

### 8.2. Чеклист безопасности для code review

```
[ ] Входные данные валидируются (тип, длина, enum)?
[ ] Ошибки не раскрывают внутреннюю структуру (нет stack trace в ответе)?
[ ] Действие логируется в changelog?
[ ] Неудачные попытки логируются?
[ ] Rate limit применяется к эндпоинту?
[ ] Нет SQL/NoSQL инъекций (параметризованные запросы)?
[ ] Нет XSS (экранирование вывода, CSP)?
[ ] Нет CSRF (SameSite cookies, Origin check)?
[ ] Секреты не попадают в git?

```

### 8.3. Мониторинг безопасности

Ключевые метрики для отслеживания:

## Приложение B: State Machine переходов

```
                    ┌─────────────────────────────────────────┐
                    │                                         │
                    ▼                                         │
┌───────┐    ┌───────┐    ┌────────────┐    ┌────────┐    ┌───────┐
│backlog│───►│ todo  │───►│in_progress │───►│ review │───►│ done  │
└───────┘    └───────┘    └────────────┘    └────────┘    └───────┘
   ▲            ▲              ▲                ▲             │
   │            │              │                │             │
   │            └──────────────┴────────────────┴─────────────┘
   │         (возврат из любого статуса — только admin)
   │
   └──────────────────────────────────────────────────────────┘
   (возврат из blocked — admin, operator)

Из любого статуса → blocked (admin, operator с причиной)
blocked → todo (admin, operator при снятии блокировки)
done → in_progress (admin, реоткрытие с причиной)
```
