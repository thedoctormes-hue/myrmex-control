## 1. Обзор системы артефактов

### 1.1. Определение

**Артефакт канбан-системы** — любая сущность, описывающая поведение, структуру или состояние доски. Код — не артефакт. Данные в `myrmex.json` — не артефакт. Описание правил, шаблонов, хуков, карточек — артефакт.

### 1.2. Пять видов артефактов

| Вид | Где живёт | Что описывает | Формат |
|-----|-----------|---------------|--------|
| **Скиллы** | `library[]` с `type: "skill"` | Навыки агентов: инструкции, проверки, протоколы | Markdown + frontmatter |
| **Хуки** | `library[]` с `type: "hook"` | Автоматизация: триггеры, действия, условия | Shell / Markdown + frontmatter |
| **Карточки** | `tasks[]` | Единицы работы на досках | JSON-объект по схеме task |
| **Конфигурация** | `settings`, `workspace` | Настройки досок, веса, лимиты, политики | JSON |
| **Знания** | `library[]` с `type: "knowledge"` | Паттерны, инсайты, протоколы, пост-мортемы | Markdown + frontmatter |

### 1.3. Принципы

1. **Один артефакт = одна сущность** — скиллы, хуки и знания хранятся в `library[]`, карточки — в `tasks[]`, конфигурация — в `settings`.
2. **Единственный источник правды** — если два артефакта описывают одно и то же, это ошибка.
3. **Frontmatter обязателен** — для скиллов, хуков и знаний в поле `content`.
4. **Идемпотентность хуков** — повторный запуск хука не должен ломать состояние.
5. **Аудит через changelog** — каждое действие артефакта записывается в `changelog[]`.

## 2. Три доски — три контекста

### 2.1. ЗАВЛАБ — Стратегическая доска

| Параметр | Значение |
|----------|----------|
| **Владелец** | Евгений + AI-агенты |
| **Цель** | Управление лабораторией: проекты, эксперименты, инфраструктура |
| **Канал** | Веб-интерфейс Myrmex Control |
| **Аудитория** | Команда лаборатории |
| **Весовой канал** | `channel: "zavlab"` |
| **Цвет доски** | `#059669` (emerald) |
| **Иконка** | `🏥` |

**Типичные задачи:** запуск проекта, деплой сервиса, аудит безопасности, планирование спринта.

**WIP-лимиты:** backlog=50, todo=10, in_progress=5, review=5, done=без лимита.

### 2.2. МУРАВЕЙ — Тактическая доска

| Параметр | Значение |
|----------|----------|
| **Владелец** | AI-агенты |
| **Цель** | Работа агентов: задачи, статусы, производительность |
| **Канал** | Telegram #1 |
| **Аудитория** | Евгений (надзор), агенты (исполнение) |
| **Весовой канал** | `channel: "muravey"` |
| **Цвет доски** | `#d97706` (amber) |
| **Иконка** | `🐜` |

**Типичные задачи:** выполнение скилла, обработка инцидента, генерация отчёта, мониторинг.

**WIP-лимиты:** backlog=100, todo=20, in_progress=8, review=8, done=без лимита.

### 2.3. КОТ — Личная доска

| Параметр | Значение |
|----------|----------|
| **Владелец** | Евгений (личное) |
| **Цель** | Личные задачи, идеи, саморазвитие |
| **Канал** | Telegram #2 |
| **Аудитория** | Только Евгений |
| **Весовой канал** | `channel: "kot"` |
| **Цвет доски** | `#7c3aed` (violet) |
| **Иконка** | `🐱` |

**Типичные задачи:** прочитать статью, подготовить выступление, личный проект, идея для лаборатории.

**WIP-лимиты:** backlog=30, todo=5, in_progress=3, review=3, done=без лимита.

### 2.4. Междосочные переходы

```
ЗАВЛАБ ──[назначение агенту]──> МУРАВЕЙ
МУРАВЕЙ ──[эскалация]──────────> ЗАВЛАБ
КОТ ──────[идея → проект]──────> ЗАВЛАБ
```

Каждый переход — создание новой задачи на целевой доске со ссылкой на исходную. Исходная задача получает тег `mirrored`.

## 3. Структура артефактов

### 3.1. Скиллы (Skills)

Скиллы хранятся в `library[]` с `type: "skill"`. Каждый скилл — инструкция для агента.

**Обязательные поля:**

```json
{
  "id": "skill-<uuid>",
  "type": "skill",
  "name": "deploy-check",
  "description": "Проверка готовности к деплою",
  "content": "# Скилл: deploy-check\n\n## Тригрер\n...\n\n## Шаги\n1. ...\n2. ...",
  "filePath": null,
  "tags": ["deploy", "checklist", "infrastructure"],
  "category": "devops",
  "createdAt": "2026-05-11T00:00:00.000Z",
  "updatedAt": "2026-05-11T00:00:00.000Z"
}
```

**Категории скиллов:**

| Категория | Описание | Примеры |
|-----------|----------|---------|
| `devops` | Деплой, инфраструктура, мониторинг | deploy-check, vpnscan, auto-deploy |
| `security` | Безопасность, аудит, обход | security-audit, dpi-bypass, security-sweep |
| `content` | Генерация контента, маркетинг | hype-protocol, metrics-storyteller |
| `workflow` | Управление процессами, планирование | council, cascade-brainstorm, sprint-planner |
| `maintenance` | Обслуживание, очистка, бэкапы | clean-backups, fix-gitignore, project-health |

### 3.2. Хуки (Hooks)

Хуки хранятся в `library[]` с `type: "hook"`. Каждый хук — автоматическое действие по триггеру.

**Обязательные поля:**

```json
{
  "id": "hook-<uuid>",
  "type": "hook",
  "name": "auto-escalate-stale",
  "description": "Эскалация задач старше 14 дней",
  "trigger": "schedule:0 9 * * *",
  "content": "#!/bin/bash\n# Хук: auto-escalate-stale\n# Ежедневно в 09:00 MSK\n...",
  "tags": ["automation", "escalation", "maintenance"],
  "enabled": true,
  "createdAt": "2026-05-11T00:00:00.000Z",
  "updatedAt": "2026-05-11T00:00:00.000Z"
}
```

**Типы триггеров:**

| Тип | Формат | Описание |
|-----|--------|----------|
| `schedule` | `cron-выражение` | По расписанию |
| `event:task:create` | Событие создания задачи | При создании новой задачи |
| `event:task:move` | Событие перемещения | При смене статуса задачи |
| `event:task:stale` | Событие старения | Когда задача старее N дней |
| `event:board:overflow` | Событие переполнения | Когда WIP-лимит превышен |
| `manual` | Ручной запуск | Только по вызову оператора |

### 3.3. Карточки (Tasks)

Карточки хранятся в `tasks[]`. Каждая карточка — единица работы.

**Стандартная структура:**

```json
{
  "id": "task-<uuid>",
  "projectId": "<project-uuid>",
  "title": "Краткое название задачи",
  "description": "Подробное описание в Markdown",
  "status": "backlog",
  "priority": "medium",
  "assigneeId": "<agent-uuid>",
  "parentId": null,
  "dependencies": [],
  "tags": ["deploy", "urgent"],
  "dueDate": null,
  "weight": 0,
  "channel": "zavlab",
  "artifactLinks": {
    "skills": ["skill-<uuid>"],
    "hooks": ["hook-<uuid>"],
    "knowledge": ["knowledge-<uuid>"]
  },
  "createdAt": "2026-05-11T00:00:00.000Z",
  "updatedAt": "2026-05-11T00:00:00.000Z",
  "completedAt": null
}
```

**Расширенные поля (не в текущей схеме, рекомендация к добавлению):**

| Поле | Тип | Описание |
|------|-----|----------|
| `weight` | `integer` | Вычисленный вес для диспетчера |
| `channel` | `string` | Канал распределения: `zavlab`, `muravey`, `kot` |
| `artifactLinks` | `object` | Ссылки на связанные артефакты (скиллы, хуки, знания) |
| `recurring` | `object` | Настройки повторения (для рекуррентных задач) |
| `blockedReason` | `string` | Причина блокировки (заполняется при `status: "blocked"`) |

### 3.4. Конфигурация (Settings)

Конфигурация досок хранится в `settings` и расширяется секцией `kanban`:

```json
{
  "settings": {
    "theme": "dark",
    "language": "ru",
    "refreshIntervalSec": 30,
    "notificationsEnabled": true,
    "kanban": {
      "boards": {
        "zavlab": {
          "wipLimits": { "backlog": 50, "todo": 10, "in_progress": 5, "review": 5 },
          "autoAssign": false,
          "escalationDays": 14
        },
        "muravey": {
          "wipLimits": { "backlog": 100, "todo": 20, "in_progress": 8, "review": 8 },
          "autoAssign": true,
          "escalationDays": 7
        },
        "kot": {
          "wipLimits": { "backlog": 30, "todo": 5, "in_progress": 3, "review": 3 },
          "autoAssign": false,
          "escalationDays": 30
        }
      },
      "weights": {
        "priority": { "critical": 40, "high": 30, "medium": 20, "low": 10, "background": 0 },
        "ageStepDays": 2,

        "depMax": 15,
        "channel": { "muravey": 5, "zavlab": 0, "kot": -5 },
        "recurringPenalty": 15
      }
    }
  }
}
```

### 3.5. Знания (Knowledge)

Знания хранятся в `library[]` с `type: "knowledge"`. Каждый элемент знания — паттерн, инсайт, протокол или пост-мортем.

```json
{
  "id": "knowledge-<uuid>",
  "type": "knowledge",
  "name": "postmortem-2026-05-10",
  "description": "Пост-мортем: падение VPN-сервера",
  "content": "# Пост-мортем\n\n## Инцидент\n...\n## Причина\n...\n## Решение\n...\n## Уроки\n...",
  "tags": ["postmortem", "vpn", "incident"],
  "category": "postmortem",
  "createdAt": "2026-05-11T00:00:00.000Z",
  "updatedAt": "2026-05-11T00:00:00.000Z"
}
```

**Категории знаний:**

| Категория | Описание |
|-----------|----------|
| `pattern` | Повторяющийся паттерн решения задач |
| `insight` | Инсайт из анализа данных или опыта |
| `protocol` | Протокол действий (CDR, эскалация, деплой) |
| `postmortem` | Пост-мортем инцидента |
| `decision` | Архитектурное решение (ADR) |

## 5. Хуки автоматизации

### 5.1. Хук: Автораспределение по весам

| Параметр | Значение |
|----------|----------|
| **ID** | `hook-weight-router` |
| **Триггер** | `event:task:create` |
| **Название** | weight-router |
| **Описание** | При создании задачи вычисляет вес и определяет целевую доску |

**Логика:**
1. Вычислить `W_total = W_priority + W_age + W_deps + W_channel + W_recurring`
2. Если `W_total >= 50` → `channel: "zavlab"` (стратегическая)
3. Если `25 <= W_total < 50` → `channel: "muravey"` (тактическая)
4. Если `W_total < 25` → оставить текущий канал
5. Записать в `changelog`: `{ action: "auto-assign", from: null, to: <channel> }`

### 5.2. Хук: Эскалация устаревших задач

| Параметр | Значение |
|----------|----------|
| **ID** | `hook-auto-escalate` |
| **Триггер** | `schedule:0 9 * * *` (ежедневно в 09:00 MSK) |
| **Название** | auto-escalate-stale |
| **Описание** | Находит задачи старше N дней и повышает приоритет |

**Логика:**
1. Найти все задачи где `status != "done"` и `createdAt < now() - N дней`
2. Для каждой задачи:
   - Если `priority == "low"` → установить `priority: "medium"`, добавить тег `auto-escalated`
   - Если `priority == "medium"` → установить `priority: "high"`, добавить тег `auto-escalated`
   - Если `priority == "high"` → установить `priority: "critical"`, добавить тег `auto-escalated`
   - Если `priority == "critical"` → добавить тег `needs-attention`, уведомить владельца доски
3. Записать каждое изменение в `changelog`

**Параметры (из конфигурации):**

| Параметр | ЗАВЛАБ | МУРАВЕЙ | КОТ |
|----------|--------|---------|-----|
| `escalationDays` | 14 | 7 | 30 |

### 5.3. Хук: Проверка WIP-лимитов

| Параметр | Значение |
|----------|----------|
| **ID** | `hook-wip-guard` |
| **Триггер** | `event:task:move` |
| **Название** | wip-limit-guard |
| **Описание** | Блокирует перемещение в колонку при превышении WIP-лимита |

**Логика:**
1. При перемещении задачи в колонку — проверить текущее количество задач в колонке
2. Если количество >= WIP-лимит:
   - Отменить перемещение
   - Добавить тег `wip-blocked` к задаче
   - Создать уведомление: "Колонка <имя> на доске <доска> переполнена (<N>/<лимит>)"
   - Предложить оператору: переместить существующую задачу или увеличить лимит
3. Записать в `changelog`: `{ action: "wip-blocked", column: <имя>, limit: <N> }`

### 5.4. Хук: Автоматическое завершение зависимых

| Параметр | Значение |
|----------|----------|
| **ID** | `hook-auto-complete` |
| **Триггер** | `event:task:move` (при перемещении в `done`) |
| **Название** | auto-complete-dependents |
| **Описание** | Проверяет, можно ли разблокировать зависимые задачи |

**Логика:**
1. При перемещении задачи в `done` — найти все задачи, которые зависят от текущей
2. Для каждой зависимой задачи:
   - Проверить, все ли её зависимости теперь в `done`
   - Если да — переместить из `blocked` в `todo`, добавить тег `auto-unblocked`
   - Уведомить исполнителя: "Задача <title> разблокирована"
3. Записать в `changelog`

### 5.5. Хук: Зеркалирование между досками

| Параметр | Значение |
|----------|----------|
| **ID** | `hook-mirror` |
| **Триггер** | `event:task:create` + тег `mirror-to:<board>` |
| **Название** | board-mirror |
| **Описание** | Создаёт зеркальную задачу на другой доске |

**Логика:**
1. При создании задачи с тегом `mirror-to:muravey` — создать копию на доске МУРАВЕЙ
2. Связать копии через `artifactLinks.mirrored: [<task-id>]`
3. При обновлении статуса оригинала — синхронизировать статус копии
4. Копия получает тег `mirrored` и ссылку на оригинал

### 5.6. Хук: Ежедневный дайджест

| Параметр | Значение |
|----------|----------|
| **ID** | `hook-daily-digest` |
| **Триггер** | `schedule:0 9 * * *` (ежедневно в 09:00 MSK) |
| **Название** | daily-digest |
| **Описание** | Формирует ежедневный отчёт по всем доскам |

**Содержимое дайджеста:**
- Задачи, созданные за последние 24 часа (по доскам)
- Задачи, перемещённые в `done` (по доскам)
- Задачи в статусе `blocked` (все доски)
- Задачи старше escalationDays (по доскам)
- WIP-статус: заполненность колонок
- Топ-3 задачи по весу на каждой доске

### 5.7. Хук: Очистка завершённых

| Параметр | Значение |
|----------|----------|
| **ID** | `hook-cleanup-done` |
| **Триггер** | `schedule:0 2 * * 0` (каждое воскресенье в 02:00 MSK) |
| **Название** | cleanup-done-tasks |
| **Описание** | Архивирует задачи в `done` старше 30 дней |

**Логика:**
1. Найти все задачи где `status == "done"` и `completedAt < now() - 30 дней`
2. Переместить в `status: "archived"`
3. Записать в `changelog`
4. Сформировать отчёт: "Архивировано N задач"
