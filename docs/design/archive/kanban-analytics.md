## 1. Обзор системы

### Принципы

- **Данные первичны.** Каждая метрика выводится из фактов (changelog + текущее состояние), а не из кэша или агрегатов.
- **Реальное время.** Аналитика пересчитывается при каждом запросе из актуального состояния myrmex.json.
- **Доказуемость.** За каждым числом стоит список задач, которые его формируют. Пользователь может «провалиться» от метрики к конкретным задачам.
- **Три уровня детализации:**
  - **Доска** — мини-виджеты прямо на канбан-странице (KPI-карточки, sparkline)
  - **Проект** — полные графики по одному проекту (CFD, cycle time, throughput)
  - **Колония** — сводная аналитика по всем доскам (сравнение ЗавЛаб vs МУРАВЕЙ vs КОТ)

## 2. Источники данных

### Первичные данные

| Источник | Поля | Использование |
|----------|------|---------------|
| `Task[]` | `id`, `project_id`, `status`, `priority`, `assignee_id`, `created_at`, `updated_at`, `completed_at` | Текущее состояние, WIP, распределение |
| `ChangelogEntry[]` | `id`, `timestamp`, `action`, `entity_type`, `entity_id`, `diff` | История переходов, CFD, cycle time, throughput |
| `Agent[]` | `id`, `name`, `status`, `project_id`, `current_task_id` | Загрузка агентов, эффективность |
| `Project[]` | `id`, `name`, `status` | Группировка по доскам |

### Вычисляемые поля

Для каждой задачи на лету вычисляются:

| Поле | Формула | Описание |
|------|---------|----------|
| `cycle_time_hours` | `completed_at - created_at` (в часах) | Полное время жизни задачи |
| `lead_time_hours` | `completed_at - (дата первого перехода из backlog)` | Время от принятия до завершения |
| `time_in_status_hours` | `updated_at - (дата последнего перехода в этот статус)` | Текущее время в текущем статусе |
| `age_hours` | `now - created_at` | Возраст задачи |
| `is_stale` | `time_in_status_hours > threshold` | Задача застряла |

### Ключевое допущение

Для точного расчёта CFD и cycle time нужна история переходов между статусами. Сейчас changelog содержит `diff` с изменениями, но структура `diff` при перемещении (`move`) содержит `{ from: oldStatus, to: newStatus }` — этого достаточно.

**Рекомендация:** при каждом изменении статуса задачи записывать в changelog структурированный diff:
```json
{
  "field": "status",
  "from": "todo",
  "to": "in_progress"
}
```

## 3. Кумулятивная диаграмма потока (CFD)

### Что это

CFD (Cumulative Flow Diagram) — основной инструмент анализа потока в Kanban. Показывает, как задачи накапливаются в каждом статусе во времени.

### Визуализация

```
Кол-во задач
    │
 40 ┤                              ╭──── done
    │                         ╭────╯
 35 ┤                    ╭────╯
    │               ╭────╯    ╭──── review
 30 ┤          ╭────╯    ╭────╯
    │     ╭────╯    ╭────╯    ╭──── in_progress
 25 ┤╭────╯    ╭────╯    ╭────╯
    │     ╭────╯    ╭────╯    ╭──── todo
 20 ┤╭────╯    ╭────╯    ╭────╯
    │     ╭────╯    ╭────╯    ╭──── backlog
 15 ┤╭────╯    ╭────╯    ╭────╯
    │     ╭────╯    ╭────╯
 10 ┤╭────╯    ╭────╯
    │     ╭────╯
  5 ┤╭────╯
    │
  0 ┼──────┬──────┬──────┬──────┬──────→ Время
   Нед.1  Нед.2  Нед.3  Нед.4  Нед.5
```

### Оси

- **X:** время (дни или недели, настраиваемый диапазон: 7 / 14 / 30 / 90 дней)
- **Y:** кумулятивное количество задач
- **Цвета слоёв:** по статусам (backlog → todo → in_progress → review → done)

### Цветовая схема статусов

| Статус | Цвет | HEX | Смысл |
|--------|------|-----|-------|
| backlog | Серый | `#6b7280` | Не запланировано |
| todo | Синий | `#3b82f6` | Запланировано |
| in_progress | Янтарный | `#f59e0b` | В работе |
| review | Фиолетовый | `#8b5cf6` | На проверке |
| done | Зелёный | `#22c55e` | Завершено |
| cancelled | Красный | `#ef4444` | Отменено |

### Что показывает

| Паттерн | Интерпретация | Действие |
|---------|---------------|----------|
| Параллельные линии | Стабильный поток | Всё хорошо |
| Расхождение линий (веер) | Накопление задач в каком-то статусе | Узкое место |
| Схождение линий | Задачи пропадают (отмены?) | Проверить cancelled |
| Полоска in_progress расширяется | WIP растёт | Остановить набор новых задач |
| Полоска done выпрямляется | Пропускная способность упала | Разобраться с блокерами |

### Алгоритм расчёта

1. Определить временной диапазон `[from, to]`
2. Для каждого дня `d` в диапазоне:
   - Пройти по changelog и восстановить состояние каждой задачи на конец дня `d`
   - Посчитать количество задач в каждом статусе
3. Построить stacked area chart

### Метрики CFD

| Метрика | Описание |
|---------|----------|
| `cfd.band_widths` | Ширина каждой полосы (количество задач в статусе) по дням |
| `cfd.bottleneck` | Статус с максимальным ростом за период |
| `cfd.stability_index` | 1 - (стандартное отклонение ширины in_progress / среднее). Близко к 1 = стабильный поток |

## 4. Время цикла (Cycle Time)

### Что это

Cycle Time — время от начала работы над задачей до её завершения. Ключевая метрика предсказуемости.

### Определения

| Тип времени | Начало | Конец | Что показывает |
|-------------|--------|-------|----------------|
| **Lead Time** | `created_at` (или переход из backlog) | `completed_at` | Время ожидания + работа |
| **Cycle Time** | Переход в `in_progress` | `completed_at` | Чистое время работы |
| **Review Time** | Переход в `review` | `completed_at` | Время на проверке |

### Визуализация: гистограмма

```
Кол-во задач
    │
 12 ┤  ████
    │  ████
 10 ┤  ████  ████
    │  ████  ████
  8 ┤  ████  ████  ████
    │  ████  ████  ████
  6 ┤  ████  ████  ████
    │  ████  ████  ████
  4 ┤  ████  ████  ████  ████
    │  ████  ████  ████  ████
  2 ┤  ████  ████  ████  ████  ████
    │  ████  ████  ████  ████  ████
  0 ┼──────┬──────┬──────┬──────┬──────→ Часы
       0-4    4-8   8-16  16-24  24-48
```

### Метрики

| Метрика | Формула | Описание |
|---------|---------|----------|
| `cycle_time.mean` | `sum(cycle_times) / count` | Среднее время цикла |
| `cycle_time.median` | `median(cycle_times)` | Медиана (устойчива к выбросам) |
| `cycle_time.p90` | `percentile(cycle_times, 90)` | 90% задач завершаются за это время |
| `cycle_time.p95` | `percentile(cycle_times, 95)` | 95% задач |
| `cycle_time.std_dev` | `std_dev(cycle_times)` | Разброс (предсказуемость) |
| `cycle_time.trend` | Линейная регрессия по неделям | Улучшается или ухудшается |

### Тренд по неделям

```
Cycle Time (часы)
    │
 30 ┤  ●
    │     ╲
 25 ┤      ●
    │        ╲
 20 ┤         ●────●
    │               ╲
 15 ┤                ●
    │
 10 ┼──────┬──────┬──────┬──────┬──────→ Недели
      Нед.1 Нед.2 Нед.3 Нед.4 Нед.5
```

Линия тренда показывает: поток ускоряется (↓) или замедляется (↑).

### Разбивка по приоритетам

| Приоритет | Ожидаемый cycle time | Цвет |
|-----------|---------------------|------|
| critical | < 4 часов | Красный |
| high | < 24 часов | Оранжевый |
| medium | < 72 часов | Синий |
| low | < 1 недели | Серый |

Если cycle_time для critical-задач превышает 24 часа — это сигнал проблем.

## 5. Пропускная способность (Throughput)

### Что это

Throughput — количество задач, завершённых за единицу времени. Мера продуктивности колонии.

### Визуализация: bar chart по дням/неделям

```
Задач/день
    │
  8 ┤        ████
    │        ████
  6 ┤  ████  ████  ████
    │  ████  ████  ████
  4 ┤  ████  ████  ████  ████
    │  ████  ████  ████  ████
  2 ┤  ████  ████  ████  ████  ████
    │  ████  ████  ████  ████  ████
  0 ┼──────┬──────┬──────┬──────┬──────→ Дни
      Пн    Вт    Ср    Чт    Пт
```

### Метрики

| Метрика | Описание |
|---------|----------|
| `throughput.daily_avg` | Среднее количество завершённых задач в день |
| `throughput.weekly_avg` | Среднее количество завершённых задач в неделю |
| `throughput.trend` | Направление тренда (растущий / стабильный / падающий) |
| `throughput.variability` | Коэффициент вариации (std_dev / mean). < 0.3 = стабильный |

### Сравнение по доскам (проектам)

```
Throughput (задач/неделю)

  ЗавЛаб   ████████████████████  18
  МУРАВЕЙ  ██████████████████    16
  КОТ      ████████████          10
```

### Прогноз

На основе throughput можно прогнозировать:

- **Когда готов бэклог:** `backlog_count / throughput.weekly_avg` = недель до завершения
- **Дата завершения N задач:** `today + (N / throughput.daily_avg)` дней

## 6. WIP-анализ

### Что это

WIP (Work In Progress) — количество задач, находящихся в работе (статусы `in_progress` + `review`). Контроль WIP — сердце Kanban.

### Текущий WIP

```
┌─────────────────────────────────────────────────────┐
│                    WIP: 12 / 15                     │
│  ████████████████████░░░░░░░░░░░░░░░░░░░░░░░░░░░  │
│  in_progress: 8/10  │  review: 4/5                  │
└─────────────────────────────────────────────────────┘
```

### Структура WIP

| Колонка | Текущий | Лимит | Статус |
|---------|---------|-------|--------|
| in_progress | 8 | 10 | 🟢 Норма |
| review | 4 | 5 | 🟢 Норма |
| **Итого** | **12** | **15** | **🟢 Норма** |

### Правила лимитов

| Состояние | Условие | Действие |
|-----------|---------|----------|
| 🟢 Норма | WIP < 80% лимита | Можно брать новые задачи |
| 🟡 Внимание | WIP 80-100% лимита | Предупреждение |
| 🔴 Превышение | WIP > лимит | Блокировка новых задач, алерт |

### Тренд WIP по времени

```
WIP
    │
 20 ┤              ╭──╮
    │         ╭──╮╭╯  ╰──╮
 15 ┤    ╭──╮╭╯  ╰╯      ╰── ← Лимит
    │╭──╮╭╯  ╰╯
 10 ┤╯  ╰╯
    │
  5 ┤
    │
  0 ┼──────┬──────┬──────┬──────┬──────→ Время
      Нед.1 Нед.2 Нед.3 Нед.4 Нед.5
```

### Метрики WIP

| Метрика | Описание |
|---------|----------|
| `wip.current` | Текущий WIP |
| `wip.limit` | Суммарный лимит |
| `wip.utilization` | `current / limit * 100%` |
| `wip.avg_duration` | Среднее время нахождения в WIP |
| `wip.violations_count` | Количество дней с превышением лимита за период |

### Закон Литтла

```
Cycle Time = WIP / Throughput
```

Если WIP = 12, Throughput = 6 задач/день → Cycle Time = 2 дня.

Это позволяет **прогнозировать** cycle time при изменении WIP или throughput.

## 7. Старение задач

### Что это

Задачи, которые слишком долго находятся в одном статусе — индикатор проблем (блокеры, потеря приоритета, забытые задачи).

### Heatmap старения

```
         backlog  todo  in_pr  review  done
 < 1 д    ██░░    ░░░░   ░░░░   ░░░░   ░░░░
 1-3 д    ████    ██░░   ░░░░   ░░░░   ░░░░
 3-7 д    ████    ████   ██░░   ░░░░   ░░░░
 7-14 д   ██░░    ██░░   ████   ██░░   ░░░░
 14+ д    █░░░    █░░░   ██░░   █░░░   ░░░░

 █ = задача в этой ячейке
 Цвет от зелёного (мало) до красного (много)
```

### Список «стареющих» задач

| Задача | Статус | Дней в статусе | Приоритет | Агент |
|--------|--------|----------------|-----------|-------|
| Настроить CI/CD | in_progress | 12 | high | — |
| Обновить доки | review | 8 | medium | — |
| Рефакторинг API | in_progress | 15 | low | — |

### Пороги старения

| Статус | Жёлтый порог | Красный порог |
|--------|-------------|---------------|
| backlog | > 30 дней | > 60 дней |
| todo | > 14 дней | > 30 дней |
| in_progress | > 5 дней | > 10 дней |
| review | > 3 дня | > 7 дней |

Пороги настраиваются в настройках проекта.

### Автоматические алерты

| Условие | Уровень | Действие |
|---------|---------|----------|
| Задача > красного порога | 🔴 Critical | Уведомление + подсветка на доске |
| Задача > жёлтого порога | 🟡 Warning | Подсветка на доске |
| 3+ задачи в in_progress > порога | 🔴 Critical | Рекомендация: остановить набор задач |
| WIP > лимит 3+ дня подряд | 🔴 Critical | Рекомендация: увеличить лимит или ускорить завершение |

### Эффективность автоматического распределения

Ключевой вопрос: насколько хорошо весы распределяют задачи между агентами внутри доски?

| Метрика | Формула | Описание |
|---------|---------|----------|
| `distribution.gini` | Коэффициент Джини загрузки агентов | 0 = идеальное распределение, 1 = всё на одном |
| `distribution.std_dev` | Стандартное отклонение количества задач на агента | Чем меньше, тем равномернее |
| `distribution.idle_rate` | Доля времени простоя агентов | Агент без задач / общее время |

### Целевые значения для весов

| Метрика | Целевое | Допустимое | Проблема |
|---------|---------|------------|----------|
| Gini загрузки | < 0.2 | 0.2 - 0.4 | > 0.4 |
| Idle rate | < 20% | 20-40% | > 40% |
| Max/Min ratio | < 2.0 | 2.0 - 3.0 | > 3.0 |

### Рекомендации по каналам

| Канал | Типичная проблема | Рекомендация |
|-------|-------------------|--------------|
| ЗАВЛАБ | IRL-задачи непредсказуемы | Увеличить WIP-лимит, добавить буфер |
| МУРАВЕЙ | Высокий throughput, но растёт CT | Проверить review-колонку на заторы |
| КОТ | Низкий throughput, высокий CT | Упростить задачи или увеличить параллелизм |

### KPI-карточки

| Карточка | Значение | Тренд | Цвет |
|----------|----------|-------|------|
| **WIP** | Текущий / Лимит | ↑↓ vs вчера | 🟢🟡🔴 |
| **Throughput** | Задач/день за неделю | % vs прошлая неделя | — |
| **Avg Cycle Time** | Среднее время цикла | % vs прошлая неделя | — |
| **Health Score** | Сводный индекс 0-100 | ↑↓ vs вчера | 🟢🟡🔴 |

### Health Score

Сводный индикатор здоровья потока (0-100):

| Фактор | Вес | Условие 100% |
|--------|-----|-------------|
| WIP в пределах лимита | 30% | WIP < 80% лимита |
| Throughput стабилен | 25% | Коэффициент вариации < 0.3 |
| Нет стареющих задач | 25% | 0 задач > красного порога |
| Cycle Time стабилен | 20% | Тренд не ухудшается > 10% |

### Мини-графики (Sparkline)

- **Throughput sparkline:** 7 столбиков за последнюю неделю
- **Cycle Time sparkline:** линия тренда за 4 недели
- **WIP sparkline:** линия WIP за 2 недели с отметкой лимита

## 12. Визуальный язык

### Цветовая система

#### Статусы (для всех графиков)

| Статус | Цвет | HEX | Использование |
|--------|------|-----|---------------|
| backlog | Серый | `#6b7280` | Фон, нейтральные элементы |
| todo | Синий | `#3b82f6` | Запланировано |
| in_progress | Янтарный | `#f59e0b` | В работе, внимание |
| review | Фиолетовый | `#8b5cf6` | На проверке |
| done | Зелёный | `#22c55e` | Завершено, успех |
| cancelled | Красный | `#ef4444` | Отменено, ошибка |

#### Здоровье (для KPI и алертов)

| Условие | Цвет | HEX | Иконка |
|---------|------|-----|--------|
| Норма (>80%) | Зелёный | `#22c55e` | 🟢 |
| Внимание (60-80%) | Жёлтый | `#eab308` | 🟡 |
| Проблема (<60%) | Красный | `#ef4444` | 🔴 |

#### Доски (для сравнения)

| Доска | Цвет | HEX |
|-------|------|-----|
| ЗАВЛАБ | Зелёный | `#22c55e` |
| МУРАВЕЙ | Синий | `#3b82f6` |
| КОТ | Оранжевый | `#f97316` |

### Типографика графиков

| Элемент | Размер | Начертание |
|---------|--------|-----------|
| Заголовок графика | 16px | Bold |
| Подпись оси | 11px | Regular |
| Значение на оси | 10px | Regular |
| Тултип | 12px | Regular |
| KPI значение | 24px | Bold |
| KPI подпись | 12px | Regular |

### Принципы визуализации

1. **Без украшений.** Никаких 3D-эффектов, градиентов, теней. Чистые flat-графики.
2. **Контраст на первом месте.** Цвета различимы при дальтонизме (использовать паттерны как дублирование).
3. **Интерактуальность.** Наведение на любой элемент показывает тултип с деталями. Клик — переход к списку задач.
4. **Адаптивность.** На мобильных sparkline заменяют полные графики. KPI-карточки перестраиваются в колонку.
5. **Пустые состояния.** Нет данных — показать заглушку «Недостаточно данных для анализа» с поясним минимального количества задач.

## Приложение A: Глоссарий

| Термин | Определение |
|--------|-------------|
| **CFD** | Cumulative Flow Diagram — кумулятивная диаграмма потока |
| **Cycle Time** | Время от начала работы до завершения задачи |
| **Lead Time** | Время от создания задачи до её завершения |
| **Throughput** | Количество завершённых задач за единицу времени |
| **WIP** | Work In Progress — задачи в работе |
| **Health Score** | Сводный индикатор здоровья потока (0-100) |
| **Gini** | Коэффициент Джини — мера неравномерности распределения |
| **Sparkline** | Мини-график без осей, показывающий тренд |
| **Stale task** | Задача, слишком долго находящаяся в одном статусе |
| **Little's Law** | CT = WIP / Throughput — фундаментальное соотношение потока |

## Приложение B: Минимальные требования к данным

Для работы аналитики необходимо:

1. **Changelog с историей переходов.** Каждое изменение статуса должно записываться с timestamp и `{ from, to }`.
2. **Минимум 10 завершённых задач** для статистически значимых метрик cycle time.
3. **Минимум 7 дней данных** для построения трендов.
4. **Корректные timestamp** — все даты в UTC или с указанием часового пояса.

Без этих данных аналитика показывает заглушку «Недостаточно данных».
