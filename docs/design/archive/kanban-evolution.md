## 1. Обзор системы эволюции

### 1.1. Проблема

Канбан-система Myrmex Control — живой организм. Три доски (ЗАВЛАБ, МУРАВЕЙ, КОТ) обслуживают разные контексты, и параметры, оптимальные для одной доски, могут быть неоптимальны для другой. Веса задач, WIP-лимиты, пороги старения, правила балансировки — всё это требует постоянной калибровки.

Без системы эволюции параметры настраиваются вручную и устаревают. С системой эволюции канбан-система **учится на собственных метриках** и адаптируется.

### 1.2. Принципы

- **Данные первичны.** Решения об изменениях принимаются на основе метрик, а не интуиции.
- **Малые шаги.** Изменения применяются инкрементально, не более одного параметра за цикл.
- **Обратимость.** Каждое изменение можно откатить. Бэкап конфигурации перед каждым изменением.
- **Человек в петле.** Критические изменения требуют подтверждения Евгения. Некритические — применяются автоматически с уведомлением.
- **Изолированное тестирование.** A/B тесты запускаются на одной доске, не затрагивая остальные.

### 1.3. Архитектура эволюции

```
======================= СИСТЕМА ЭВОЛЮЦИИ =======================

  +--------------+    +--------------+    +--------------+
  |  СБОР ДАННЫХ |--->|  АНАЛИЗ      |--->|  РЕШЕНИЕ     |
  |  (Metrics    |    │  (Insight    │    │  (Decision   │
  │   Collector) │    │   Engine)    │    │   Engine)    │
  +--------------+    +--------------+    +-------+-------+
                                               |
  +--------------+    +--------------+    +---v-----------+
  |  ОБРАТНАЯ    |<---|  ПРИМЕНЕНИЕ  |<---|  ТЕСТИРОВАНИЕ │
  |  СВЯЗЬ       │    │  (Apply      │    │  (A/B Test   │
  │  (Feedback   │    │   Engine)    │    │   Runner)    │
  │   Loop)      │    │              │    │              │
  +--------------+    +--------------+    +--------------+

  +----------------------------------------------------------+
  |                    ХРАНИЛИЩЕ ЗНАНИЙ                      │
  │  evolution_backlog.json │ ab_tests.json │ history.log    │
  +----------------------------------------------------------+
```

### 1.4. Связь с существующей системой

Система эволюции НЕ заменяет `self_evolve.sh` и `evolution_backlog.json` — она **надстраивается** над ними:

- `self_evolve.sh` остаётся инструментом применения инсайтов к слоям (rules, memory, skills, agents).
- Система эволюции добавляет **метрический контур**: сбор данных, анализ, A/B тестирование, обратную связь.
- `evolution_backlog.json` расширяется задачами типа `weight_tuning`, `wip_adjustment`, `threshold_calibration`.

## 2. Цикл эволюции

### 2.1. Пятифазный цикл (PDCA-подобный)

```
    +---------------------------------------------------+
    |                                                   |
    v                                                   |
+--------+    +--------+    +--------+    +--------+    |
│ 1.СБОР |--->│ 2.АНАЛ |--->│ 3.ТЕСТ |--->│ 4.ПРИМ |--->+
│ ДАННЫХ │    │ ИЗАЦИЯ │    │ ИРОВАНИЕ    │ ЕНЕНИЕ │
+--------+    +--------+    +--------+    +--------+
                                               |
                                               v
                                         +--------+
                                         │ 5.ОБРАТ│
                                         │ НАЯ   │
                                         │ СВЯЗЬ │
                                         +--------+
```

**Фаза 1 — Сбор данных (Collect)**
- Интервал: каждые 6 часов (4 раза в сутки)
- Собираются: метрики потока, метрики весов, метрики агентов, метрики уведомлений
- Результат: снимок состояния системы (`snapshot_YYYYMMDD_HHMM.json`)

**Фаза 2 — Анализ (Analyze)**
- Интервал: после каждого сбора данных
- Сравнение с предыдущим снимком и базовой линией
- Выявление аномалий, трендов, отклонений
- Результат: список инсайтов с уровнем уверенности

**Фаза 3 — Тестирование (Test)**
- Интервал: при наличии инсайтов с уверенностью > 60%
- A/B тест на одной доске (или на синтетических данных)
- Длительность: 24-72 часа
- Результат: подтверждён или опровергнут инсайт

**Фаза 4 — Применение (Apply)**
- Интервал: при успешном A/B тесте
- Применение изменения ко всем доскам (или только к целевой)
- Бэкап предыдущей конфигурации
- Результат: обновлённая конфигурация

**Фаза 5 — Обратная связь (Feedback)**
- Интервал: 48 часов после применения
- Мониторинг: улучшились ли метрики?
- Если ухудшение — автоматический откат
- Результат: запись в историю эволюции

### 2.2. Триггеры эволюции

| Триггер | Частота | Описание |
|---------|---------|----------|
| **Таймер** | Каждые 6 часов | Регулярный сбор данных и анализ |
| **Аномалия** | Мгновенно | Отклонение метрики > 2σ от нормы |
| **Ручной** | По запросу | Евгений запускает эволюцию вручную |
| **Деградация** | Мгновенно | Ухудшение метрики > 20% за 24 часа |
| **Стагнация** | Еженедельно | Нет улучшений за 7 дней — принудительный пересмотр |

### 2.3. Жизненный цикл инсайта

Каждый инсайт проходит через состояния:

```
pending -> analyzing -> testing -> validated -> applied -> confirmed
                                        |
                                        +---> rejected -> archived
```

| Состояние | Описание |
|-----------|----------|
| `pending` | Инсайт обнаружен, ожидает анализа |
| `analyzing` | Проводится углублённый анализ |
| `testing` | A/B тест запущен |
| `validated` | Тест пройден, ожидает применения |
| `applied` | Изменение применено |
| `confirmed` | Изменение подтверждено обратной связью |
| `rejected` | Тест провалился или обратная связь отрицательная |
| `archived` | Отклонённый инсайт сохранён для истории |

## 3. Метрики для улучшений

### 3.1. Категории метрик

**A. Метрики потока (Flow Metrics)**

| Метрика | Что измеряет | Целевое значение |
|---------|-------------|------------------|
| Cycle Time | Скорость выполнения (started -> done) | Снижение тренда |
| Lead Time | Полное время жизни (created -> done) | Снижение тренда |
| Throughput | Производительность (done / period) | Рост или стабильность |
| WIP | Загруженность (in_progress + review) | В пределах лимитов |
| Blocked Ratio | Доля заблокированных (blocked / active) | < 15% |
| Flow Efficiency | Эффективность (work_time / total_time) | > 40% |

**B. Метрики весов (Weight Metrics)**

| Метрика | Что измеряет | Целевое значение |
|---------|-------------|------------------|
| Weight Accuracy | Корреляция веса с реальной сложностью | > 0.7 |
| Priority Inversion Rate | Инверсии приоритетов (high ждёт, low делается) | < 5% |
| Channel Balance | Разница нагрузки МУРАВЕЙ vs КОТ | < threshold |
| Gini Coefficient | Равномерность загрузки агентов | < 0.3 |
| Weight Drift | Стабильность весовых компонентов | Без резких скачков |

**C. Метрики качества (Quality Metrics)**

| Метрика | Что измеряет | Целевое значение |
|---------|-------------|------------------|
| DoR Pass Rate | Качество подготовки задач | > 90% |
| DoD Pass Rate | Качество завершения задач | > 95% |
| Rework Rate | Доработки (reopened / completed) | < 5% |
| Stale Task Rate | Доля застоявшихся задач | < 10% |
| SLA Compliance | Соблюдение SLA | > 90% |

**D. Метрики уведомлений (Notification Metrics)**

| Метрика | Что измеряет | Целевое значение |
|---------|-------------|------------------|
| Notification Noise | Доля проигнорированных (ignored / sent) | < 30% |
| Alert Response Time | Скорость реакции на алерты | < SLA |
| Digest Open Rate | Открытие дайджестов | > 50% |
| False Positive Rate | Ложные срабатывания | < 10% |

### 3.2. Базовые линии (Baselines)

Для каждой метрики устанавливается базовая линия:

1. Сбор данных за первые 7 дней работы системы
2. Расчёт медианы и интерквартильного размаха (IQR)
3. Базовая линия = медиана
4. Допустимый диапазон = [Q1 - 1.5*IQR, Q3 + 1.5*IQR]
5. Обновление базовой линии — раз в 30 дней (скользящее окно)

### 3.3. Пороги для инсайтов

| Уровень | Отклонение от базовой линии | Действие |
|---------|----------------------------|----------|
| Норма | < 10% | Нет действий |
| Внимание | 10-20% | Логирование, мониторинг |
| Предупреждение | 20-40% | Генерация инсайта, анализ |
| Критический | > 40% | Немедленный инсайт, приоритетный анализ |

### 3.4. Корреляционный анализ

Система отслеживает корреляции между параметрами и метриками:

| Параметр (изменение) | Ожидаемая корреляция | Метрика |
|---------------------|---------------------|---------|
| WIP-лимит (снижение) | Положительная | Cycle Time (снижение) |
| W_age (рост) | Положительная | Stale Task Rate (снижение) |
| W_deps (рост) | Положительная | Blocked Ratio (снижение) |
| Balancing threshold (снижение) | Положительная | Channel Balance (улучшение) |
| Notification batching (рост) | Отрицательная | Notification Noise (рост) |

Если корреляция нарушается (например, снижение WIP-лимита не уменьшает Cycle Time) — сигнал для пересмотра модели.

## 4. A/B тестирование весов

### 4.1. Концепция

A/B тестирование сравнивает две конфигурации весов на реальных данных, не нарушая работу системы. Одна доска (или один канал) работает с текущими весами (контроль), другая — с изменёнными (эксперимент).

### 4.2. Что тестируем

| Параметр | Текущее значение | Диапазон тестирования | Где тестируем |
|----------|-----------------|----------------------|---------------|
| W_priority (P0-P4) | 40, 30, 20, 10, 0 | +/- 10 для каждого | По доскам |
| W_age (per_days / max) | 2 / 20 | 1-4 / 10-30 | По доскам |
| W_deps (per_blocked / max) | 5 / 15 | 3-10 / 10-25 | По доскам |
| W_channel (adjustment) | +/- 10 | +/- 5..15 | МУРАВЕЙ vs КОТ |
| W_recurring (per_day / max) | 3 / 15 | 1-5 / 10-25 | По доскам |
| Balancing threshold | 20 | 10-40 | МУРАВЕЙ vs КОТ |
| WIP-лимит (per column) | 3-10 | +/- 2 | По доскам |
| Stale threshold (per status) | 3-60 дней | +/- 30% | По доскам |

### 4.3. Методология A/B теста

**Шаг 1: Гипотеза**
Формулируется конкретная гипотеза:
> «Увеличение W_age max с 20 до 30 снизит Stale Task Rate на 15% без увеличения Cycle Time более чем на 5%.»

**Шаг 2: Разделение**
- Вариант A (контроль): текущая конфигурация
- Вариант B (эксперимент): изменённый параметр
- Разделение: по доскам (ЗАВЛАБ = контроль, МУРАВЕЙ = эксперимент) или по времени (чётные дни = A, нечётные = B)

**Шаг 3: Длительность**
- Минимум: 24 часа (высокочастотные метрики: throughput, WIP)
- Оптимально: 72 часа (среднечастотные: cycle time, blocked ratio)
- Максимум: 7 дней (низкочастотные: stale rate, SLA compliance)

**Шаг 4: Критерии успеха**
Определяются до начала теста:
- Первичная метрика: должна улучшиться на X%
- Вторичные метрики: не должны ухудшиться более чем на Y%
- Статистическая значимость: p < 0.05 (или минимальный размер выборки)

**Шаг 5: Решение**
- Гипотеза подтверждена → применение
- Гипотеза опровергнута → архивирование
- Результат неоднозначен → продление теста или отмена

### 4.4. Ограничения

- **Один тест за раз.** Нельзя тестировать два параметра одновременно — невозможно атрибутировать результат.
- **Минимальный размер выборки.** Не менее 20 задач в каждом варианте.
- **Исключение для ЗАВЛАБ.** IRL-задачи слишком немногочисленны и непредсказуемы для A/B тестирования. Тесты проводятся на МУРАВЕЕ и КОТе.
- **Запрет на критические изменения.** Изменения, которые могут заблокировать всю систему, не тестируются через A/B — только через ручной ревью.

### 4.5. Примеры A/B тестов

**Тест #1: Увеличение штрафа за старение**
- Гипотеза: Увеличение W_age max с 20 до 30 снизит количество задач, застревающих в BACKLOG > 14 дней
- Контроль: W_age max = 20
- Эксперимент: W_age max = 30
- Первичная метрика: Stale Task Rate (BACKLOG > 14 дней)
- Вторичная метрика: Throughput (не должен упасть)
- Длительность: 72 часа
- Критерий успеха: Stale Task Rate снижение на 20%, Throughput не снижение более чем на 5%

**Тест #2: Снижение порога балансировки**
- Гипотеза: Снижение threshold с 20 до 15 улучшит баланс загрузки между МУРАВЕЕМ и КОТом
- Контроль: threshold = 20
- Эксперимент: threshold = 15
- Первичная метрика: Channel Balance (Gini)
- Вторичная метрика: количество перемещений задач (не должно расти > 50%)
- Длительность: 48 часов
- Критерий успеха: Gini снижение на 15%, перемещения не рост более чем на 30%

**Тест #3: Корректировка W_priority для P2**
- Гипотеза: Увеличение W_priority для P2 с 20 до 25 снизит инверсии приоритетов
- Контроль: P2 = 20
- Эксперимент: P2 = 25
- Первичная метрика: Priority Inversion Rate
- Вторичная метрика: Cycle Time для P2 (не должен вырасти)
- Длительность: 72 часа
- Критерий успеха: Inversion Rate снижение на 30%, Cycle Time P2 не рост более чем на 10%

### 4.6. Мультивариантное тестирование

На данном этапе НЕ применяется. Причина: недостаточный объём данных для статистической значимости при разделении на 3+ варианта. Будет рассмотрено при росте количества задач > 500/неделю.

## 5. Автоматический сбор обратной связи

### 5.1. Источники обратной связи

**A. Явная обратная связь (Explicit Feedback)**

| Метод | Когда | Что собираем |
|-------|-------|-------------|
| Оценка рекомендации | После автоматического действия | «Полезно / Бесполезно / Вредно» |
| Оценка уведомления | После получения уведомления | «Релевантно / Шум / Пропустить тип» |
| Оценка приоритета | При ручном переопределении приоритета | «Система ошиблась на N уровней» |
| Еженедельный опрос | В конце каждой недели | NPS-подобная оценка (0-10) + комментарий |

**B. Неявная обратная связь (Implicit Feedback)**

| Сигнал | Интерпретация | Вес |
|--------|--------------|-----|
| Задача переопределена вручную (приоритет) | Система ошиблась | Высокий |
| Задача перемещена назад (DONE -> REVIEW) | DoD не выполнен | Высокий |
| Уведомление проигнорировано (> 1 часа) | Шум | Средний |
| Задача в BACKLOG > 30 дней без движения | Не нужна или недооценена | Средний |
| Задача выполнена быстрее оценки | Переоценена сложность | Низкий |
| Задача выполнена медленнее оценки | Недооценена сложность | Низкий |
| Ручная ребалансировка каналов | Автоматическая балансировка не работает | Высокий |

**C. Системная обратная связь (System Feedback)**

| Сигнал | Интерпретация | Вес |
|--------|--------------|-----|
| Cycle Time растёт | Поток замедляется | Критический |
| Throughput падает | Производительность снижается | Критический |
| Blocked Ratio растёт | Проблемы с зависимостями | Высокий |
| WIP превышен 3+ дня | Лимиты не работают | Высокий |
| SLA Compliance < 80% | Приоритеты не работают | Критический |
| Notification Noise > 40% | Слишком много уведомлений | Средний |

**D. Контекстная обратная связь (Context Feedback)**

| Сигнал | Интерпретация | Действие |
|--------|--------------|----------|
| Новый агент добавлен | Изменилась ёмкость системы | Пересчитать WIP-лимиты |
| Агент удален/отключен | Снизилась ёмкость | Перераспределить задачи |
| Новый проект создан | Новый контекст | Создать профиль весов |
| Проект завершён | Контекст удалён | Архивировать профиль |

### 5.2. Механизм сбора

**Пассивный сбор (непрерывный):**
- Каждое действие пользователя логируется в `feedback_log.json`
- Каждые 6 часов лог анализируется на паттерны
- Паттерны агрегируются в метрики обратной связи

**Активный сбор (по событиям):**
- После автоматического действия — мини-опрос (1 клик)
- При завершении спринта — расширенный опрос
- При обнаружении аномалии — запрос контекста у Евгения

**Периодический сбор (по расписанию):**
- Ежедневный дайджест с кнопкой оценки
- Еженедельный NPS-опрос
- Ежемесячный стратегический опрос

### 5.3. Модель данных обратной связи

Каждая единица обратной связи содержит:

- **id** — уникальный идентификатор
- **source** — explicit | implicit | system | context
- **category** — weights | wip | notifications | priorities | balance | other
- **signal** — что именно зафиксировано (текстовое описание)
- **value** — нормализованная оценка [-1, +1], где -1 = негативно, +1 = позитивно
- **confidence** — уверенность в интерпретации [0, 1]
- **context** — контекст события (board, task_id, agent_id, parameter, old_value, new_value)
- **timestamp** — дата и время
- **processed** — была ли учтена в эволюции

### 5.4. Агрегация обратной связи

| Окно | Агрегация | Использование |
|------|-----------|---------------|
| 24 часа | Суммарный score по категориям | Обнаружение острых проблем |
| 7 дней | Средневзвешенный score (implicit вес выше) | Генерация инсайтов |
| 30 дней | Тренд по категориям | Стратегические решения |

**Взвешивание по источникам:**
- Explicit: вес 1.0 (прямая оценка пользователя)
- Implicit: вес 0.7 (выведенная оценка)
- System: вес 0.9 (объективные метрики)
- Context: вес 0.5 (фоновые изменения)

### 5.5. Замкнутый контур

Обратная связь замыкает цикл эволюции:

```
Сбор данных -> Анализ -> A/B тест -> Применение -> Обратная связь -> Сбор данных
                                            ^                              |
                                            |______________________________|
```

Если после применения изменения обратная связь отрицательная (score < -0.3) в течение 48 часов — автоматический откат к предыдущей конфигурации.

## 6. Движок эволюции (Evolution Engine)

### 6.1. Компоненты

Движок эволюции состоит из пяти компонентов:

**A. Metrics Collector (Сборщик метрик)**
- Собирает метрики из myrmex.json и changelog
- Рассчитывает производные метрики (CFD, cycle time, throughput)
- Хранит снимки для исторического сравнения
- Интервал: каждые 6 часов

**B. Insight Engine (Генератор инсайтов)**
- Анализирует метрики на аномалии и тренды
- Сравнивает с базовыми линиями
- Генерирует инсайты с уровнем уверенности
- Приоритизирует инсайты по потенциальному влиянию

**C. AB Test Runner (Запускатель тестов)**
- Создаёт A/B тесты для подтверждения инсайтов
- Мониторит ход теста
- Определяет статистическую значимость результатов
- Принимает решение: validated / rejected

**D. Apply Engine (Применитель изменений)**
- Применяет подтверждённые изменения к конфигурации
- Создаёт бэкап предыдущей конфигурации
- Логирует изменение в history.log
- Уведомляет Евгения о применении

**E. Feedback Loop (Контур обратной связи)**
- Собирает обратную связь после применения изменений
- Оценивает эффективность изменений
- Инициирует откат при отрицательной обратной связи
- Обновляет базовые линии

### 6.2. Хранилище знаний

Движок использует три файла: `evolution_backlog.json`, `ab_tests.json`, `evolution_history.log`.

**evolution_backlog.json** — очередь задач эволюции с типами: `weight_tuning`, `wip_adjustment`, `threshold_calibration`, `notification_tuning`. Каждая задача содержит: id, parameter, current_value, proposed_value, hypothesis, status, confidence, result.

**ab_tests.json** — активные и завершённые A/B тесты. Каждый тест содержит: id, insight_id, parameter, control_value, experiment_value, board_control, board_experiment, status, sample sizes, metrics (primary + secondary), verdict.

**evolution_history.log** — журнал всех изменений в формате:
```
[timestamp] ACTION: parameter old -> new (reason, result)
```

## 7. Уровни автоматизации

### 7.1. Пять уровней

| Уровень | Название | Описание |
|---------|----------|----------|
| 0 | Ручной | Система только собирает метрики и показывает дашборд |
| 1 | Рекомендательный | Генерирует инсайты, Евгений подтверждает/отклоняет |
| 2 | Полуавтоматический | Некритические изменения применяются автоматически |
| 3 | Автоматический | Все изменения применяются автоматически, уведомления постфактум |
| 4 | Автономный | Система сама определяет, что тестировать, еженедельный отчёт |

### 7.2. Рекомендуемый уровень

**Начальный: 1 (Рекомендательный).** Причина: система новая, данных мало, модель весов не откалибрована.

**Переход на 2:** после 30 дней работы и минимум 5 успешных A/B тестов.

**Переход на 3:** после 90 дней работы и стабильных улучшений метрик.

### 7.3. Классификация изменений по критичности

| Критичность | Примеры | Уровень для авто-применения |
|-------------|---------|---------------------------|
| Низкая | Корректировка W_age +/-5, порог старения +/-2 дня | Уровень 2 |
| Средняя | Корректировка W_priority +/-10, WIP-лимит +/-2 | Уровень 3 |
| Высокая | Изменение формулы веса, новый компонент | Уровень 4 |
| Критическая | Изменение архитектуры распределения | Только ручное |

## 8. Безопасность эволюции

### 8.1. Принцип «Не навреди»

1. **Бэкап перед изменением.** Полная копия конфигурации сохраняется.
2. **Автоматический откат.** Если метрики ухудшились > 15% в течение 48 часов — откат.
3. **Ограничение скорости.** Не более 1 изменения за 24 часа.
4. **Ограничение масштаба.** Изменение параметра не более чем на 20% за раз.
5. **Человек может остановить.** Евгений может в любой момент приостановить эволюцию.

### 8.2. Защитные механизмы

**Watchdog (Сторожевой таймер):**
- Каждые 15 минут проверяет состояние системы
- При критических отклонениях — принудительный откат + уведомление

**Rate Limiter (Ограничитель скорости):**
- Максимум 1 изменение в 24 часа
- Максимум 3 активных A/B теста одновременно
- Максимум 10 откатов в месяц (иначе — остановка и ревью)

**Circuit Breaker (Автоматический выключатель):**
- 3 последовательных отклонения → остановка на 72 часа
- 5 откатов за 7 дней → остановка до ручного ревью
- Любая метрика ухудшилась > 30% → немедленная остановка

### 8.3. Аудит-трейл

Каждое действие логируется: `timestamp | action | parameter | old_value | new_value | reason | result`

### 8.4. Ручное переопределение

Евгений может: заблокировать параметр, установить значение вручную, приостановить эволюцию, откатить к любой предыдущей конфигурации.

## 9. Дашборд эволюции

### 9.1. Структура

Отдельная вкладка `/evolution` с шестью виджетами:

**A. Обзор состояния** — статус, уровень автоматизации, последний сбор, активные тесты, применённые изменения, Health Score.

**B. График метрик (Metrics Timeline)** — линейные графики Cycle Time, Throughput, Stale Task Rate, SLA Compliance с отметками моментов изменений и A/B тестов.

**C. Активные тесты** — таблица текущих A/B тестов с прогрессом.

**D. История изменений** — таблица всех применённых/отклонённых/откаченных изменений.

**E. Очередь инсайтов** — список инсайтов с уверенностью и приоритетом.

**F. Карта корреляций** — тепловая карта корреляций между параметрами и метриками.

### 9.2. Интеграция с основным дашбордом

На основном дашборде — мини-виджет эволюции с кнопками «Подробнее» и «Приостановить».
