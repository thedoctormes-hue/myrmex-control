## 1. Проблема: потеря контекста

### 1.1. Симптомы

Текущая система хранит задачи как плоские записи в `myrmex.json`. Этого достаточно для канбан-доски, но недостаточно для работы AI-агентов и человека в потоке. Типичные проблемы:

**«Задача-призрак».** Агент видит задачу «Рефакторинг API» на доске. Но не знает:
- Почему эта задача важна (какой эпик, какая цель)?
- Какие решения уже приняты (обсуждения, trade-offs)?
- Какие сессии работали над ней (кто, когда, сколько)?
- Какие артефакты созданы (файлы, коммиты, документы)?
- Какие задачи блокируются этой задачей?

**«Сессия в вакууме».** Qwen Code начинает новую сессию. Сессия не знает:
- Какие задачи сейчас в работе?
- Какие решения приняты в предыдущих сессиях?
- Какой контекст у задачи, над которой работает агент?

**«Решение потеряно».** Команда обсудила архитектурное решение, но оно не записано. Через месяц — те же дебаты.

**«История недоступна».** Changelog фиксирует факты (создал, переместил, удалил), но не фиксирует контекст (почему, обсуждение, альтернативы).

### 1.2. Корневая причина

Текущая модель данных разделяет **состояние** (tasks, projects) и **историю** (changelog), но не хранит **контекст** — знания, связи и решения, которые делают задачу осмысленной.

### 1.3. Целевое состояние

```
Сейчас:
  Task + Changelog → «Что произошло»

Цель:
  Task + Changelog + TaskContext + SessionLinks + DecisionLog + ContextGraph
  → «Что произошло, почему, кто участвовал, какие решения приняты,
     как это связано с другими задачами, и что из этого следует»
```

## 2. Обзор системы контекста

### 2.1. Пять слоёв контекста

```
┌─────────────────────────────────────────────────────────────┐
│                    Слой 5: ContextGraph                     │
│   Граф связей между задачами, решениями, сессиями, файлами  │
├─────────────────────────────────────────────────────────────┤
│                    Слой 4: DecisionLog                      │
│   Память решений: что решили, почему, какие альтернативы    │
├─────────────────────────────────────────────────────────────┤
│                  Слой 3: SessionTaskLink                    │
│   Связь задач и сессий: кто работал, когда, сколько        │
├─────────────────────────────────────────────────────────────┤
│                    Слой 2: TaskContext                      │
│   Контекст задачи: ЗАЧЕМ, ДЛЯ КОГО, критерии, ссылки        │
├─────────────────────────────────────────────────────────────┤
│                    Слой 1: Task + Changelog                 │
│   Текущее состояние + история изменений (уже есть)          │
└─────────────────────────────────────────────────────────────┘
```

### 2.2. Принципы

1. **Контекст — первоклассный объект.** Контекст задачи не комментарий в свободной форме, а структурированная сущность с собственным жизненным циклом.

2. **Связность важнее полноты.** Лучше меньше полей, но с чёткими связями, чем больше полей без связей.

3. **Контекст накапливается.** Каждая сессия, каждое решение, каждое обсуждение добавляют слой контекста. Контекст не перезаписывается — дополняется.

4. **Поиск — основной сценарий доступа.** Контекст бесполезен, если его нельзя найти. Полнотекстовый поиск, фильтрация, граф связей — обязательны.

5. **JSON-first.** Всё хранится в `myrmex.json` (или связанных JSON-файлах). Никакой внешней БД. Атомарная запись через tmp + rename.

### 2.3. Место в архитектуре

```
myrmex.json (текущее)
├── _meta
├── workspace
├── projects[]
├── agents[]
├── tasks[]              ← Слой 1: состояние (есть)
├── library[]
├── files[]
├── servers[]
├── settings
├── mcp_servers[]
├── changelog[]           ← Слой 1: история (есть)
├── users[]
└── refresh_tokens[]

myrmex.json (расширение — этот документ)
├── ... (всё существующее)
├── task_contexts[]       ← Слой 2: контекст задач (новое)
├── session_links[]       ← Слой 3: связи сессий (новое)
├── decisions[]           ← Слой 4: память решений (новое)
└── context_graph         ← Слой 5: граф контекста (новое)
```

## 3. Контекст задачи (TaskContext)

### 3.1. Назначение

TaskContext — структурированное описание «смысла» задачи. Не что делать (это `Task.title`), а зачем, для кого, как понять что готово, что нужно знать.

### 3.2. Структура

```
TaskContext {
  id: string
  task_id: string               // ссылка на Task

  // Формула контекста (из kanban-architecture.md §5):
  why: string                   // ЗАЧЕМ: обоснование, цель, проблема
  who: string                   // ДЛЯ КОГО: аудитория, получатель
  acceptance_criteria: string[] // КРИТЕРИИ ГОТОВНОСТИ: чеклист DoD
  context_links: ContextLink[]  // КОНТЕКСТ: ссылки на ресурсы
  notes: string                 // ЗАМЕТКИ: свободная форма (ограниченная)

  // Метаданные
  created_at: string
  updated_at: string
  created_by: string            // кто создал контекст (agent id или "human")
  version: number               // версия контекста (инкремент при каждом изменении)
}
```

### 3.3. ContextLink — ссылка на ресурс

```
ContextLink {
  type: 'doc' | 'file' | 'commit' | 'pr' | 'discussion' | 'external' | 'session'
  label: string                 // человекочитаемое название
  url: string                   // путь или URL
  description?: string          // краткое описание (1-2 предложения)
}
```

Примеры:
- `{ type: 'doc', label: 'API спецификация', url: '/docs/api/tasks.md' }`
- `{ type: 'file', label: 'Миграция БД', url: 'server/migrations/003_deps.sql' }`
- `{ type: 'commit', label: 'feat: add dependencies field', url: 'abc1234' }`
- `{ type: 'discussion', label: 'Обсуждение архитектуры', url: '#42' }`
- `{ type: 'session', label: 'Сессия OWL 2026-05-11', url: 'session:owl:20260511-001' }`

### 3.4. Правила

1. **Один TaskContext на задачу.** TaskContext не заменяет Task — дополняет. У задачи может быть 0 или 1 TaskContext.

2. **TaskContext создаётся лениво.** Не нужно заполнять контекст при создании задачи. Контекст добавляется, когда задача переходит из backlog в todo (DoR — Definition of Ready).

3. **Версионирование.** При каждом изменении TaskContext — version инкрементируется. Старые версии не хранятся в основном файле (для экономии места), но факт изменения логируется в changelog.

4. **Максимальные длины:** why — 2000 символов, who — 500 символов, notes — 5000 символов, acceptance_criteria — до 20 пунктов по 500 символов каждый.

5. **Обязательные поля для перехода в todo:** why и хотя бы один acceptance_criterion. Без этого — задача остаётся в backlog.

### 3.5. Пример

```
TaskContext {
  id: "tc-001",
  task_id: "task-123",
  why: "Задачи должны ссылаться друг на друга для отображения блокировок. Без этого агенты не видят зависимости и начинают работу над задачами, которые ещё не готовы.",
  who: "Агенты, работающие над связанными задачами. Евгений — для визуализации на доске.",
  acceptance_criteria: [
    "Миграция БД: поле dependencies[] добавлено в схему",
    "API endpoint PATCH /api/tasks/:id принимает dependencies",
    "Валидация циклических зависимостей на сервере",
    "Тесты: 10 юнит-тестов + 5 интеграционных",
    "UI: отображение связей на карточке задачи"
  ],
  context_links: [
    { type: 'doc', label: 'Схема данных', url: '/docs/design/kanban-architecture.md#связи-и-блокировки' },
    { type: 'doc', label: 'API спецификация', url: '/docs/api/openapi.yaml' },
    { type: 'external', label: 'GitHub Projects API (пример)', url: 'https://docs.github.com/en/rest/projects' }
  ],
  notes: "Рассмотрели два варианта: (1) хранить зависимости в отдельной таблице, (2) хранить как массив в tasks[]. Выбрали вариант 2 — проще, достаточно для single-user сценария.",
  created_at: "2026-05-11T10:00:00Z",
  updated_at: "2026-05-11T12:00:00Z",
  created_by: "agent-owl",
  version: 3
}
```

## 4. Связь задач и сессий (SessionTaskLink)

### 4.1. Назначение

SessionTaskLink фиксирует факт: «сессия X работала над задачей Y в период Z». Это мост между канбан-доской и Qwen Code сессиями.

### 4.2. Структура

```
SessionTaskLink {
  id: string
  session_id: string            // идентификатор сессии Qwen Code
  task_id: string               // ссылка на Task

  // Временные метки
  started_at: string            // когда сессия начала работу над задачей
  ended_at: string | null       // когда закончила (null = ещё работает)

  // Кто работал
  agent_id: string              // ID агента (owl, cat, ant и т.д.)
  agent_name: string            // человекочитаемое имя

  // Что делали
  actions: SessionAction[]      // список действий в сессии
  summary: string               // краткое резюме работы (1-3 предложения)

  // Результат
  status: 'in_progress' | 'completed' | 'blocked' | 'abandoned'
  outcome: string | null        // результат: что получилось / не получилось

  // Артефакты
  artifacts: SessionArtifact[]  // созданные/изменённые файлы, коммиты

  // Связи
  related_sessions: string[]    // другие сессии, связанные с этой задачей
}
```

### 4.3. SessionAction — действие в сессии

```
SessionAction {
  timestamp: string
  type: 'code_write' | 'code_edit' | 'code_review' | 'test_run' | 'doc_write'
       | 'decision_made' | 'discussion' | 'research' | 'debug' | 'deploy'
  description: string           // что сделано
  files_changed: string[]       // затронутые файлы
}
```

### 4.4. SessionArtifact — артефакт сессии

```
SessionArtifact {
  type: 'file' | 'commit' | 'pr' | 'doc' | 'test' | 'config'
  path: string                  // путь к файлу или идентификатор
  description: string           // что это и зачем
  created: boolean              // создан или изменён
}
```

### 4.5. Правила

1. **Одна сессия может работать над несколькими задачами.** Одна задача может быть в нескольких сессиях (последовательно или параллельно — разными агентами).

2. **SessionTaskLink создаётся автоматически.** Когда агент начинает работу над задачей — создаётся запись. Когда заканчивает — обновляется ended_at и status.

3. **Session ID формат:** `{agent}:{date}:{sequence}`, например: `owl:20260511:001`.

4. **Максимум 50 SessionTaskLink на задачу.** Старые архивируются в отдельный файл.

5. **Summary генерируется автоматически.** Агент в конце сессии обязан написать краткое резюме. Это ключевое поле для быстрого понимания истории задачи.

### 4.6. Пример

```
SessionTaskLink {
  id: "stl-042",
  session_id: "owl:20260511:003",
  task_id: "task-123",
  started_at: "2026-05-11T10:00:00Z",
  ended_at: "2026-05-11T12:30:00Z",
  agent_id: "agent-owl",
  agent_name: "OWL",
  actions: [
    {
      timestamp: "2026-05-11T10:05:00Z",
      type: "code_write",
      description: "Создал миграцию для поля dependencies",
      files_changed: ["server/migrations/003_deps.sql"]
    },
    {
      timestamp: "2026-05-11T10:30:00Z",
      type: "code_edit",
      description: "Добавил поле dependencies в Task interface",
      files_changed: ["src/shared/types.ts"]
    },
    {
      timestamp: "2026-05-11T11:00:00Z",
      type: "test_run",
      description: "Написал тесты валидации зависимостей",
      files_changed: ["tests/api/tasks.test.ts"]
    },
    {
      timestamp: "2026-05-11T12:00:00Z",
      type: "decision_made",
      description: "Решение: хранить зависимости как массив ID, а не отдельную таблицу",
      files_changed: []
    }
  ],
  summary: "Реализовал поле dependencies в Task API. Миграция, типы, валидация, тесты. Циклические зависимости проверяются на сервере.",
  status: "completed",
  outcome: "Полностью реализовано. Все тесты проходят.",
  artifacts: [
    { type: 'file', path: 'server/migrations/003_deps.sql', description: 'Миграция БД', created: true },
    { type: 'file', path: 'tests/api/tasks.test.ts', description: 'Тесты зависимостей', created: false },
    { type: 'commit', path: 'abc1234', description: 'feat: add task dependencies', created: true }
  ],
  related_sessions: ["owl:20260510:001"]
}
```

## 5. Память решений (DecisionLog)

### 5.1. Назначение

DecisionLog — структурированный журнал архитектурных и проектных решений. Каждое решение фиксируется с контекстом, альтернативами и последствиями. Это ADR (Architecture Decision Record) в машиночитаемом формате.

### 5.2. Структура

```
Decision {
  id: string
  task_id: string | null        // ссылка на задачу (если решение связано с задачей)
  epic_id: string | null        // ссылка на эпик (если решение на уровне эпика)

  // Суть решения
  title: string                 // краткое название решения
  context: string               // какая проблема/ситуация привела к решению
  decision: string              // что решили (конкретно)
  rationale: string             // почему так (обоснование)

  // Альтернативы
  alternatives: Alternative[]    // какие варианты рассматривали

  // Последствия
  consequences: {
    positive: string[]           // позитивные последствия
    negative: string[]           // негативные последствия (trade-offs)
    risks: string[]              // риски
  }

  // Статус
  status: 'proposed' | 'accepted' | 'rejected' | 'superseded' | 'deprecated'
  superseded_by: string | null  // ID решения, которое заменило это

  // Метаданные
  created_at: string
  updated_at: string
  created_by: string            // кто предложил
  decided_by: string            // кто утвердил
  session_id: string | null     // сессия, в которой принято решение
  tags: string[]                // теги для поиска
}
```

### 5.3. Alternative — рассмотренная альтернатива

```
Alternative {
  option: string                // что рассматривали
  pros: string[]                // плюсы
  cons: string[]                // минусы
  reason_rejected: string       // почему отвергли
}
```

### 5.4. Правила

1. **Одно решение — один Decision.** Не смешивайте несколько решений в одну запись.

2. **Decision может быть независимым.** Не каждое решение привязано к задаче. Некоторые — на уровне проекта или лаборатории.

3. **Superseded, не deleted.** Когда решение пересматривается — старое помечается `superseded`, создаётся новое. Старое не удаляется.

4. **Максимум 500 Decision в основном файле.** Старые архивируются.

5. **Tags обязательны.** Минимум 2 тега для каждого решения. Примеры: `architecture`, `database`, `api`, `security`, `performance`, `ux`.

### 5.5. Пример

```
Decision {
  id: "dec-007",
  task_id: "task-123",
  epic_id: "epic-003",
  title: "Хранение зависимостей задач в массиве, а не в отдельной таблице",
  context: "Нужно реализовать зависимости между задачами. Рассмотрели два варианта хранения.",
  decision: "Хранить зависимости как массив идентификаторов (dependencies: string[]) прямо в задаче.",
  rationale: "Для single-user JSON-first архитектуры отдельная таблица избыточна. Массив проще реализовать, читать и поддерживать. Запросы к зависимостям редкие — производительность не пострадает.",
  alternatives: [
    {
      option: "Отдельная таблица task_dependencies (task_id, depends_on_id)",
      pros: ["Нормализация данных", "Быстрые запросы на поиск зависимостей", "Масштабируемость"],
      cons: ["Сложнее реализация", "Нужна отдельная миграция", "Избыточно для JSON-first"],
      reason_rejected: "Избыточная сложность для текущего масштаба. Если перейдём на PostgreSQL — пересмотрим."
    }
  ],
  consequences: {
    positive: ["Простая реализация", "Меньше кода", "Быстрый старт"],
    negative: ["Нет индексов на зависимости", "Поиск всех зависимостей задачи — O(n)"],
    risks: ["При росте числа задач >1000 возможна деградация производительности"]
  },
  status: "accepted",
  superseded_by: null,
  created_at: "2026-05-11T12:00:00Z",
  updated_at: "2026-05-11T12:00:00Z",
  created_by: "agent-owl",
  decided_by: "human-eugene",
  session_id: "owl:20260511:003",
  tags: ["architecture", "database", "tasks", "dependencies"]
}
```

## 6. Поиск по истории задач

### 6.1. Назначение

Полнотекстовый поиск по всем слоям контекста: задачи, контексты, сессии, решения, changelog. Основной инструмент навигации по накопленным знаниям.

### 6.2. Индексируемые поля

| Источник | Поля для индексации | Вес релевантности |
|----------|---------------------|-------------------|
| Task | title, description, tags | 1.0 (базовый) |
| TaskContext | why, who, notes, acceptance_criteria | 1.5 (контекст важнее) |
| SessionTaskLink | summary, outcome, actions[].description | 0.8 (история) |
| Decision | title, context, decision, rationale, alternatives[].option | 1.2 (решения важны) |
| Changelog | action, entity_type, diff | 0.5 (факты) |

### 6.3. Режимы поиска

**Простой поиск (по умолчанию):**
- Ввод: строка запроса
- Поиск по всем индексируемым полям
- Ранжирование по весу релевантности + свежести
- Максимум 20 результатов

**Расширенный поиск:**
- Фильтр по источнику: `source:task` | `source:context` | `source:session` | `source:decision`
- Фильтр по дате: `after:2026-05-01` | `before:2026-05-11`
- Фильтр по агенту: `agent:owl` | `agent:cat`
- Фильтр по статусу задачи: `status:done` | `status:in_progress`
- Фильтр по тегу: `tag:architecture` | `tag:security`
- Точное совпадение: `"exact phrase"`
- Комбинация: `source:decision tag:architecture after:2026-05-01`

**Поиск по графу:**
- Ввод: ID задачи или решения
- Результат: подграф связей (задача → контекст → сессии → решения → связанные задачи)
- Глубина: настраиваемая (1-3 прыжка)

### 6.4. Формат результата

```
SearchResult {
  query: string
  total: number
  results: SearchHit[]
}

SearchHit {
  id: string
  type: 'task' | 'context' | 'session_link' | 'decision' | 'changelog'
  title: string                 // заголовок найденного объекта
  snippet: string               // фрагмент текста с подсветкой совпадений
  score: number                 // релевантность (0-1)
  source: string                // источник (task id, decision id и т.д.)
  task_id: string | null        // связанная задача (если есть)
  created_at: string
  updated_at: string
  highlights: string[]          // подсвеченные фрагменты
}
```

### 6.5. Реализация поиска

Поскольку система JSON-first и single-user, полноценная поисковая 引擎 (Elasticsearch, Meilisearch) избыточна. Достаточно:

1. **In-memory инвертированный индекс** — при старте сервера строится Map<term, Set<docId>>.
2. **TF-IDF ранжирование** — упрощённое, с учётом весов полей.
3. **Fuzzy matching** — расстояние Левенштейна <= 2 для опечаток.
4. **Перестроение индекса** — при каждом изменении myrmex.json (writeState).

Оценка: при 10000 документов и средней длине 200 слов — индекс займёт ~5MB памяти, поиск <50ms.
