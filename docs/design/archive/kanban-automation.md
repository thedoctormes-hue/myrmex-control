## 1. Обзор архитектуры

```
┌─────────────────────────────────────────────────────────┐
│                    ОБЩИЙ БЭКЛОГ                          │
│  (единый пул задач с весами, приоритетами, тегами)      │
└──────────────────────┬──────────────────────────────────┘
                       │
            ┌──────────▼──────────┐
            │  ДИСПЕТЧЕР ВЕСОВ   │
            │  (Weight Router)    │
            └──────────┬──────────┘
                       │
          ┌────────────┼────────────┐
          │            │            │
   ┌──────▼─────┐ ┌───▼────┐ ┌────▼─────┐
   │  ЗАВЛАБ    │ │ МУРАВЕЙ│ │   КОТ    │
   │  (IRL)     │ │ (TG #1)│ │ (TG #2)  │
   └────────────┘ └────────┘ └──────────┘
```

**Принцип:** Бэклог — единый источник правды. Доски — проекции с фильтрацией по каналу. Задача физически живёт в бэклоге, на досках — только ссылки + статус выполнения.

## 2. Система весов (Weight Model)

### 2.1. Состав веса задачи

Каждая задача имеет **итоговый вес** `W_total`, вычисляемый как сумма компонентов:

```
W_total = W_priority + W_age + W_deps + W_channel + W_recurring
```

| Компонент | Диапазон | Описание |
|-----------|----------|----------|
| `W_priority` | 0–40 | Базовый приоритет (P0–P4) |
| `W_age` | 0–20 | Штраф за старение (растёт со временем) |
| `W_deps` | 0–15 | Бонус за разблокировку зависимых задач |
| `W_channel` | -10..+10 | Корректировка под канал (МУРАВЕЙ/КОТ) |
| `W_recurring` | 0–15 | Штраф за просрочку повторяющейся задачи |

### 2.2. Приоритеты (W_priority)

| Уровень | Вес | Описание | SLA реакции |
|---------|-----|----------|-------------|
| P0 — Критический | 40 | Прод лежит, инцидент | 15 мин |
| P1 — Высокий | 30 | Блокер, дедлайн < 24ч | 1 час |
| P2 — Средний | 20 | Стандартная работа | 4 часа |
| P3 — Низкий | 10 | Улучшения, рефакторинг | 24 часа |
| P4 — Фоновый | 0 | Бэклоговый шум | Нет SLA |

### 2.3. Старение (W_age)

```
W_age = min(20, floor(days_since_creation / 2))
```

Каждые 2 дня задача получает +1 к весу. Максимум — 20. Цель: задачи не гниют в бэклоге.

**Анти-гниение:** если задача старше 14 дней и не взята — автоматический эскалационный эвент.

### 2.4. Зависимости (W_deps)

```
W_deps = min(15, count(blocked_tasks) * 5)
```

Если задача блокирует другие — бонус за разблокировку. Каждая заблокированная задача даёт +5. Максимум — 15.

### 2.5. Канальная корректировка (W_channel)

Задача может иметь теги `channel:muravey`, `channel:kat`, `channel:any`.

- `channel:muravey` → на доске МУРАВЕЙ: +10, на доске КОТ: -10
- `channel:kat` → на доске КОТ: +10, на доске МУРАВЕЙ: -10
- `channel:any` или нет тега → 0 на обеих досках

### 2.6. Повторяющиеся задачи (W_recurring)

```
W_recurring = min(15, floor(days_overdue * 3))
```

Каждый день просрочки повторяющейся задачи даёт +3. Максимум — 15.

## 3. Алгоритм распределения (Weight Router)

### 3.1. Триггеры диспетчера

Диспетчер запускается при:
1. **Новый тикет в бэклоге** — немедленная маршрутизация
2. **Завершение задачи** — пересчёт нагрузки, ребалансировка
3. **Таймер** — каждые 5 минут (cron)
4. **Ручной вызов** — API endpoint `/api/kanban/rebalance`

### 3.2. Алгоритм маршрутизации

```
1. Загрузить все незавершённые задачи из бэклога
2. Для каждой задачи:
   a. Вычислить W_total для МУРАВЕЯ и W_total для КОТА
   b. Если задача привязана к каналу — отправить туда
   c. Если channel:any — выбрать канал с меньшей текущей нагрузкой
3. Рассчитать нагрузку канала:
   Load(channel) = Σ W_total всех активных задач канала
4. Балансировка:
   - Если |Load(МУРАВЕЙ) - Load(КОТ)| > threshold (20):
     - Перенести задачу с наименьшим |W_channel| на менее загруженный канал
     - Повторять пока разница < threshold
5. Обновить проекции на досках
```

### 3.3. Ограничения маршрутизации

- Задача с `channel:muravey` / `channel:kat` **не переносится** на другой автоматически
- Задача с `assignee_id` закреплена за исполнителем, канал определяется по его профилю
- Задачи ЗАВЛАБ (IRL) **не маршрутизируются** — живут только на своей доске
- Перенос между каналами логируется в audit trail

### 3.4. Поддержка третьего канала

Архитектура диспетчера — N-канальная. Добавление третьего канала:

1. Добавить `channel:new_channel` в теги
2. Зарегистрировать канал в конфиге `kanban.channels`
3. Диспетчер автоматически начитывает все каналы из конфига
4. Порог балансировки пересчитывается: `threshold = base_threshold * (N - 1)`

## 4. Автоматические переходы статусов

### 4.1. Жизненный цикл задачи

```
BACKLOG → READY → IN_PROGRESS → IN_REVIEW → DONE
  │          │          │            │         │
  │          │          │            │         └─→ ARCHIVED (через 7 дней)
  │          │          │            │
  │          │          └────────────┴─→ BLOCKED (ручной переход)
  │          │
  └──────────┴─→ CANCELLED (ручной переход)
```

### 4.2. Правила автоматических переходов

| Триггер | Из статуса | В статус | Условие |
|---------|-----------|----------|---------|
| Задача назначена на исполнителя | BACKLOG | READY | assignee_id установлен + DoR выполнен |
| Исполнитель начал работу | READY | IN_PROGRESS | Первый коммит / комментарий / таймер запущен |
| PR создан / работа завершена | IN_PROGRESS | IN_REVIEW | Ссылка на PR или отметка "ready for review" |
| Ревью пройдено | IN_REVIEW | DONE | Аппрув ревьюера |
| Задача в DONE > 7 дней | DONE | ARCHIVED | Автоматически по таймеру |
| Обнаружена зависимость на неготовую задачу | READY | BLOCKED | Зависимость не в DONE |
| Зависимость завершена | BLOCKED | READY | Все зависимости в DONE |
| Задача в BACKLOG > 14 дней без движения | BACKLOG | — | Эскалация (уведомление, не смена статуса) |

### 4.3. Защитные условия

- **IN_PROGRESS → IN_REVIEW**: минимум 30 минут в работе (защита от случайного клика)
- **IN_REVIEW → DONE**: требуется минимум 1 аппрув от другого участника
- **Авто-переходы не применяются** к задачам с тегом `manual:only`

## 5. Уведомления

### 5.1. Каналы доставки

| Канал | Назначение |
|-------|-----------|
| Telegram (МУРАВЕЙ) | Задачи канала МУРАВЕЙ |
| Telegram (КОТ) | Задачи канала КОТ |
| Email | Эскалации, дайджесты |
| Webhook | Интеграции (Slack, Discord и т.д.) |

### 5.2. Таблица уведомлений

| Событие | Кому | Канал | Задержка |
|---------|------|-------|----------|
| Новая задача назначена | Исполнителю | TG | Мгновенно |
| Задача перешла в BLOCKED | Исполнителю + менеджеру | TG | Мгновенно |
| Задача перешла в IN_REVIEW | Ревьюеру | TG | Мгновенно |
| Задача перешла в DONE | Создателю + команде | TG | Мгновенно |
| Задача просрочена (SLA) | Исполнителю + менеджеру | TG + Email | Мгновенно |
| Задача в BACKLOG > 14 дней | Менеджеру | Email | Ежедневный дайджест |
| Ребалансировка каналов | Админу | TG | Мгновенно |
| Ежедневный дайджест | Всем | TG | 09:00 по расписанию |
| Еженедельный отчёт | Менеджеру | Email | Понедельник 10:00 |

### 5.3. Правила анти-спама

- **Дедупликация:** не более 1 уведомления по одной задаче за 30 минут
- **Батчинг:** уведомления группируются в окне 5 минут (кроме P0)
- **P0 — всегда мгновенно**, без батчинга
- **Тихие часы:** 23:00–08:00 — только P0, остальное — в утренний дайджест

## 6. Повторяющиеся задачи (Recurring Tasks)

### 6.1. Модель

Каждая повторяющаяся задача имеет шаблон:

```
{
  pattern: "daily" | "weekly" | "monthly" | "cron",
  cron_expr: "0 9 * * 1-5",  // если pattern = "cron"
  template: { title, description, priority, assignee_id, tags },
  due_time: "10:00",          // крайний срок в день создания
  auto_assign: true,
  max_overdue: 3              // после N просрочек — эскалация
}
```

### 6.2. Жизненный цикл повторяющейся задачи

```
1. По расписанию создаётся новый экземпляр из шаблона
2. Экземпляр попадает в бэклог с W_recurring = 0
3. Диспетчер маршрутирует как обычную задачу
4. Если не выполнена к due_time:
   W_recurring начинаёт расти (+3/день)
5. Если просрочена > max_overdue дней:
   - Эскалация менеджеру
   - Автоматическое повышение приоритета (+1 уровень)
```

### 6.3. Примеры шаблонов

| Шаблон | Расписание | Приоритет | Канал |
|--------|-----------|-----------|-------|
| Утренний стендап-репорт | Ежедневно 09:00 | P2 | channel:any |
| Бэкап базы | Ежедневно 02:00 | P1 | channel:muravey |
| Обновление зависимостей | Понедельник 10:00 | P3 | channel:kat |
| Ревью безопасности | 1-е число месяца | P1 | channel:muravey |

## 7. Definition of Ready (DoR)

Задача переходит из BACKLOG в READY **автоматически** при выполнении ВСЕХ условий:

### 7.1. Чеклист DoR

- [ ] **Заголовок** — не пустой, минимум 10 символов
- [ ] **Описание** — не пустое, содержит контекст (что, зачем, ожидаемый результат)
- [ ] **Приоритет** — установлен (не P4 по умолчанию, если задача не фоновая)
- [ ] **Исполнитель** — назначен (`assignee_id`) ИЛИ тег `unassigned-ok`
- [ ] **Теги** — минимум 1 тег из разрешённого набора
- [ ] **Зависимости** — все зависимости в статусе DONE ИЛИ зависимостей нет
- [ ] **Оценка** — указана (story points или time estimate) ИЛИ тег `no-estimate`
- [ ] **Канал** — определён (тег `channel:*` или исполнитель привязан к каналу)

### 7.2. Автоматическая проверка DoR

При каждом изменении задачи система проверяет DoR. Если все условия выполнены — автоматический переход `BACKLOG → READY`. Если условие нарушено (например, зависимость откатилась из DONE) — переход `READY → BACKLOG` с уведомлением.

### 7.3. DoR для повторяющихся задач

Шаблон повторяющейся задачи должен иметь полный DoR в шаблоне. Экземпляр создаётся уже в статусе READY.

---

## 8. Definition of Done (DoD)

Задача переходит из IN_REVIEW в DONE **автоматически** при выполнении ВСЕХ условий:

### 8.1. Чеклист DoD

- [ ] **Код написан** — есть ссылка на PR / коммит / артефакт
- [ ] **Тесты пройдены** — CI green (если применимо) ИЛИ тег `no-tests`
- [ ] **Ревью пройдено** — минимум 1 аппрув от другого участника
- [ ] **Документация обновлена** — если есть тег `needs-docs`
- [ ] **Миграции применены** — если есть тег `has-migration`
- [ ] **Чейнджлог обновлён** — если есть тег `user-facing`

### 8.2. Автоматическая проверка DoD

- Интеграция с CI: проверка статуса пайплайна
- Интеграция с Git: проверка аппрувов PR
- Если все условия выполнены — автоматический переход `IN_REVIEW → DONE`
- Если условие нарушено — возврат в `IN_PROGRESS` с комментарием

### 8.3. DoD для IRL-задач (ЗАВЛАБ)

Упрощённый DoD:
- [ ] **Фото/подтверждение** — приложено
- [ ] **Отметка о выполнении** — от исполнителя или менеджера

## 9. Интеграция между досками

### 9.1. Модель связей

```
┌──────────────┐     mirrors     ┌──────────────┐
│  ЗАВЛАБ      │◄──────────────►│  БЭКЛОГ      │
│  (IRL)       │                │  (source of  │
└──────────────┘                │   truth)     │
                                └──────┬───────┘
                                       │
                          ┌────────────┼────────────┐
                          │            │            │
                   ┌──────▼─────┐ ┌───▼────┐ ┌────▼─────┐
                   │  МУРАВЕЙ   │ │  КОТ   │ │ Канал N  │
                   │  (TG #1)   │ │(TG #2) │ │(future)  │
                   └────────────┘ └────────┘ └──────────┘
```

### 9.2. Правила синхронизации

| Событие в бэклоге | Действие на досках |
|-------------------|-------------------|
| Создана задача | Создана проекция на целевой доске |
| Изменён статус | Статус обновлен на всех досках |
| Изменён приоритет | Пересчитан вес, возможна перемаршрутизация |
| Задача завершена | Проекция архивирована на доске |
| Задача удалена | Проекция удалена со всех досок |

### 9.3. Кросс-досковые зависимости

Задача на доске МУРАВЕЙ может зависеть от задачи на доске КОТ и наоборот. Зависимости хранятся в бэклоге (source of truth), на досках — только отображение статуса зависимости.

**Визуализация на доске:**
- Зависимость на той же доске — сплошная стрелка
- Зависимость на другой доске — пунктирная стрелка + иконка канала

### 9.4. Конфликты синхронизации

Если задача отредактирована одновременно на доске и в бэклоге:
1. **Приоритет — бэклог** (source of truth)
2. Конфликт полей разрешён по принципу: последнее изменение выигрывает (timestamp-based)
3. Лог конфликта записывается в audit trail

## 11. Метрики и мониторинг

### 11.1. KPI канбан-системы

| Метрика | Описание | Целевое значение |
|---------|----------|-----------------|
| Lead Time | От создания до завершения | < 3 дней (P2) |
| Cycle Time | От IN_PROGRESS до DONE | < 1 день (P2) |
| Throughput | Задач в неделю | Рост 10% месяц к месяцу |
| WIP Limit | Активных задач на канал | < 10 на канал |
| Blocked Ratio | % заблокированных задач | < 15% |
| SLA Compliance | % задач в рамках SLA | > 90% |
| Balance Index | Разница нагрузки каналов | < threshold |

### 11.2. Дашборд

- Визуализация нагрузки по каналам (график)
- Распределение задач по статусам (воронка)
- Тепловая карта активности (по дням/часам)
- Топ-5 самых старых задач
- Топ-5 заблокированных задач
