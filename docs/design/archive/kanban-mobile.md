# Мобильный опыт канбан-досок — концепция

> **Версия:** 1.0 | **Дата:** 2026-05-11 | **Автор:** Mobile-first UX инженер (OWL)
>
> **Контекст:** Myrmex Control — PWA + Telegram Web App. Текущий Board.tsx использует HTML5 drag-and-drop, который не работает на мобильных. 5 колонок горизонтально не помещаются на экране телефона. BottomBar: 5 пунктов навигации. Touch optimization уже частично есть: min 44px tap targets, font-size 16px для inputs.

---

## 1. Мобильный Layout

### 1.1. Проблема

Текущий Board.tsx рендерит 5 колонок (`flex` + `overflow-x-auto`) — на телефоне это приводит к горизонтальному скроллу, который конфликтует с нативным жестом «назад» в мобильных браузерах и TWA. Карточки сжимаются до нечитаемости или пользователь скроллит влево-вправо, теряя контекст.

### 1.2. Решение: Одна колонка + горизонтальный таб-переключатель

**Принцип:** На экране всегда видна ровно одна колонка. Переключение между колонками — через горизонтальный ряд табов вверху экрана.

```
┌─────────────────────────────────┐
│  ← Проект «Альфа»          [+]  │  ← хедер (sticky)
├─────────────────────────────────┤
│ [Бэклог] [К выполн] [В работе] →│  ← табы колонок (scrollable)
│                                 │
│ ┌─────────────────────────────┐ │
│ │ Задача 1                    │ │  ← карточки (vertical scroll)
│ │ приоритет · теги            │ │
│ └─────────────────────────────┘ │
│ ┌─────────────────────────────┐ │
│ │ Задача 2                    │ │
│ │ приоритет · теги            │ │
│ └─────────────────────────────┘ │
│         ...                     │
│                                 │
├─────────────────────────────────┤
│  🏠    📁    📚    📈    📋    │  ← BottomBar
└─────────────────────────────────┘
```

**Детали реализации:**

- **Табы колонок** — горизонтальный скроллируемый ряд чипов вверху экрана (под хедером). Активная колонка выделена цветом (из текущей `color` конфигурации колонки). Каждый таб показывает название + счётчик задач.
- **Свайп между колонками** — поверх зоны карточек обрабатывается горизонтальный свайп (swipe left/right). Свайп влево = следующая колонка, вправо = предыдущая. Анимация перехода — slide (как ViewPager в Android). Это уже частично реализовано в `useSwipeNav.ts`, но сейчас он переключает страницы, а не колонки.
- **Вертикальный скролл карточек** — внутри активной колонки карточки скроллятся вертикально. Никакого горизонтального скролла на мобильных.
- **Sticky-хедер** — название проекта и кнопка создания задачи закреплены вверху. При скролле карточек хедер остаётся видимым.
- **Sticky-табы** — табы колонок закреплены под хедером (при скролле карточек табы остаются на месте).

**Почему не другие варианты:**

| Вариант | Почему отклонён |
|---------|----------------|
| Вертикальный скролл всех колонок | Пользователь теряет контекст — не видит соседние колонки, не понимает где находится. 5 колонок × ~10 карточек = 50 карточек в одном потоке — хаос. |
| Аккордеон (раскрывающиеся колонки) | Требует много тапов для навигации. Нельзя видеть две колонки одновременно для сравнения. |
| Миниатюры колонок (как Trello mobile) | Карточки становятся нечитаемыми. Теряется смысл канбан-доски. |

### 1.3. Адаптивный breakpoint

Переключение между desktop (5 колонок) и mobile (1 колонка + табы) происходит на `md` (768px):

- `>= 768px` — текущий layout: все 5 колонок в горизонтальном скролле
- `< 768px` — mobile layout: одна колонка + табы

На планшетах (768–1024px) можно показывать 2–3 колонки одновременно с уменьшенными карточками.

### 1.4. Индикатор текущей позиции

В правой части таб-бара показывается индикатор вида `2/5` (текущая колонка / общее количество). При свайпе индикатор анимированно обновляется.

## 2. Touch-взаимодействия

### 2.1. Проблема

Текущий Board.tsx использует HTML5 Drag and Drop API (`draggable`, `onDragStart`, `onDrop`). Этот API **не работает на мобильных браузерах** — нет нативного touch-события `drag`. Карточки нельзя переместить пальцем.

### 2.2. Решение: трёхрежимная модель взаимодействия

#### Режим 1: Тап — открытие карточки

**Однократный тап** на карточку открывает **Bottom Sheet** (модальная панель снизу) с полной информацией о задаче:

```
┌─────────────────────────────────┐
│  ═══ (handle для свайпа вниз)   │
│                                 │
│  Название задачи                │
│  ─────────────────────────────  │
│  Описание задачи...             │
│                                 │
│  Приоритет: 🔴 Высокий          │
│  Теги: #backend #urgent         │
│  Создано: 2026-05-10            │
│                                 │
│  ┌───────────────────────────┐  │
│  │  ✏️ Редактировать          │  │
│  └───────────────────────────┘  │
│  ┌───────────────────────────┐  │
│  │  ➡️ Переместить →          │  │
│  └───────────────────────────┘  │
│  ┌───────────────────────────┐  │
│  │  🗑️ Удалить               │  │
│  └───────────────────────────┘  │
│                                 │
└─────────────────────────────────┘
```

**Детали:**
- Bottom Sheet открывается с анимацией slide-up (300ms ease-out)
- Handle-бар вверху для визуальной подсказки
- Свайп вниз по Bottom Sheet закрывает его
- Затемнение фона (backdrop) с тапом для закрытия
- Haptic feedback: `selectionChanged()` при открытии

#### Режим 2: «Переместить» — выбор целевой колонки

При нажатии кнопки «Переместить →» в Bottom Sheet открывается **второй экран** (или заменяет содержимое Bottom Sheet) со списком колонок:

```
┌─────────────────────────────────┐
│  ← Переместить задачу           │
│                                 │
│  Выберите колонку:              │
│                                 │
│  ○ Бэклог              (3)      │
│  ● К выполнению        (5)  ← текущая
│  ○ В работе            (2)      │
│  ○ На проверке         (1)      │
│  ○ Готово              (8)      │
│                                 │
│  ┌───────────────────────────┐  │
│  │      Переместить           │  │
│  └───────────────────────────┘  │
│                                 │
└─────────────────────────────────┘
```

**Детали:**
- Текущая колонка отмечена и визуально выделена
- Счётчик задач в каждой колонке помогает принять решение
- Тап на колонку → мгновенное перемещение + haptic `impact('medium')` + toast «Перемещено в В работе»
- BackButton (TWA) возвращает к карточке

#### Режим 3: Long press — быстрое перемещение вправо

**Долгое нажатие** (500ms) на карточку — быстрое действие: перемещение задачи в **следующую** колонку (по порядку: backlog → todo → in_progress → review → done).

**Детали:**
- При long press: haptic `impact('heavy')` + визуальный feedback (карточка подсвечивается, scale до 0.97)
- Если задача в последней колонке (done) — long press перемещает в backlog (цикл)
- Toast с подтверждением: «Перемещено в В работе» + кнопка «Отменить» (undo, 3 секунды)
- Это самый частый сценарий: «задача готова, двигаю вправо» — должен выполняться одним жестом

### 2.3. Swipe по карточке (дополнительно)

**Swipe влево** по карточке — раскрывает кнопку быстрого действия (как в почтовых клиентах):

- Первая зона (swipe на 40–80px): показывает кнопку «Переместить →»
- Вторая зона (swipe > 80px): показывает кнопку «Удалить» (красная)
- Отпускание без довода до зоны — возврат в исходное положение (spring animation)

**Swipe вправо** по карточке — быстрое перемещение в следующую колонку (аналог long press, но быстрее).

### 2.4. Сводка жестов

| Жест | Действие | Haptic |
|------|----------|--------|
| Тап | Открыть карточку (Bottom Sheet) | `selectionChanged` |
| Long press (500ms) | Быстро переместить вправо | `impact('heavy')` |
| Swipe влево (короткий) | Показать «Переместить» | `selectionChanged` |
| Swipe влево (длинный) | Показать «Удалить» | `impact('medium')` |
| Swipe вправо | Быстро переместить вправо | `impact('light')` |
| Свайп по зоне карточек (горизонт.) | Переключить колонку | `selectionChanged` |
| Свайп вниз по Bottom Sheet | Закрыть | — |
| Тап по backdrop | Закрыть | — |

## 3. Создание задач на мобильных

### 3.1. Проблема

Текущий Board.tsx использует inline-форму (input + кнопка «Создать»), которая появляется при нажатии «+ Задача». На мобильных это занимает ценное пространство, а клавиатура перекрывает форму.

### 3.2. Решение: FAB + Full-screen форма

#### Плавающая кнопка (FAB)

В правом нижнем углу экрана (над BottomBar) — плавающая кнопка «+»:

```
┌─────────────────────────────────┐
│                                 │
│  (контент доски)                │
│                                 │
│                    ┌─────────┐  │
│                    │    +    │  │  ← FAB (56x56dp, emerald)
│                    └─────────┘  │
│  🏠  📁  📚  📈  📋             │
└─────────────────────────────────┘
```

**Детали:**
- Размер: 56x56dp (Material Design стандарт для FAB)
- Цвет: `--primary` (emerald)
- Иконка: `Plus` (Lucide)
- Haptic: `impact('medium')` при нажатии
- При скролле вниз FAB скрывается, при скролле вверх — появляется (поведение как в Telegram)
- На экране создания задачи FAB скрыт

#### Full-screen форма создания

При нажатии FAB открывается **полноэкранная форма** (не модалка, а отдельный «экран» с анимацией slide-up):

```
┌─────────────────────────────────┐
│  ← Новая задача            ✓    │  ← хедер: назад / сохранить
├─────────────────────────────────┤
│                                 │
│  Название *                     │
│  ┌───────────────────────────┐  │
│  │                           │  │  ← autoFocus, font-size 16px
│  └───────────────────────────┘  │
│                                 │
│  Описание                       │
│  ┌───────────────────────────┐  │
│  │                           │  │  ← textarea, min-height 100px
│  │                           │  │
│  └───────────────────────────┘  │
│                                 │
│  Приоритет                      │
│  [Низкий] [Средний] [Высокий]   │  ← chip-селектор
│  [Критический]                  │
│                                 │
│  Теги                           │
│  [+ Добавить тег]               │  ← чипы + ввод
│                                 │
│  Колонка                        │
│  [Бэклог ▼]                     │  ← select, по умолчанию текущая
│                                 │
└─────────────────────────────────┘
```

**Детали:**
- Название — обязательное поле, autoFocus, клавиатура открывается автоматически
- Приоритет — chip-селектор (не dropdown), текущий по умолчанию: «Средний»
- Теги — ввод с автодополнением из существующих тегов
- Колонка — по умолчанию текущая открытая колонка
- Кнопка «✓» в хедере — сохранить и вернуться к доске
- BackButton (TWA) — закрыть без сохранения (с подтверждением если есть введённые данные)
- После сохранения: haptic `notification('success')` + toast «Задача создана» + прокрутка к новой карточке

#### Быстрое создание (из MainButton TWA)

При пустой колонке (0 задач) — вместо пустого состояния показывается кнопка «Создать первую задачу», которая открывает упрощённую форму (только название + кнопка создать).

### 3.3. Голосовой ввод (TWA)

Telegram Web App поддерживает голосовой ввод через `WebApp.showPopup()` или нативный `input[type="file"]` с `accept="audio/*"`. Однако более надёжный подход:

- Кнопка микрофона в поле «Название» (иконка Mic от Lucide)
- При нажатии: `WebApp.showPopup()` с просьбой отправить голосовое сообщение
- Полученный голос → отправка на Whisper API → текст в поле «Название»
- Fallback: нативный Speech Recognition API (`webkitSpeechRecognition`) для браузеров

## 4. Telegram Web App интеграция

### 4.1. Текущее состояние

TWA интеграция уже реализована в `src/client/shared/lib/twa.ts`: `initTWA()`, `haptic()`, `isTWA()`, `getTWAUser()`, `getTWATheme()`, MainButton, BackButton. Но MainButton и BackButton **не используются на странице Board**.

### 4.2. Haptic feedback — карта событий

| Событие | Haptic тип | Стиль |
|---------|-----------|-------|
| Тап по карточке | `selectionChanged` | — |
| Long press | `impactOccurred` | `heavy` |
| Перемещение задачи | `impactOccurred` | `medium` |
| Удаление задачи | `notificationOccurred` | `warning` |
| Создание задачи | `notificationOccurred` | `success` |
| Ошибка (сеть) | `notificationOccurred` | `error` |
| Переключение колонки (свайп) | `selectionChanged` | — |
| Достижение края колонки (нельзя дальше) | `impactOccurred` | `light` |

### 4.3. MainButton — контекстное поведение

**На экране доски (просмотр):**
- MainButton виден, текст: «+ Новая задача», цвет: `--primary`
- Нажатие → открыть full-screen форму создания
- Haptic: `impact('medium')`

**На экране создания/редактирования задачи:**
- MainButton виден, текст: «Сохранить», цвет: `--primary`
- Нажатие → валидация + сохранение + возврат к доске
- Если форма невалидна (пустое название) → MainButton `disable()`, при вводе → `enable()`

**На экране выбора колонки (перемещение):**
- MainButton скрыт (`hide()`)
- Выбор колонки происходит тапом

### 4.4. BackButton — контекстное поведение

**На экране доски (просмотр):**
- BackButton виден
- Нажатие → навигация назад (к списку проектов)

**На экране создания задачи:**
- BackButton виден
- Нажатие → если форма пуста: закрыть. Если есть данные: `WebApp.showConfirm('Отменить создание?')` → закрыть

**На Bottom Sheet (карточка открыта):**
- BackButton виден
- Нажатие → закрыть Bottom Sheet

**На экране выбора колонки:**
- BackButton виден
- Нажатие → вернуться к карточке (Bottom Sheet)

### 4.5. Тема из Telegram

`getTWATheme()` уже реализован. На мобильных нужно расширить интеграцию:

- Использовать `themeParams.button_color` для цвета FAB и MainButton
- Использовать `themeParams.bg_color` для фона Bottom Sheet
- Использовать `themeParams.text_color` для текста карточек
- При изменении темы в Telegram (переключение тёмная/светлая) — мгновенное обновление без перезагрузки

### 4.6. Вибрация при закрытии

`enableClosingConfirmation()` уже вызывается в `initTWA()`. На странице доски это работает корректно — если пользователь случайно смахнёт приложение, Telegram покажет подтверждение.

## 5. Offline режим

### 5.1. Текущее состояние

PWA настроен через `vite-plugin-pwa` с Workbox. `globPatterns: ['**/*.{js,css,html,svg,woff2}']` — кэшируются только статические ресурсы. **API-запросы не кэшируются.** При отсутствии сети доска показывает пустой экран или ошибку.

### 5.2. Стратегия: Optimistic UI + Action Queue

#### 5.2.1. Кэширование состояния

- При каждом успешном `getState()` — сохранять полное состояние в `localStorage` (ключ: `myrmex_state_cache`) + timestamp
- При загрузке приложения: сначала показать кэшированные данные (instant), затем фоновый запрос к API
- Если API недоступен — показать кэш + индикатор «Офлайн» в хедере (иконка `CloudOff` от Lucide, amber цвет)

#### 5.2.2. Очередь действий (Action Queue)

Все мутации (create, move, delete, update) при недоступности сети ставятся в очередь:

```typescript
interface QueuedAction {
  id: string;           // UUID
  timestamp: number;    // Date.now()
  type: 'create' | 'move' | 'delete' | 'update';
  payload: unknown;     // данные действия
  retryCount: number;   // счётчик попыток
}
```

**Поведение:**
1. Пользователь перемещает задачу → оптимистично обновляется UI (карточка перемещается мгновенно) → действие отправляется в API
2. Если API недоступен → действие ставится в очередь (`localStorage`, ключ: `myrmex_action_queue`)
3. Карточка визуально помечается индикатором «ожидает синхронизации» (точка `--warning` цветом в углу карточки)
4. При восстановлении связи → очередь последовательно отправляется
5. При конфликте (сервер вернул другую версию) — показать toast с предупреждением

#### 5.2.3. Синхронизация

- Мониторинг сети: `navigator.onLine` + события `online`/`offline`
- При событии `online` → автоматическая отправка очереди
- Между попытками — экспоненциальный бэкофф: 1s → 2s → 4s → 8s → макс 30s
- Максимум 5 попыток, затем — пометить как failed + уведомить пользователя
- Кнопка «Повторить» для failed-действий

#### 5.2.4. Конфликты

Если два устройства изменили одну задачу:
- Last-write-wins для простых полей (название, описание)
- Для перемещения (status) — серверная версия приоритетна, клиенту показывается toast: «Задача была перемещена на другом устройстве»

#### 5.2.5. Service Worker — расширение

Текущая конфигурация Workbox кэширует только статику. Нужно добавить:

- Runtime caching для `/api/state` (stale-while-revalidate, макс 5 минут)
- Runtime caching для `/api/tasks` (network-first, fallback к кэшу)
- Background sync для мутаций (через `workbox-background-sync`)

### 5.3. Индикатор офлайн-режима

В хедере доски (между названием проекта и FAB):

- **Офлайн:** иконка `CloudOff` (amber) + текст «Офлайн» (мелкий, `--muted-foreground`)
- **Синхронизация:** спиннер + «Синхронизация...»
- **Ошибка синхронизации:** иконка `AlertTriangle` (red) + «N действий не отправлено»

## 6. Производительность

### 6.1. Текущее состояние

- Code splitting уже реализован (`React.lazy()` в App.tsx)
- Skeleton loaders уже есть (`Skeleton.tsx`)
- `useMyrmex` загружает всё состояние одним запросом (`/api/state`)

### 6.2. Виртуализация списка карточек

Для колонок с большим количеством задач (>20) — виртуализация:

- Использовать `react-virtuoso` или `@tanstack/react-virtual`
- Рендерить только видимые карточки + буфер (5 карточек сверху/снизу)
- Каждая карточка ~120px высотой → при 20 карточках = 2400px DOM, при виртуализации = ~10 карточек = 1200px DOM
- Критично для колонки «Готово», где может накопиться 50+ задач

### 6.3. Lazy loading колонок

- При переключении на колонку — карточки этой колонки рендерятся сразу (из кэшированного состояния)
- Изображения/аттачменты в карточках — lazy loading (`loading="lazy"` для `<img>`)
- Для колонок с 0 задач — не рендерить пустой контейнер, показать компактное empty state

### 6.4. Skeleton loaders для мобильных

Расширить текущие `Skeleton.tsx` компоненты:

- `BoardSkeleton` — скелет доски: таб-бар (5 пульсирующих чипов) + 3 карточки-скелета
- `CardSkeleton` — уже есть, использовать при загрузке деталей задачи в Bottom Sheet
- При pull-to-refresh — показывать нативный loading-индикатор (не skeleton)

### 6.5. Pull-to-refresh

На мобильных — нативный pull-to-refresh для обновления данных доски:

- Использовать CSS `overscroll-behavior: contain` на контейнере карточек
- При _pull_ → вызов `refresh()` из `useMyrmex` + haptic `impact('light')`
- Визуальный индикатор — нативный браузерный или кастомный (spinner вверху)

### 6.6. Мемоизация

- `React.memo()` для `TaskCard` — предотвращать ре-рендер при изменении другой карточки
- `useMemo()` для фильтрации задач по колонке (не пересчитывать при каждом рендере)
- `useCallback()` для всех обработчиков событий (тап, long press, swipe)

### 6.7. Оптимистичные обновления

Все мутации (перемещение, создание, удаление) — оптимистичные:

1. Мгновенно обновить UI (без ожидания ответа сервера)
2. Отправить запрос к API
3. При ошибке — откатить изменение + haptic `notification('error')` + toast с кнопкой «Повторить»
4. При успехе — haptic `notification('success')` (если действие не было инициировано свайпом/тапом, которые уже дали haptic)

## 8. Принципы дизайна

1. **Одна рука.** Все основные действия выполняются одной рукой (большим пальцем). Никаких действий в верхнем левом углу без альтернативы.

2. **Мгновенная обратная связь.** Каждое касание — визуальный отклик (ripple, scale, highlight). Каждое действие — haptic. Никаких «подождите ответ сервера».

3. **Undo > Confirm.** Вместо «Вы уверены?» — действие выполняется сразу, но можно отменить (toast с undo на 3 секунды). Исключение: удаление из колонки «Готово».

4. **Прогрессивное раскрытие.** Простые действия (просмотр, быстрое перемещение) — одним жестом. Сложные (редактирование, выбор колонки) — через Bottom Sheet.

5. **Контекстная навигация.** MainButton и BackButton всегда делают то, что ожидает пользователь в текущем контексте. Никаких «внезапных» переходов.

6. **Graceful degradation.** Если TWA API недоступен — работаем как PWA. Если нет сети — работаем офлайн. Если нет haptic — работаем без него. Ничего не ломается.
