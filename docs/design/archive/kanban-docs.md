**Файл:** `README.ru.md` (основной), `README.md` (EN), `README.zh.md` (ZH)

### 3.1. Структура README

README — первое, что видит посетитель репозитория. Оно должно ответить на четыре вопроса: **что это**, **зачем**, **как запустить**, **куда дальше**.

```markdown
# 🐜 Myrmex Control

## Бейджи
Стек, версия, CI, покрытие тестами, лицензия.

## Описание (2-3 предложения)
Что это и для кого. Ключевая метафора: "пульт управления муравейником".

## Возможности (таблица)
15-20 фич с иконками. Группировка: ядро, безопасность, мобильность, локализация.

## Быстрый старт (5 команд)
git clone → npm install → npm run dev → открыть браузер.

## Скриншот / Демо-ссылка
Визуальная якорная точка. Ссылка на живое демо.

## Структура проекта (дерево)
Краткое дерево директорий с пояснениями.

## API (сводная таблица)
Группы эндпоинтов: Auth, Tasks, Projects, Library, Files, Servers, Analytics.

## Документация (ссылки)
Таблица: документ → описание.

## Развёртывание
Продакшн, демо, systemd-сервисы.

## Дизайн-система
Цветовая палитра, токены.

## Компромиссы
Честная таблица: решение → что теряем → почему оправдано.

## Дорожная карта
v1.0 (done) → v1.1 (in progress) → v2.0 (planned).

## Лицензия, Автор
```

### 3.2. Принципы README

- **Один экран без скролла** для верхней части (бейджи + описание + быстрый старт).
- **Три клика до любой документации** — ссылки на все ключевые документы.
- **Нет устаревшей информации** — версию и дату обновлять при каждом релизе.
- **Три языка** — EN, RU, ZH. Содержимое синхронизировано.

## 4. Руководство пользователя (User Guide)

**Целевая аудитория:** Безумный Доктор (admin), AI-агенты (operator).

**Где разместить:** Отдельная страница в интерфейсе Myrmex Control (вкладка «Справка») + Markdown-файл в `docs/user-guide/`.

### 4.1. Структура руководства пользователя

```markdown
# Руководство пользователя — Канбан-система Myrmex Control

## 1. Начало работы
   1.1. Вход в систему (логин, TOTP 2FA)
   1.2. Обзор интерфейса (дашборд, боковая панель, навигация)
   1.3. Три доски: ЗавЛаб, МУРАВЕЙ, КОТ — что где живёт
   1.4. Первые 5 минут: создать задачу, переместить, завершить

## 2. Работа с задачами
   2.1. Создание задачи (заголовок, описание, приоритет, теги, дедлайн)
   2.2. Типы задач: фича, баг, техдолг, исследование, рутина
   2.3. Шаблоны задач (предзаполненные формы)
   2.4. Перемещение задач между колонками (drag-and-drop)
   2.5. Зависимости и блокировки (что блокирует мою задачу)
   2.6. Критерии готовности (DoD) — когда задача считается выполненной
   2.7. Подзадачи и чеклисты
   2.8. Поиск и фильтрация задач

## 3. Три доски — три контекста
   3.1. Доска «ЗавЛаб» — стратегическая
       - Для кого: лаборатория, проекты, эпики
       - Колонки: Backlog → Todo → In Progress → Review → Done
       - Группировка по умолчанию: по эпикам
   3.2. Доска «МУРАВЕЙ» — тактическая
       - Для кого: AI-агенты
       - Колонки: Queued → Working → Blocked → Testing → Completed
       - Группировка по умолчанию: по исполнителям
       - Ограничения: агент может двигать только свои задачи
   3.3. Доска «КОТ» — личная
       - Для кого: только admin
       - Колонки: Inbox → This Week → Doing → Waiting → Done
       - Группировка: плоский список

## 4. Система весов и автоматическое распределение
   4.1. Как вычисляется приоритет задачи (весы: W_priority, W_age, W_deps, W_channel, W_recurring)
   4.2. Уровни приоритетов: P0 (Критический) — P4 (Фоновый)
   4.3. Автоматическое распределение по доскам (Weight Router)
   4.4. Ручное переопределение приоритета
   4.5. Эскалация стареющих задач

## 5. Аналитика и метрики
   5.1. Дашборд доски: WIP, Throughput, Cycle Time, Health Score
   5.2. Кумулятивная диаграмма потока (CFD) — как читать
   5.3. Гистограмма Cycle Time — предсказуемость
   5.4. Пропускная способность (Throughput) — продуктивность
   5.5. Старение задач — поиск застрявших
   5.6. Эффективность агентов — загрузка и производительность

## 6. Ритуалы
   6.1. Ежедневный стендап (15 мин)
   6.2. Пополнение бэклога (30-60 мин/неделя)
   6.3. Планирование спринта (30-60 мин/2 недели)
   6.4. Ретроспектива (30 мин/2 недели)
   6.5. Еженедельный обзор (20-30 мин)
   6.6. Ежемесячный стратегический обзор (60 мин)

## 7. Настройки
   7.1. Настройка TOTP 2FA
   7.2. Управление колонками доски
   7.3. WIP-лимиты
   7.4. Пороги старения задач
   7.5. Переключение языка (RU/EN)

## 8. Часто задаваемые вопросы (FAQ)
   - Почему задача не двигается в In Progress? (WIP-лимит, зависимости, state machine)
   - Как переназначить задачу другому агенту?
   - Что делать, если задача застряла в Review?
   - Как экспортировать данные доски?
   - Почему мой вес изменился автоматически?

## 9. Горячие клавиши
   Таблица: комбинация → действие.
```

## 5. API-документация весов и распределения

**Целевая аудитория:** Разработчики, интегрирующие систему весов.

**Где разместить:** `docs/api/weights-api.yaml` (OpenAPI) + раздел в `docs/design/kanban-automation.md`.

### 5.1. Структура API-документации весов

```markdown
# API: Система весов и автоматическое распределение

## Обзор
Система весов (Weight System) автоматически вычисляет приоритет каждой задачи
и распределяет задачи по доскам на основе канальных тегов.

## Модель веса

### Формула
W_total = W_priority + W_age + W_deps + W_channel + W_recurring

### Компоненты

| Компонент | Диапазон | Формула | Описание |
|-----------|----------|---------|----------|
| W_priority | 0–40 | По таблице P0–P4 | Базовый приоритет |
| W_age | 0–20 | min(20, floor(days / 2)) | Штраф за старение |
| W_deps | 0–15 | min(15, blocked_count * 5) | Бонус за разблокировку |
| W_channel | -10..+10 | По канальному тегу | Корректировка под доску |
| W_recurring | 0–15 | min(15, overdue_days * 3) | Штраф за просрочку повтора |

## Эндпоинты

### GET /api/tasks/:id/weight
Получить текущий вес задачи с разбивкой по компонентам.

**Ответ:**
{
  "task_id": "task-123",
  "total_weight": 67,
  "components": {
    "priority": 30,    // P1 — Высокий
    "age": 8,          // 16 дней в бэклоге
    "deps": 10,        // блокирует 2 задачи
    "channel": 10,     // channel:muravey на доске МУРАВЕЙ
    "recurring": 9     // просрочена на 3 дня
  },
  "calculated_at": "2026-05-11T12:00:00Z"
}

### POST /api/tasks/:id/weight/recalculate
Принудительный пересчёт веса (обычно автоматический).

### GET /api/weights/config
Получить текущую конфигурацию весов (пороги, коэффициенты).

### PUT /api/weights/config
Обновить конфигурацию весов (только admin).

### POST /api/weights/distribute
Запустить ручное распределение задач по доскам (Weight Router).

**Ответ:**
{
  "distributed": 15,
  "skipped": 3,
  "by_board": {
    "zavlab": 5,
    "muravey": 7,
    "kot": 3
  }
}

### GET /api/weights/queue/:board
Получить очередь задач для доски, отсортированную по весу (убывание).

## Канальные теги

| Тег | Доска МУРАВЕЙ | Доска КОТ | Доска ЗавЛаб |
|-----|---------------|-----------|--------------|
| channel:muravey | +10 | -10 | 0 |
| channel:kat | -10 | +10 | 0 |
| channel:zavlab | 0 | 0 | +10 |
| channel:any / нет | 0 | 0 | 0 |

## Автоматические триггеры

| Событие | Действие |
|---------|----------|
| Создание задачи | Расчёт W_total, распределение по доске |
| Изменение статуса | Пересчёт W_age, W_deps |
| Добавление зависимости | Пересчёт W_deps для блокирующей задачи |
| Каждые 24 часа (cron) | Пересчёт W_age, W_recurring для всех задач |
| Задача > 14 дней без движения | Эскалация: уведомление admin |

## Коды ошибок

| Код | Описание |
|-----|----------|
| 400 | Невалидная конфигурация весов |
| 403 | Нет прав на изменение конфигурации |
| 404 | Задача не найдена |
| 409 | Конфликт: задача уже распределена |
```

### 5.2. Принципы API-документации

- **OpenAPI 3.0** — единый формат, генерируемый из кода.
- **Примеры запросов и ответов** — для каждого эндпоинта.
- **Интерактивная песочница** — Swagger UI по адресу `/api/docs`.
- **Версионирование** — версия API в URL (`/api/v1/...`) или в заголовке.

Используется формат **[Keep a Changelog](https://keepachangelog.com/ru/1.0.0/)** с семантическим версионированием (SemVer).

### 6.2. Структура релиза

```markdown
## [X.Y.Z] — YYYY-MM-DD

### 🚨 Breaking Changes
Изменения, нарушающие обратную совместимость.

Новые фичи, эндпоинты, возможности.

### 🔧 Changed (Изменения)
Изменения в существующей функциональности.

Исправленные баги. Ссылка на issue, если есть.

### 🗑️ Deprecated (Устаревает)
Скоро будет удалено. Альтернатива.

### 🔒 Security (Безопасность)
Устранённые уязвимости. CVE, если есть.

### 📖 Documentation (Документация)
Обновление документации.

### 🧪 Tests (Тесты)
Новые тесты, улучшение покрытия.

### 📦 Dependencies (Зависимости)
Обновление зависимостей.
```

### 6.3. Категории изменений для канбан-системы

| Категория | Иконка | Пример |
|-----------|--------|--------|
| Доски | 📋 | Новая доска «КОТ» |
| Задачи | ✅ | Шаблоны задач |
| Веса | ⚖️ | Новый компонент W_recurring |
| Аналитика | 📊 | CFD-график на дашборде |
| Безопасность | 🔒 | State machine для переходов |
| Автоматизация | 🤖 | Weight Router v2 |
| Мобильная версия | 📱 | Touch-жесты для drag-and-drop |
| Производительность | ⚡ | Оптимизация рендеринга 500+ карточек |
| UX | 🎨 | Анимации перемещения карточек |
| Инфраструктура | 🏗️ | Docker Compose для деплоя |

### 6.4. Секция Backlog

В конце CHANGELOG.md — секция «Backlog» с идеями, которые пока не реализованы:

```markdown
## Backlog

### 🧠 Название идеи
**Problem:** Описание проблемы.
**Goal:** Что хотим получить.
**Ideas to explore:**
- Вариант А
- Вариант Б
**Status:** 📋 Pending | 🔄 In Progress | ✅ Done | ❌ Rejected
```

### 6.5. Правила ведения changelog

1. **Каждый релиз — новая секция** в начале файла (новые сверху).
2. **Дата обязательна** — формат `YYYY-MM-DD`.
3. **SemVer строго**: MAJOR.MINOR.PATCH.
4. **Breaking changes — всегда в начале** релиза с инструкцией миграции.
5. **Ссылки на issues/PR** — в формате `#123`.
6. **Не удалять старые релизы** — только добавлять новые.
7. **Backlog обновлять** — реализованные идеи перемещать в релиз.

## 7. Onboarding для новых пользователей

**Целевая аудитория:** Новый пользователь системы (человек или агент).

**Где разместить:** `docs/user-guide/onboarding.md` + интерактивный тур в UI.

### 7.1. Структура onboarding

```markdown
# Onboarding — Первые шаги в Myrmex Control

## Шаг 0: Вход в систему
- Откройте [myrmexcontrol.shtab-ai.ru](https://myrmexcontrol.shtab-ai.ru)
- Введите логин и пароль
- Введите TOTP-код (если 2FA включена)
- Вы на дашборде

## Шаг 1: Обзор интерфейса (2 мин)
- Боковая панель: Дашборд, Доски, Проекты, Библиотека, Файлы
- Верхняя панель: поиск, уведомления, профиль
- Главная область: содержимое текущей страницы

## Шаг 2: Ваша первая задача (3 мин)
- Перейдите на доску «ЗавЛаб»
- Нажмите «+ Задача»
- Заполните: заголовок, описание, приоритет
- Нажмите «Создать»
- Задача появится в колонке «Backlog»

## Шаг 3: Перемещение задачи (1 мин)
- Перетащите задачу из «Backlog» в «Todo»
- Назначьте исполнителя (себя или агента)
- Перетащите в «In Progress»
- Перетащите в «Review», затем в «Done»

## Шаг 4: Три доски — три контекста (3 мин)
- ЗавЛаб: стратегические задачи лаборатории
- МУРАВЕЙ: тактические задачи агентов
- КОТ: личные задачи (только для admin)

## Шаг 5: Система весов (2 мин)
- У каждой задачи есть автоматический вес
- Чем выше вес — тем выше приоритет
- Вес зависит от: приоритета, возраста, зависимостей, канала

## Шаг 6: Аналитика (2 мин)
- Откройте вкладку «Аналитика»
- Посмотрите: WIP, Throughput, Cycle Time
- Прочитайте Health Score — здоровье потока

## Шаг 7: Настройка профиля (2 мин)
- Откройте профиль (иконка в правом верхнем углу)
- Настройте TOTP 2FA
- Выберите язык (RU/EN)
- Настройте уведомления

## Шаг 8: Ритуалы (1 мин)
- Ежедневный стендап: каждый день, 15 мин
- Еженедельный обзор: пятница, 20 мин
- Пополнение бэклога: понедельник, 30 мин

## Шаг 9: Горячие клавиши (1 мин)
- `N` — новая задача
- `/` — поиск
- `B` — переключить доску
- `F` — открыть фильтры
- `?` — справка по горячим клавишам

## Шаг 10: Куда идти дальше
- [Руководство пользователя](user-guide.md) — полное описание
- [API-документация](api/openapi.yaml) — для разработчиков
- [FAQ](user-guide.md#faq) — ответы на частые вопросы
```

### 7.2. Принципы onboarding

- **10 шагов, 15-20 минут** — не больше. Если дольше — пользователь бросит.
- **Интерактивность** — пользователь ДЕЛАЕТ, а не читает.
- **Прогресс-бар** — показывать «Шаг 3 из 10».
- **Возможность пропустить** — кнопка «Пропустить тур».
- **Контекстные подсказки** — tooltips на элементах интерфейса при первом посещении.
- **Повторный запуск** — кнопка «Пройти тур заново» в настройках.

## 8. Architecture Decision Records (ADR)

**Где разместить:** `docs/adr/`

### 8.1. Формат ADR

```markdown
# ADR-XXX: Название решения

## Статус
Proposed | Accepted | Deprecated | Superseded by ADR-YYY

## Контекст
Какую проблему решаем? Какие ограничения? Какие альтернативы рассматривались?

## Решение
Что выбрали и почему. Конкретно.

## Последствия
Что изменится. Что станет проще. Что станет сложнее. Какие новые ограничения появятся.

## Альтернативы
Что рассматривали и почему отвергли.

## Ссылки
Связанные ADR, issues, документы.
```

## 9. Процесс обновления документации

### 9.1. Когда обновлять

| Событие | Что обновлять |
|---------|---------------|
| Новый релиз | CHANGELOG, README (версия, бейджи), API docs |
| Новый эндпоинт | OpenAPI spec, API-документация весов |
| Изменение модели данных | kanban-architecture.md, ADR, OpenAPI spec |
| Изменение RBAC | kanban-security.md, матрица прав |
| Новый ритуал | Руководство пользователя, onboarding |
| Изменение весов | kanban-automation.md, API docs |
| Новый ADR | docs/adr/, kanban-docs.md (карта) |

### 9.2. Ответственные

| Документ | Ответственный |
|----------|---------------|
| README | Любой разработчик (PR) |
| CHANGELOG | Автор релиза |
| API docs | Автор эндпоинта |
| kanban-*.md | Архитектор / технический писатель |
| ADR | Автор решения |
| Руководство пользователя | Технический писатель |
| Onboarding | UX-дизайнер + технический писатель |

### 9.3. Чеклист перед релизом

```
[ ] CHANGELOG.md обновлён с новой версией
[ ] README: бейджи версии обновлены
[ ] OpenAPI spec: новые эндпоинты задокументированы
[ ] ADR: принятые решения зафиксированы
[ ] Руководство пользователя: новые фичи описаны
[ ] Onboarding: новые шаги добавлены (если нужно)
[ ] Документация синхронизирована между EN/RU
```
