## 1. Обзор стратегии

### 1.1. Пирамида тестирования

```
           ┌───────────┐
           │   E2E     │  ← 10% сценариев (критические пути)
          ┌┴───────────┴┐
          │ Интеграц.   │  ← 20% сценариев (API + компоненты)
         ┌┴─────────────┴┐
         │    Unit       │  ← 70% сценариев (функции, хуки, утилиты)
         └───────────────┘
```

### 1.2. Что тестируем

| Модуль | Что покрываем | Приоритет |
|--------|--------------|-----------|
| **Board.tsx** | Drag-and-drop, создание/удаление задач, фильтры | Критический |
| **tasks.ts (API)** | CRUD, валидация, state machine переходов | Критический |
| **Weight Router** | Расчёт весов, маршрутизация, балансировка | Высокий |
| **Auth** | JWT, RBAC, TOTP, brute-force | Критический |
| **Mobile layout** | ColumnTabs, Bottom Sheet, touch-жесты | Высокий |
| **Offline** | Кэширование, action queue, синхронизация | Средний |
| **Automation** | DoR/DoD, авто-переходы, уведомления | Высокий |
| **Analytics** | CFD, cycle time, throughput | Средний |

### 1.3. Текущее состояние тестов

- `src/__tests__/auth.test.ts` — базовые тесты bcrypt (3 теста)
- `vitest.config.ts` — настроен для `tests/**/*.test.ts`, node environment
- Покрытие: server-only, пороги 80/75/80/80 (statements/branches/functions/lines)
- **Нет тестов для:** Board.tsx, tasks API, weight system, RBAC, state machine, клиентских компонентов

### 1.4. Что НЕ тестируем

- Визуальное оформление (цвета, шрифты, анимации) — это область visual regression
- Сторонние библиотеки (React, Express, bcrypt) — доверяем их тестам
- Инфраструктуру (systemd, nginx) — это область infra testing

## 5. Интеграционные тесты

### 5.1. API + State: Полный цикл задачи

**Файл:** `tests/integration/task-lifecycle.test.ts`

**Сценарий: Задача проходит полный цикл backlog → done**

```
1. POST /api/tasks { title: "Тест", project_id: "p1" } → 201, status=backlog
2. PUT /api/tasks/:id { status: "todo" } → 409 (backlog→todo требует DoR)
3. PUT /api/tasks/:id { description: "Контекст", priority: "high", assignee_id: "a1" } → 200
4. POST /api/tasks/:id/move { status: "todo" } → 200 (DoR выполнен)
5. POST /api/tasks/:id/move { status: "in_progress" } → 200
6. POST /api/tasks/:id/move { status: "review" } → 200
7. POST /api/tasks/:id/move { status: "done" } → 200, completed_at != null
8. GET /api/tasks/:id → 200, status=done, completed_at заполнен
9. GET /api/state → changelog содержит все действия
```
