## 1. Обзор крипто-слоя

### 1.1. Концепция

Крипто-слой — это **надстройка над существующей канбан-системой**, а не отдельный продукт. Он добавляет:

- **Тип задачи `crypto`** — для задач, связанных с крипто-операциями
- **Тип задачи `alert`** — для автоматически создаваемых алертов по токенам
- **Тип задачи `defi`** — для отслеживания DeFi-позиций и метрик
- **Крипто-виджеты** — мини-панели с ценами, объёмами, алертами прямо на доске
- **Весовую корректировку `W_crypto`** — для приоритизации крипто-задач на основе рыночных данных

### 1.2. Архитектура

```

│                    Канбан-система Myrmex Control                │
│  ┌──────────┐  ┌──────────┐  ┌──────────┐                      │
│  │  ЗАВЛАБ  │  │  МУРАВЕЙ │  │   КОТ    │                      │
│  └────┬─────┘  └────┬─────┘  └────┬─────┘                      │
│       │              │              │                            │
│  ┌────▼──────────────▼──────────────▼─────┐                     │
│  │          Weight Router (весы)          │                     │

│  └────────────────┬───────────────────────┘                     │

│  │         Crypto Engine (новый модуль)   │                     │
│  │  ┌──────────┐ ┌──────────┐ ┌────────┐ │                     │
│  │  │ Price    │ │ Alert    │ │ DeFi   │ │                     │
```

### 1.3. Принципы

- **Крипто-задачи — те же задачи.** Они живут в том же бэклоге, подчиняются тем же правилам, имеют те же поля. Дополнительные поля — только специфичные для крипты.
- **Алерты — это задачи.** Не уведомления в вакууме, а полноценные карточки на доске с приоритетом, исполнителем и критериями закрытия.
- **Весы учитывают рынок.** Волатильность, объём, просрочка алерта — всё влияет на приоритет.
- **Данные кешируются.** Внешние API не опрашиваются при каждом запросе. Кэш: 60 сек для цен, 5 мин для DeFi.

## 3. Трекинг крипто-задач

### 3.1. Тип задачи `crypto`

Крипто-задача — это расширенная задача с дополнительными полями:

```
CryptoTask {
  // --- Стандартные поля Task ---
  id, projectId, title, description, status, priority
  assigneeId, tags, dueDate, epicId, why, who
  acceptanceCriteria, contextLinks, storyPoints
  type: "crypto"           // ← новый тип
  board: zavlab | muravey | kot

  // --- Крипто-расширение ---
  crypto: {
    token: "NOT" | "HMSTR" | "DeFi" | string
    action: "buy" | "sell" | "hold" | "stake" | "farm"
           | "analyze" | "market" | "partner" | "audit"
    targetPrice: number        // Целевая цена (USD)
    stopLoss: number           // Стоп-лосс (USD)
    amount: number             // Сумма (USD или токены)
    exchange: string           // Биржа/DEX
    txHash: string             // Хэш транзакции (если есть)
    notes: string              // Заметки по сделке
  }
}
```

### 3.2. Примеры крипто-задач

**Задача «Маркетинговая кампания $NOT»:**
```
📋 Запустить реферальную программу $NOT
├── Тип: crypto
├── Токен: NOT
├── Действие: market
├── Доска: ЗАВЛАБ
├── Эпик: Q2 Маркетинг
├── Приоритет: P1 (Высокий)
├── Критерии:
│   □ Текст реферальной программы
│   □ Лендинг с UTM-метками
│   □ Пост в Telegram-канале
│   □ Отчёт через 7 дней (конверсии)
└── Контекст:
    • CoinGecko: /coins/notcoin
    • Канал: @notcoin
```

**Задача «Стейкинг $HMSTR»:**
```
📋 Застейкать 50,000 $HMSTR в пуле X
├── Тип: crypto
├── Токен: HMSTR
├── Действие: stake
├── Доска: МУРАВЕЙ
├── Сумма: 50,000 HMSTR
├── Пул: HMSTR/ETH 0.3% fee
├── APY: 47% (auto-compound)
├── Критерии:
│   □ Транзакция подтверждена
│   □ TxHash записан в карточку
│   □ Позиция отслеживается в DeFi-мониторе
└── Контекст:
    • DexScreener: /pairs/0x...
    • Кошелёк: 0x1234...
```

### 3.3. Жизненный цикл крипто-задачи

```
BACKLOG → TODO → IN PROGRESS → PENDING TX → COMPLETED
                                       │
                                       → FAILED → (возврат в TODO)
```

**Статус `PENDING TX`** — промежуточный статус для задач, требующих транзакции в блокчейне. Задача ожидает подтверждения транзакции.

**Статус `FAILED`** — транзакция не прошла (slippage, out of gas, отмена). Задача возвращается в TODO с пометкой о причине неудачи.

### 3.4. Связи с другими задачами

Крипто-задачи могут блокировать и быть заблокированными:

- **«Купить $NOT»** → блокирует → **«Застейкать $NOT»**
- **«Аудит пула»** → блокирует → **«Инвестировать в пул»**
- **«Анализ конкурентов»** → блокирует → **«Маркетинговая кампания»**

## 4. Алерты по токенам как задачи

### 4.1. Концепция

Каждый алерт — это **автоматически создаваемая задача** на доске МУРАВЕЙ. Алерт имеет:

- Тип `alert`
- Автоматически вычисляемый приоритет (на основе серьёзности условия)
- Исполнителя — агент-наблюдатель (Watcher Agent)
- Критерий закрытия — подтверждение или отклонение

### 4.2. Типы алертов

| Тип алерта | Условие | Приоритет | TTL |
|------------|---------|-----------|-----|
| **Price Spike** | Цена изменилась >±10% за 1ч | P1 — Высокий | 2 часа |
| **Price Dump** | Цена упала >-20% за 24ч | P0 — Критический | 1 час |
| **Volume Surge** | Объём вырос >3x от среднего | P2 — Средний | 4 часа |
| **Liquidity Drop** | Ликвидность упала >-30% | P1 — Высокий | 2 часа |
| **Whale Move** | Крупная транзакка >$100k | P1 — Высокий | 1 час |
| **APY Change** | APY изменился >±20% за 24ч | P2 — Средний | 6 часов |
| **Smart Contract** | Обновление контракта | P0 — Критический | 30 мин |
| **Listing** | Листинг на новой бирже | P1 — Высокий | 4 часа |

**TTL (Time To Live)** — время жизни алерта. Если алерт не обработан за TTL — автоматическая эскалация (уведомление Евгению + повышение приоритета).

### 4.3. Формат алерта-задачи

```
🚨 [ALERT] $NOT +12.5% за 1 час
├── Тип: alert
├── Токен: NOT
├── Условие: Price Spike > +10%
├── Текущая цена: $0.00892 (+12.5%)
├── Предыдущая: $0.00793
├── Объём 24ч: $4.2M (2.1x средний)
├── Приоритет: P1 (авто)
├── Исполнитель: Watcher Agent
├── TTL: 2 часа (до 14:30 MSK)
├── Критерии закрытия:
│   □ Подтвердить: тренд продолжается
│   │   → Создать задачу "Анализ причины роста"
│   □ Отклонить: ложный сигнал
│   │   → Закрыть с пометкой "noise"
│   □ Мониторить: недостаточно данных
│   │   → Продлить TTL на 1 час
└── Авто-действия:
    • Уведомление в Telegram
    • Обновление виджета цены на доске
```

### 4.4. Правила обработки алертов

1. **Алерт не может быть удалён.** Только закрыт с пометкой (confirmed / dismissed / expired).
2. **При подтверждении** — автоматическое создание аналитической задачи.
3. **При отклонении** — запись в лог ложных срабатываний (для калибровки порогов).
3. **При истечении TTL** — эскалация на доску КОТ (личная задача для Евгения).
4. **Максимум 5 открытых алертов** на один токен. При превышении — уведомление о необходимости ревизии правил.

## 5. Интеграция с CoinGecko

### 5.1. API Endpoints

| Endpoint | Назначение | Частота | Кэш |
|----------|-----------|---------|-----|
| `/simple/price` | Текущая цена токена | 60 сек | 60 сек |
| `/coins/{id}/market_chart` | История цен (7д/30д/90д) | 5 мин | 5 мин |
| `/coins/{id}/ohlc` | OHLC-свечи (1д/7д) | 5 мин | 5 мин |
| `/coins/markets` | Рыночные данные (cap, volume) | 2 мин | 2 мин |
| `/coins/{id}/tickers` | Тикеры по биржам | 5 мин | 5 мин |
| `/global` | Глобальные данные рынка | 5 мин | 5 мин |

### 5.2. Сопоставление токенов

| Токен | CoinGecko ID | Contract (Ethereum) |
|-------|-------------|---------------------|
| $NOT | `notcoin` | `0x388d258...` |
| $HMSTR | `hamster-kombat` | TON contract |

### 5.3. Данные для виджетов

**Минимальный набор (обновление каждые 60 сек):**
- Цена USD
- Изменение за 24ч (%)
- Объём 24ч
- Рыночная капитализация
- FDV (Fully Diluted Valuation)

**Расширенный набор (обновление каждые 5 мин):**
- Изменения за 1ч / 7д / 30д
- Средний объём 7д
- ATH / ATL + дата
- Рыночный ранг
- Доминация сектора

### 5.4. Rate Limits

- **Free tier:** 10-30 вызовов/мин
- **Планируемый объём:** ~15 вызовов/мин (3 токена × 5 endpoint-ов)
- **Стратегия:** приближение к лимиту — переход на приоритетные endpoint-ы (цена + объём), остальные — реже.

### 5.5. Fallback

При недоступности CoinGecko:
1. Использовать кешированные данные (до 10 мин)
2. Переключиться на DexScreener как источник цен
3. Пометить виджеты как "stale" (серый индикатор)
4. Создать задачу-алерт `[SYSTEM] CoinGecko API недоступен`

## 7. DeFi-метрики на доске

### 7.1. Тип задачи `defi`

```
DefiTask {
  // --- Стандартные поля Task ---
  id, projectId, title, description, status, priority
  assigneeId, tags, dueDate, epicId, why, who
  acceptanceCriteria, contextLinks, storyPoints
  type: "defi"             // ← новый тип
  board: muravey            // ← всегда на доске МУРАВЕЙ

  // --- DeFi-расширение ---
  defi: {
    protocol: string           // Uniswap, Aave, Lido, etc.
    chain: string              // Ethereum, BSC, TON, etc.
    pool: string               // Пара или пул
    position: {
      entryPrice: number       // Цена входа
      currentPrice: number     // Текущая цена
      amount: number           // Количество токенов
      valueUSD: number         // Текущая стоимость (USD)
      pnl: number              // P&L (USD)
      pnlPercent: number       // P&L (%)
    }
    yield: {
      apy: number              // Текущий APY
      apyAvg7d: number         // Средний APY за 7д
      rewards: string[]        // Токены вознаграждений
      compounding: boolean     // Авто-компаундинг
    }
    risk: {
      ilRisk: "low" | "medium" | "high"  // Impermanent Loss risk
      smartContractRisk: 1-10            // Риск контракта
      liquidityRisk: 1-10                // Риск ликвидности
      overallScore: 1-10                 // Общий скоринг
    }
    txHash: string             // Хэш транзакции входа
    txHashExit: string         // Хэш транзакции выхода (если закрыта)
  }
}
```

### 7.3. Автоматические DeFi-задачи

На основе данных DeFi-мониторинга автоматически создаются задачи:

| Условие | Авто-задача | Приоритет |
|---------|-------------|-----------|
| APY упал >30% за 24ч | «Проверить пул [POOL] — APY упал» | P2 |
| Impermanent Loss >15% | «Ребалансировать позицию [POOL]» | P1 |
| Награды накоплены >$50 | «Собрать награды [PROTOCOL]» | P3 |

### 7.4. DeFi Health Score

Для каждой DeFi-позиции вычисляется скоринг здоровья:

```
health_score = 100
  - (il_risk * 5)              // 0-50 штраф

  - (liq_risk * 2)             // 0-20 штраф
  + (apy_normalized * 10)      // 0-10 бонус
  + (age_days / 30 * 5)        // 0-5 бонус за стаж

  50-79:  🟡 Требует внимания
  0-49:   🔴 Критическая
```
