## 1. Обзор системы мониторинга

### 1.1. Что мониторим

| Объект | Что проверяем | Частота |
|--------|---------------|---------|
| **Доска ЗАВЛАБ** | Задачи без исполнителя, просрочки, старение, WIP | Каждые 5 мин |
| **Доска МУРАВЕЙ** | Загрузка агентов, сбои распределения, blocked ratio | Каждые 5 мин |
| **Доска КОТ** | Просрочки, старение, перегрузка | Каждые 15 мин |
| **Weight Router** | Балансировка каналов, ошибки маршрутизации | Каждые 5 мин |
| **Сервис myrmex-control** | systemd state, RAM, CPU, uptime | Каждую минуту |
| **API health endpoint** | HTTP 200, latency < 200ms | Каждые 30 сек |
| **myrmex.json** | Целостность файла, размер, changelog overflow | Каждые 5 мин |

### 1.2. Архитектура мониторинга

```
┌─────────────────────────────────────────────────────────┐
│                   WATCHDOG (systemd timer)               │
│                  Запуск каждые 60 сек                    │
└──────────────────────┬──────────────────────────────────┘
                       │
          ┌────────────┼────────────┐
          │            │            │
   ┌──────▼─────┐ ┌───▼────┐ ┌────▼─────┐
   │ API Health │ │ Board  │ │ Weight   │
   │ Check      │ │ Health │ │ Router   │
   │ (HTTP)     │ │ Scan   │ │ Audit    │
   └──────┬─────┘ └───┬────┘ └────┬─────┘
          │            │           │
          └────────────┼───────────┘
                       │
            ┌──────────▼──────────┐
            │   ALERT ENGINE      │
            │   Сравнение с       │
            │   порогами          │
            └──────────┬──────────┘
                       │
          ┌────────────┼────────────┐
          │            │            │
   ┌──────▼─────┐ ┌───▼────┐ ┌────▼─────┐
   │ Telegram   │ │ Audit  │ │ Dashboard│
   │ Alert      │ │ Trail  │ │ Update   │
   └────────────┘ └────────┘ └──────────┘
```

### 1.3. Принципы

- **Три уровня алерта:** INFO (лог), WARNING (уведомление), CRITICAL (немедленная эскалация).
- **Ни один инцидент не теряется.** Всё логируется в `audit.json` и `journalctl`.
- **Алерты не дублируются.** Дедупликация по ключу инцидента в окне 15 минут (кроме P1).
- **P1 — всегда мгновенно**, без батчинга, без тихих часов.
- **Автоматическая эскалация.** Если P1 не подтверждён за 15 минут — повторная отправка каждые 5 минут.

## 2. Метрики здоровья досок

### 2.1. Метрики уровня задач

| Метрика | Описание | Источник | Порог WARNING | Порог CRITICAL |
|---------|----------|----------|---------------|----------------|
| `tasks.total` | Всего активных задач на доске | `tasks[]` | — | — |
| `tasks.unassigned` | Задачи без исполнителя | `assignee_id == null` | > 5 | > 15 |
| `tasks.overdue` | Просроченные задачи (due_date < now, status != done) | `due_date`, `status` | > 3 | > 10 |
| `tasks.stale` | Задачи без движения > порога | `updated_at` | > 5 | > 10 |
| `tasks.blocked_ratio` | Доля заблокированных задач | `status == blocked / total` | > 15% | > 30% |
| `tasks.backlog_growth` | Рост бэклога за неделю | `created_at` за 7 дней | > 20% | > 50% |
| `tasks.avg_age_days` | Средний возраст задач в бэклоге | `created_at` | > 14 дней | > 30 дней |

### 2.2. Метрики уровня потока (Flow Metrics)

| Метрика | Описание | Формула | Порог WARNING | Порог CRITICAL |
|---------|----------|---------|---------------|----------------|
| `flow.throughput` | Завершённых задач в неделю | `count(status=done, last 7d)` | < 5 | < 2 |
| `flow.cycle_time_hours` | Среднее время IN_PROGRESS -> DONE | `avg(completed_at - in_progress_at)` | > 48ч | > 96ч |
| `flow.lead_time_hours` | Среднее время BACKLOG -> DONE | `avg(completed_at - created_at)` | > 72ч | > 168ч |
| `flow.wip` | Текущие задачи в работе | `count(status=in_progress) + count(status=review)` | > 80% лимита | > 100% лимита |
| `flow.wip_limit` | WIP-лимит доски | Конфиг | — | — |
| `flow.cfd_stability` | Индекс стабильности CFD | `1 - (std_dev(in_progress) / mean)` | < 0.7 | < 0.5 |

### 2.3. Метрики уровня распределения (Weight Router)

| Метрика | Описание | Формула | Порог WARNING | Порог CRITICAL |
|---------|----------|---------|---------------|----------------|
| `distribution.gini` | Коэффициент Джини загрузки агентов | Статистика по `assignee_id` | > 0.3 | > 0.5 |
| `distribution.idle_agents` | Агенты без задач | `count(agent.tasks == 0)` | > 30% | > 50% |
| `distribution.overloaded_agents` | Агенты с задачами > лимита | `count(agent.tasks > max_tasks)` | > 0 | > 2 |
| `distribution.channel_imbalance` | Разница нагрузки между каналами | `abs(load(muravey) - load(kat))` | > threshold (20) | > threshold x 2 |
| `distribution.routing_errors` | Ошибки маршрутизации за час | Audit trail: `action == routing_failed` | > 0 | > 5 |
| `distribution.rebalance_failures` | Сбои ребалансировки за сутки | Audit trail: `action == rebalance_failed` | > 0 | > 3 |

### 2.4. Метрики уровня сервиса

| Метрика | Описание | Источник | Порог WARNING | Порог CRITICAL |
|---------|----------|----------|---------------|----------------|
| `service.uptime` | Аптайм сервиса | `systemctl show` | — | < 99% за сутки |
| `service.memory_mb` | Потребление RAM | `/proc/<pid>/status` | > 256 MB | > 512 MB |
| `service.cpu_percent` | Загрузка CPU | `/proc/stat` | > 50% | > 80% |
| `service.api_latency_ms` | Ответ API health endpoint | HTTP-замер | > 200ms | > 1000ms |
| `service.api_errors_5xx` | 5xx ошибки за час | Access log | > 5 | > 20 |
| `service.myrmex_file_size_mb` | Размер myrmex.json | `stat` | > 5 MB | > 10 MB |
| `service.changelog_overflow` | Достигнут лимит changelog (1000) | `changelog.length` | > 800 | > 950 |

## 3. Health Score — индекс здоровья

### 3.1. Определение

**Health Score** — единый индекс здоровья доски от 0 до 100. Вычисляется как взвешенная сумма по четырём доменам.

### 3.2. Формула

```
Health Score = Σ (domain_score x domain_weight)

Где:
  domain_score = 100 - Σ (metric_penalty)
  metric_penalty = max(0, (metric_value - warning_threshold) /
                        (critical_threshold - warning_threshold)) x max_penalty
```

### 3.3. Веса доменов

| Домен | Вес | Метрики внутри |
|-------|-----|----------------|
| **Flow** (поток) | 35% | throughput, cycle_time, wip_utilization, cfd_stability |
| **Distribution** (распределение) | 25% | gini, idle_agents, channel_imbalance, routing_errors |
| **Quality** (качество) | 25% | blocked_ratio, overdue, stale, unassigned |
| **Service** (сервис) | 15% | uptime, api_latency, memory, myrmex_file_size |

### 3.4. Интерпретация

| Score | Статус | Цвет | Действие |
|-------|--------|------|----------|
| 90–100 | Отлично | Зелёный | Плановый мониторинг |
| 70–89 | Хорошо | Зелёный | Плановый мониторинг, обратить внимание на жёлтые метрики |
| 50–69 | Требует внимания | Жёлтый | Анализ причин, подготовка корректирующих действий |
| 30–49 | Плохо | Оранжевый | Немедленный анализ, эскалация на ежедневный стендап |
| 0–29 | Критично | Красный | Немедленная эскалация, P1/P2 инцидент |

### 3.5. Health Score по доскам — сводка

```
+----------+-------+--------+--------------------------------------+
| Доска    | Score | Статус | Главная проблема                     |
+----------+-------+--------+--------------------------------------+
| ЗАВЛАБ   | 88    | 🟢     | Нет                                 |
| МУРАВЕЙ  | 72    | 🟡     | Cycle Time растёт (8.3ч vs цель 6ч)  |
| КОТ      | 61    | 🟡     | Inbox overflow (12 задач > 30 дней)  |
+----------+-------+--------+--------------------------------------+
```

## 4. Система алертов

### 4.1. Уровни алертов

| Уровень | Название | Канал доставки | Время реакции | Эскалация |
|---------|----------|----------------|--------------|-----------|
| **INFO** | Информация | Только лог (audit trail) | Нет | Нет |
| **WARNING** | Предупреждение | Telegram (батч 5 мин) | 4 часа | Если не снят за 4ч → CRITICAL |
| **CRITICAL** | Критичный | Telegram (мгновенно) + Email | 30 минут | Если не подтверждён за 15 мин → повтор каждые 5 мин |
| **P1** | Авария | Telegram (мгновенно) + Email + Webhook | 15 минут | Повтор каждые 5 мин до подтверждения |

### 4.2. Правила анти-спама

- **Дедупликация:** одинаковый алерт (по ключу `board:metric:level`) не отправляется повторно в окне 15 минут.
- **P1 — без дедупликации:** каждый P1 алерт доставляется, даже если предыдущий не снят.
- **Батчинг WARNING:** группируются в окне 5 минут в одно сообщение.
- **Тихие часа:** 23:00–08:00 — только CRITICAL и P1. WARNING копится в утренний дайджест.
- **Максимум:** не более 10 алертов в минуту (кроме P1). Избыточные — в лог.

### 4.3. Формат алерта в Telegram

```
🔴 [CRITICAL] МУРАВЕЙ — Weight Router Failure

Доска: МУРАВЕЙ
Метрика: distribution.routing_errors
Текущее значение: 7 ошибок/час
Порог: > 5 = CRITICAL
Время: 2026-05-11 14:23:00

Последние ошибки:
  14:22 — task-456: channel tag missing, fallback to channel:any
  14:20 — task-451: agent not found, assignee_id = "agent-deleted"
  14:18 — task-448: circular dependency detected, routing skipped

Действия:
  [Проверить аудит] [Открыть доску] [Подтвердить]
```

## 5. Классификация инцидентов (P1–P4)

### 5.1. P1 — Критический (Авария)

**Определение:** Система полностью неработоспособна или данные потеряны/повреждены.

**Критерии:**
- API не отвечает > 5 минут (health endpoint → timeout или 503)
- myrmex.json повреждён (невалидный JSON, нулевой размер)
- Weight Router не работает > 30 минут (все задачи копятся в бэклоге)
- Потеря данных (changelog пуст после рестарта, задачи исчезли)
- Безопасность скомпрометирована (несанкционированный доступ)

**Время реакции:** 15 минут
**Время восстановления:** 1 час
**Эскалация:** Немедленно на Евгения, повтор каждые 5 минут

**Примеры:**
- `myrmex.json` повреждён из-за неатомарной записи
- Weight Router зациклился, 1000+ ошибок/минуту
- Сервис myrmex-control упал и не поднимается через systemd

### 5.2. P2 — Высокий (Деградация)

**Определение:** Критическая функция работает с ошибками или серьёзно деградировала.

**Критерии:**
- API отвечает, но latency > 1 секунды устойчиво
- Weight Router работает, но с ошибками (> 5 ошибок/час)
- Одна из досок недоступна (например, КОТ не загружается)
- WIP-лимит превышен на 50% и растёт
- Blocked ratio > 30%
- Распределение задач сломано (всё на одном агенте)

**Время реакции:** 1 час
**Время восстановления:** 4 часа
**Эскалация:** Уведомление в Telegram, обсуждение на ближайшем стендапе

**Примеры:**
- Диспетчер весов не балансирует каналы (разница > 40 в течение 2 часов)
- Агент циклически перемещает задачу (100+ перемещений за минуту)
- Доска МУРАВЕЙ не обновляется > 30 минут

### 5.3. P3 — Средний (Частичная деградация)

**Определение:** Некритичная функция работает некорректно, есть обходные пути.

**Критерии:**
- Аналитика показывает некорректные данные (но доска работает)
- Уведомления не доставляются (но задачи создаются)
- Старение задач не пересчитывается (W_age заморожен)
- Один из агентов не получает задачи (но другие работают)
- Changelog близок к переполнению (> 800 записей)

**Время реакции:** 4 часа
**Время восстановления:** 1 рабочий день
**Эскалация:** Задача создаётся на доске ЗАВЛАБ, обсуждается на еженедельном обзоре

**Примеры:**
- Ежедневный дайджест не отправляется 2 дня подряд
- Heatmap старения показывает устаревшие данные
- Фильтр по тегам не работает на доске КОТ

### 5.4. P4 — Низкий (Косметический)

**Определение:** Незначительная проблема, не влияющая на работу системы.

**Критерии:**
- Опечатка в уведомлении
- Некорректный цвет бейджа приоритета
- Дашборд загружается на 1 секунду дольше обычного
- Устаревшая запись в changelog (дубликат)

**Время реакции:** 1 рабочий день
**Время восстановления:** 1 неделя (или бэклог)
**Эскалация:** Задача в бэклог, без уведомлений

## 6. Процесс обработки инцидентов

### 6.1. Жизненный цикл инцидента

```
  ОБНАРУЖЕНИЕ -> КЛАССИФИКАЦИЯ -> ЭСКАЛАЦИЯ -> ДИАГНОСТИКА -> УСТРАНЕНИЕ -> ВОССТАНОВЛЕНИЕ -> ПОСТ-МОРТЕМ
       |              |              |            |             |              |              |
  Алерт /       P1-P4          TG / Email    Логи /      Исправление   Проверка     Запись в
  Ручное        по критериям   + Audit       myrmex.json  + тест        метрик        INCIDENTS.md
  обнаружение                   trail        + journalctl
```

### 6.2. Обнаружение

| Источник | Что ловит | Задержка |
|----------|-----------|----------|
| Watchdog timer (systemd) | Сервис упал, API не отвечает | 60 сек |
| Alert Engine | Метрики за порогами | 5 мин |
| Ручной мониторинг (стендап) | Аномалии на доске | До 24 ч |
| Агент-оператор | Ошибки при работе с задачами | Мгновенно |
| Евгений | Любое подозрительное поведение | Мгновенно |

### 6.3. Классификация

При обнаружении аномалии — немедленная классификация по матрице P1–P4 (см. раздел 5).

**Правило:** Если сомневаешься — классифицируй выше. Лучше переоценить, чем недооценить.

### 6.4. Диагностика

**Порядок действий при диагностике:**

1. **Проверить статус сервиса:**
   ```bash
   systemctl status myrmex-control
   journalctl -u myrmex-control --since "30 min ago" --no-pager
   ```

2. **Проверить API health:**
   ```bash
   curl -s -o /dev/null -w "%{http_code} %{time_total}s" http://localhost:8167/api/health
   ```

3. **Проверить myrmex.json:**
   ```bash
   jq '. | length' /root/LabDoctorM/projects/myrmex-control/myrmex.json
   jq '.changelog | length' /root/LabDoctorM/projects/myrmex-control/myrmex.json
   ```

4. **Проверить последние ошибки в audit trail:**
   ```bash
   jq '.changelog | map(select(.success == false)) | .[-10:]' myrmex.json
   ```

5. **Проверить метрики досок:**
   ```bash
   curl -s http://localhost:8167/api/analytics/health | jq .
   ```

### 6.5. Устранение

| Тип проблемы | Действие | Ответственный |
|--------------|----------|---------------|
| Сервис упал | `systemctl restart myrmex-control` | Автоматический watchdog |
| myrmex.json повреждён | Восстановить из бэкап (`.bak`), проверить целостность | Евгений |
| Weight Router сбой | Остановить диспетчер, очистить очередь, перезапустить | Евгений + агент |
| Перекос распределения | Ручной запуск ребалансировки через API | Евгений |
| Утечка памяти | Рестарт сервиса, анализ логов | Евгений |
| Задача-зомби (циклическое перемещение) | Заблокировать задачу, сбросить счётчик | Агент-оператор |

### 6.6. Восстановление

После устранения проблемы — обязательная проверка:

1. **Health Score** вернулся в зелёную зону (> 70).
2. **Алерт снят** — метрика ниже порога WARNING.
3. **Данные целы** — `myrmex.json` валиден, changelog не повреждён.
4. **Агенты работают** — нет агентов в статусе error.
5. **Распределение корректно** — Gini < 0.3, нет перекоса каналов.

**Критерий закрытия инцидента:** все 5 пунктов выполнены + 10 минут стабильной работы.

- **Дата инцидента:** 2026-05-11

- **Дата устранения:** 14:47

- **Доска:** МУРАВЕЙ
- **Ответственный:** Евгений
- **Статус:** Закрыт

| Время | Событие |
|-------|---------|
| 14:20 | Алерт A04: routing_errors > 5/час |
| 14:23 | Обнаружено Евгением через Telegram |
| 14:25 | Диагностика: Weight Router зациклен на task-448 |

Weight Router попал в бесконечный цикл при обработке задачи task-448,
которая имела циклическую зависимость (A->B->A). Диспетчер не имел
защиты от циклов и генерировал ошибки маршрутизации до исчерпания лимита.

## Корневая причина
Отсутствие валидации циклических зависимостей в Weight Router.
API принимал зависимость B->A без проверки на циклы.

## Что сработало хорошо
- Алерт сработал за 3 минуты после начала проблемы
- Watchdog не дал сервису упасть (изоляция диспетчера)
- Бэкап myrmex.json был актуален

## Что можно улучшить
- Добавить валидацию циклических зависимостей при создании
- Добавить circuit breaker для Weight Router (max 100 операций/мин)

## Метрики

- **MTTR** (Mean Time to Resolve): 24 минуты

## 10. Интеграция с systemd и journalctl

### 10.1. systemd unit для watchdog

Файл: `/etc/systemd/system/myrmex-watchdog.service`

```
[Unit]
Description=Myrmex Kanban Watchdog
After=myrmex-control.service
Requires=myrmex-control.service

[Service]
Type=oneshot
ExecStart=/root/LabDoctorM/projects/myrmex-control/scripts/watchdog.sh
User=root
StandardOutput=journal
StandardError=journal

[Install]
WantedBy=multi-user.target
```

### 10.2. systemd timer для периодического запуска

Файл: `/etc/systemd/system/myrmex-watchdog.timer`

```
[Unit]
Description=Run Myrmex Watchdog every 60 seconds

[Timer]
OnBootSec=30
OnUnitActiveSec=60
AccuracySec=5

[Install]
WantedBy=timers.target
```

### 10.3. Команды диагностики через journalctl

```bash
# Последние 50 строк логов сервиса
journalctl -u myrmex-control -n 50 --no-pager

# Логи за последние 30 минут
journalctl -u myrmex-control --since "30 min ago" --no-pager

# Логи watchdog
journalctl -u myrmex-watchdog --since "1 hour ago" --no-pager

# Только ошибки
journalctl -u myrmex-control -p err --since today --no-pager

# Мониторинг в реальном времени
journalctl -u myrmex-control -f

# Рестарт с логированием
systemctl restart myrmex-control && journalctl -u myrmex-control -f
```

### 10.4. Логирование инцидентов в journalctl

Каждый инцидент логируется с тегом уровня:

```
# CRITICAL
logger -p user.crit -t myrmex-watchdog "P1: Weight Router failure, routing_errors=7/h"

# WARNING
logger -p user.warn -t myrmex-watchdog "W: МУРАВЕЙ gini=0.35 > threshold 0.3"

# INFO
logger -p user.info -t myrmex-watchdog "I: Health check OK, all boards green"
```

### 10.5. Автоматический рестарт при падении

В unit-файле `myrmex-control.service`:

```
[Service]
Restart=on-failure
RestartSec=5
StartLimitIntervalSec=300
StartLimitBurst=3
```

**Правило:** если сервис упал 3 раза за 5 минут — остановить и ждать ручного вмешательства. Отправить CRITICAL алерт.

| **CFD** | Cumulative Flow Diagram — кумулятивная диаграмма потока |

| **Channel Imbalance** | Разница нагрузки между каналами (МУРАВЕЙ vs КОТ) |
| **Stale Task** | Задача без движения дольше порога (по умолчанию 5-10 дней) |
| **Circuit Breaker** | Паттерн защиты: автоматическая остановка при превышении лимита ошибок |
